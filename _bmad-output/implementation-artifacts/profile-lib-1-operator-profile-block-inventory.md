## Story

As an **admin/operator of the 3d-portal**,
I want **a small in-product inventory surface for SEPARATE Orca profile blocks — upload/import an Orca profile block (process profiles first; filament + machine as supporting blocks), list the imported blocks, open one block's curated metadata + validation state, and delete a block — with the backend classifying each block as machine/process/filament (or rejecting it as unsupported/error), extracting only curated metadata, and flagging inheritance/compatibility problems**,
so that **I am no longer dependent on manual file placement to manage Orca profile blocks, while the curated-metadata-only surface never exposes raw Orca JSON, and the shipped 33.1/33.2 fixed-grid compiled-intent projection (the resolver intent path, `bundle_hash`, append-only stores, provenance snapshots) stays completely untouched.**

This is the **re-pointed next slice** of Epic E33 (the 2026-06-06 profile-model correction froze Story 33.3 and re-targeted the next slice here). It is the **first surface of the corrected canonical model**: separate Orca-like building blocks, **process profiles first**. It is deliberately **inventory CRUD only** — it is **NOT** the `PrintProfileOffer` / `ProfileChain` offer layer (PROFILE-OFFER-1, the slice after this), **NOT** an N×M relationship editor, **NOT** a member spool-picker, and **NOT** a raw Orca field editor or JSON viewer (SCP § 6 deferred register).

## Acceptance Criteria

### Backend — classification + curated-metadata engine (`slicer/profile_library.py`)

1. **AC-1 — New additive engine module, grid untouched.** A new `apps/api/app/modules/slicer/profile_library.py` holds the classifier, curated-metadata extractor, validation-state derivation, governance check, and the block storage layer. It is **purely additive**: it does NOT edit `resolver.py`'s `resolve()`, the `intents/<printer_ref>/<material_class>/<quality_tier>.json` grid layout (`VendoredProfileSource.intent_path`), the 33.1 inventory read, the 33.2 grid-import endpoint, the append-only bundle/snapshot/estimate stores, `bundle_hash`, or `compatibility.py`. A test asserts `git diff` against the grid surfaces is body-unchanged (mirrors the 33.2 AC-23 fence). (SCP § 4 coexist-no-migration; § 1 additive.)
2. **AC-2 — Classification contract (machine / process / filament / unsupported).** `classify_profile(body: dict) -> ProfileType | None` classifies a parsed Orca profile block. **Precedence:** (a) an explicit Orca **`type` field** when present, mapped through a small normalization table to `machine` / `process` / `filament`; (b) otherwise **structural heuristics** over discriminating keys verifiable against the bench fixtures — `filament_type` ⇒ filament, `printer_model` / `nozzle_diameter` / `printable_height` ⇒ machine, `layer_height` (and process-only keys) ⇒ process; (c) **ambiguous / none** ⇒ `None` ⇒ the import is rejected `422 unsupported_profile` (nothing stored). The explicit-`type` key name + value vocabulary (branch a) is **operator data (G-DATA)** — the spec names the *mechanism* (prefer explicit type, fall back to verifiable heuristics) but the concrete `type` strings are confirmed from real Orca exports before the branch-(a) fixtures are frozen; the heuristic branch (b) is grounded in `tests/fixtures/slicer/system/*.json` today. (SCP § 3.1-3.3; Appendix A "backend validates and classifies"; magic-constant discipline [[feedback_scp_pre_enumeration_phase]] § C.)
3. **AC-3 — Curated-metadata extraction (minimized, no raw body).** `extract_curated_metadata(body, *, profile_type, system_tree) -> CuratedMetadata` extracts ONLY: `name` (Orca `name`); `profile_type`; `inherit` (the declared `inherit`/`inherits` parent) + a best-effort resolved `inherit_chain` (names walked against `system_tree`, truncated/halted on an unknown parent — never raises); `settings_id` (the `*_settings_id` / `setting_id` when present, else `None`); `source` (`from` field — `system` / `user` / unknown); for **filament** blocks the normalized `material_type` (from `filament_type` through the small generic category table — PLA/PETG/PCTG/TPU, see AC-9) and `compatible_printers` (when present); `is_system` (`from == "system"`). It carries **NO** raw layer-height / temps / volumetric-speed / density / g-code / full Orca key set — the same FR20-PRESET-1 / NFR21-OBS-1 leak fence the `schemas.py` DTOs already enforce. (SCP § 3.1-3.3 fields; Appendix A "extract minimized metadata only".)
4. **AC-4 — Validation-state derivation (`usable` / `requires_attention` / `error`).** `derive_validation_state(...) -> (ValidationState, reasons)` maps a classified block to one of three states: **`error`** — the block could not be classified (AC-2 reject path; never stored); **`requires_attention`** — classified + stored, but a curated problem is flagged (e.g. `unknown_inherit_parent` — the declared `inherit` is not in the system tree; or the AC-5 user-process governance flag); **`usable`** — classified, inheritance resolvable (or a self-contained system block), no governance flag. Each non-`usable` state carries a machine-readable reason CATEGORY (the FE localizes it; no display text from the backend). (SCP § 6 "usable / requires-attention / error feedback".)
5. **AC-5 — Process-inheritance governance (the Fenrir Orca rule).** For a **user** process block (`from` ≠ `system`) whose `inherit` parent resolves to a **non-system** process profile (or to an unknown parent), the block is stored but flagged **`requires_attention`** with reason `user_process_invalid_inheritance`. Rationale (SCP § 3.2 governance): an Orca user **process** profile must inherit a **system** process profile; Orca may **silently drop** an invalid user-process inheritance — surfacing it as a visible flag is the whole point (the portal must not silently repeat Orca's drop). The check is **flag, not hard-reject** (the operator sees *why* and can fix the source); a hard reject is reserved for the AC-2 unclassifiable `error` path. (SCP § 3.2; § 6 "enforce process-inheritance governance".)

### Backend — block storage layer (additive subtree, no DB)

6. **AC-6 — Separate-block storage layout (additive `library/` subtree).** Imported blocks live in a NEW subtree under the existing `slicer_vendored_profiles_dir` root, **disjoint from** the `system/` and `intents/` (grid) trees:
   `<root>/library/<profile_type>/<block_id>.json` (the verbatim imported Orca block body) + `<root>/library/<profile_type>/<block_id>.manifest.json` (the curated-metadata sidecar v1). `profile_type ∈ {machine, process, filament}`. The layout is owned by a single `block_path(profile_type, block_id) -> Path` helper (single SoT, mirroring `VendoredProfileSource.intent_path`). (SCP § 4 "separate, additive ... does not delete or rewrite the grid".)
7. **AC-7 — Server-derived, path-safe `block_id` (no attacker-controlled path segment).** `block_id` is **derived server-side**, NOT taken from the upload: `block_id = uuid5(NAMESPACE_URL, f"{profile_type}:{name}").hex` (deterministic over `(profile_type, Orca name)`, so a re-import of the same named block of the same type is an **upsert**, and the id is always a 32-char hex string — no path separators, traversal, or attacker control). This **structurally avoids** the path-traversal class the 33.2 `printer_ref` write path had to gate against (`is_safe_printer_ref` / `is_within_intents_root`); a belt-and-braces containment assert that `block_path(...)` stays at/below `<root>/library` is still applied (defense-in-depth). The `GET`/`DELETE` `{block_id}` path param is validated as 32-char lowercase hex → `404` otherwise (never joined raw without the charset gate). (SCP § 7 path-safety foundation carried forward; magic-constant: uuid5 chosen because the id must be path-safe + stable-on-re-import, NOT for brevity.)
8. **AC-8 — Atomic store + metadata preservation (33.2 foundations reused).** Storing a block writes the body JSON + the sidecar manifest using the **same rollback-safe two-phase atomic publish** the 33.2 `import_service.publish_intent` uses (stage both temps fsynced → commit body → commit manifest → roll back the body if the manifest commit fails), and the **same owner/mode preservation** (`_apply_metadata` — files land `ezop:ezop 664` on the operator-managed bind mount, never root-owned `0600`; the `31bcf0c` runtime-regression fix). On ANY failure the `library/` subtree is byte-identical to before and no temp remains. **Reuse the existing helpers** (`_stage_temp`, `_apply_metadata`, the two-phase commit) — do not re-implement them; factor them into a shared private surface if needed without changing their behavior. (SCP § 7 atomic-publish + permission-preservation foundations.)

### Backend — CRUD endpoints (on the existing `slicer/admin_router.py`)

9. **AC-9 — `POST /api/admin/profiles/library` — import/upload a block.** Multipart upload of a single Orca profile-block JSON (`file: UploadFile`, optional `portal_label: str | None`; **no slot fields** — the target tree is derived from the classified type, not the form). Gate order: **size 413** (reuse `_MAX_PROFILE_BYTES = 1 MiB`, AC-4 of 33.2, same contract — a single Orca block is a small JSON object) → **parse 422 `invalid_json`** → **classify 422 `unsupported_profile`** when `classify_profile` returns `None` → extract curated metadata → derive validation state (AC-4/5; `requires_attention` does NOT block storage) → **atomic store** (AC-8) → **audit** → **201** with the `ProfileLibraryBlock` curated DTO (AC-13). Admin-gated (`current_admin`), absent from `_PUBLIC_ROUTES`, CSRF automatic (middleware) — same posture as the 33.2 endpoint. The `material_type` normalization (AC-3) maps Orca `filament_type` through the AC-9 category table (PLA/PETG/PCTG/TPU); an out-of-table filament type is surfaced as `material_type=null` + a `requires_attention` `unknown_material_type` flag, never minted as a new narrow category (SCP § 3.6). (SCP § 6 add/import; Appendix A.1.)
10. **AC-10 — `GET /api/admin/profiles/library` — list imported blocks.** Returns every imported block's **curated metadata** (the `ProfileLibraryBlock` DTO), read from the sidecar manifests under `<root>/library/**`, optionally filtered by `profile_type` query param (`?profile_type=process` etc.; absent ⇒ all). Ordering is deterministic (by `profile_type` then `name`). It reads **only** the manifests (curated), never the raw bodies, so the list cannot leak Orca internals. A missing/empty `library/` tree ⇒ `200` with an empty list (the empty state IS an empty inventory). (SCP § 6 list; Appendix A.1.)
11. **AC-11 — `GET /api/admin/profiles/library/{block_id}` — get one block's curated detail.** Returns the single `ProfileLibraryBlock` curated DTO for `block_id` (validated hex, AC-7), `404 not_found` when absent. **It returns curated metadata + validation state ONLY — NOT the raw Orca JSON body** (explicit: there is **no** raw-Orca-JSON preview/detail endpoint or field in this story — SCP § 6, Appendix A "do not build a raw Orca JSON preview/viewer"). (SCP § 6 get; Appendix A.1.)
12. **AC-12 — `DELETE /api/admin/profiles/library/{block_id}` — remove a block (audited).** Removes both the body + manifest files for `block_id` atomically (best-effort-consistent: remove manifest then body so a torn delete never leaves a manifest pointing at a gone body in the list), `404 not_found` when absent, `204` on success, **audited** (AC-14). Re-deleting an absent block is a `404` (idempotent-safe; not a 500). (SCP § 6 delete.)
13. **AC-13 — Curated DTOs (`extra="forbid"`, no raw body).** New `slicer/schemas.py` DTOs: `ProfileLibraryBlock` (`block_id`, `profile_type`, `name`, `source`/`is_system`, `inherit`, `inherit_chain: list[str]`, `settings_id: str|null`, `material_type: str|null`, `compatible_printers: list[str]`, `validation_state`, `reasons: list[str]`, `portal_label: str|null`, `imported_at`, `imported_by`), `ProfileLibraryListResponse` (`blocks: list[ProfileLibraryBlock]`). Reuse the existing `ProfileImportRejection` for 413/422 detail. Every DTO is `ConfigDict(extra="forbid")` and carries **no raw Orca key body, no filesystem path, no g-code** (the AC-9-of-32.6 leak fence; a negative-assertion test mirrors it). `validation_state = Literal["usable","requires_attention","error"]`; `profile_type = Literal["machine","process","filament"]`. (Pre-enumeration: extend the module, same fence.)

The 33.2 `test_slicer_worker.py` admin-router route-surface fence is extended from ONE-GET+ONE-POST to also admit the four sanctioned library routes (one mechanical fence update, mirroring how 33.2 evolved it from 33.1). (NFR21-AUTH-1.)

### Magic-constant discipline (per [[feedback_scp_pre_enumeration_phase]] § C)
