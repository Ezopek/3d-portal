## What surprised us

1. **The TYPE-DRIFT failure mode emerged in Story 8.6 — caught by docker build, NOT by Codex review.** Story 8.6's dev commit (`d511c74`) shipped Admin Invites tab components but omitted 7 required TypeScript type exports in `api-types.ts` (InviteRoleRequest, InviteTTLPreset, InviteStatus, AdminInviteRow, AdminInvitesListResponse, GenerateInviteRequest, GenerateInviteResponse). Local `npm run typecheck` passed (types inferred from the consumer files), but production `vite build` in the docker image failed with TS2305 errors. Fix-up `56eec7d` added the missing exports + test helper bodies. **Lesson:** local typecheck and production build have different type-resolution semantics; the dev → Codex loop didn't catch this because Codex reviews source diffs not build output. Mitigation candidate: extend `check-all.sh` to run a production-mode build (or `tsc --noEmit --strict` with isolatedModules) as a stage before commit.

2. **Story 8.6 was the first Epic 8 story with ZERO backend changes — and it shipped fastest.** No new endpoints, no Pydantic schemas, no Alembic migration, no audit-action name, no rate-limit scope, no entity_type, no middleware, no infra/. Pure UI consumer of already-shipped Story 6.3 endpoints. The story spec was authored, dev-implemented, fix-up'd, and shipped within a single sprint segment — demonstrating that UI-only stories on stable backend surfaces have a fundamentally lower defect surface area. **This shifts the cost model for Epic 9+10 planning:** Epic 10's UI-only stories (if any) should be estimated against this lower-bound, not against Epic 8's full-stack stories like 8.3 or 8.5.

3. **Story 8.3's typecheck fix-up (`ddb9f14` P1) was caused by the V8/V9 test assertion-shape regression introduced by Story 8.4's dev session.** The V8/V9 tests used `toHaveBeenCalledWith({single-arg})` while the source code calls `mutate(variables, {onSuccess, onError})` — vitest strictly checks all args. Story 8.4's dev session needed to fix this in-scope to satisfy its own AC-11 "14/14 green" gate. **This is the inverse of the Story 8.1→8.3 cross-story integration catch (good): Story 8.3→8.4 surfaced a pre-existing test-shape defect (mediocre).** Logged as a pre-existing-issue candidate per `feedback_preexisting_issue_threshold` — 1 occurrence so far, threshold not crossed.

4. **Story 8.1's monotonic UPDATE + asyncio.to_thread P2 fix-up surfaced a concurrency class that wasn't in E7's catalog.** The throttle UPDATE could move `last_active_at` BACKWARDS under TTL+handler-overrun race: a slow request captures `now` pre-handler, exceeds 300s throttle TTL, a parallel request lands a newer timestamp, then the slow request commits its older timestamp. Fix: capture `write_now` AFTER handler returns + add monotonic predicate `WHERE last_active_at IS NULL OR last_active_at < :write_now`. **Plus the sync engine.begin() blocking the async event loop under SQLite contention** — fix via `asyncio.to_thread`. Both findings on the same Codex pass; both shipped in `195fb22`. This is a NEW concurrency pattern (monotonic CAS for time-stamped updates) that should join the still-unauthored concurrency-patterns.md catalog (E7 retro item §7).

5. **Story 8.2's stable tie-breaker P2 was a pagination correctness defect, not a security or concurrency defect.** Non-unique sort keys (role, last_active_at NULL groups) without a tie-breaker → SQL is free to reorder equal rows → OFFSET/LIMIT pagination could repeat or skip users across pages. Fix: every sort branch terminates with `.order_by(..., User.email.asc(), User.id.asc())`. **This is a NEW class of finding for Codex** — it's not a race, not a security gap, not a wiring oversight; it's a relational-database correctness invariant ("ORDER BY must be deterministic when consumed by paginated reads"). Worth noting that Codex catches this class consistently.

6. **Story 8.5's "token burned on weak-password retry" P2 fix-up was a subtle UX correctness defect.** GETDEL claim ran BEFORE password validation, so a 422 weak-password attempt consumed the only reset token. The UI surfaced 422 as recoverable; user retries; gets 404 token_invalid; needs to ask admin to mint another link. Fix: password validates pre-claim; token stays for retry; only consumed on the successful commit path. **The audit-emission entity_id asymmetry** that resulted (pre-claim emission has `entity_id=None` since identity is token-bound; post-claim has `entity_id=user_id`) was documented in spec § Codex review surface area — Codex flagged it anyway, and the fix shifted weak-password to be the pre-claim path (audit row has `entity_id=None`, `after.reason=weak_password`). Tests `c5/c6` updated to assert the new shape. **The right defensive answer was already in the spec; Codex caught the actual code drift.**

7. **Story 8.5's absolute-URL fix-up became a verbatim copy-paste reference for Story 8.6's token modal.** The fix-up to `ResetLinkDisplayModal.tsx:45-51` resolves the relative `/reset-password?token=...` against `window.location.origin` so out-of-band SMS/Messenger paste delivers a working link. Story 8.6's `InviteTokenDisplayModal.tsx` followed the exact same pattern from initial commit — no fix-up needed. **The dev → Codex loop produces reusable code snippets that propagate forward to the next story's spec, reducing the next story's fix-up surface.** This is the "lessons applied within-epic" pattern at its strongest.

8. **Story 8.4 added a NEW Alembic migration (0015) for the `force_2fa_enrollment` BOOLEAN column.** Original Story 8.4 acceptance check shape (epics §1798) allowed either reusing 0014 OR a new tiny 0015 "per implementer's call". The implementer chose 0015 — a clean single-column migration kept the chain auditable per-story rather than bundling unrelated columns into 0014. **The downgrade round-trip was verified end-to-end in test_admin_users_2fa_overrides.py** (upgrade head → downgrade -1 → upgrade head on tmp SQLite, all clean transitions). The alembic chain continuity invariant from epics §1753 holds at 0015 head.

9. **Story 8.6 regenerated 12 admin-users baseline PNGs (NOT just added 16 new admin-invites PNGs) because the AdminTabs styling change cascaded.** The transition of the Invites tab from `<span aria-disabled="true">` placeholder to `<Link to="/admin/invites">` changed cursor + opacity + hover state on EVERY admin page. **The baseline-acceptance gate (project-context.md §245) requires per-PNG sign-off in commit message** — Story 8.6's commit message included 28 baseline-reviewed sign-off lines (16 new + 12 regenerated). This is the largest single-commit baseline batch in Initiative 5; the convention scaled cleanly.

## Breakthrough moments / patterns reused

1. **Monotonic CAS predicate on time-stamped updates (Story 8.1 fix-up `195fb22`)** — `UPDATE users SET last_active_at = :write_now WHERE id = :id AND (last_active_at IS NULL OR last_active_at < :write_now)`. Even if a parallel request sneaks a newer timestamp in between TTL expiry + this UPDATE, the predicate refuses the backward write. Pattern reusable for any time-stamped column with concurrent writers; should join the still-unauthored `docs/concurrency-patterns.md` catalog.
2. **`asyncio.to_thread` for sync engine.begin() in async ASGI middleware (Story 8.1 same fix-up)** — keeps event loop non-blocking under SQLite disk contention. Pattern from E7 7.3 (bcrypt.checkpw offload) extended to DB writes; reusable for any sync DB call in async handlers (specifically: ANY `engine.begin()` block called from middleware/dependency that runs on the request hot path).
3. **Deterministic ORDER BY terminator for paginated reads (Story 8.2 fix-up `ec5ac5d`)** — `.order_by(..., User.email.asc(), User.id.asc())` on every sort branch. Pattern reusable for any List endpoint with OFFSET/LIMIT + non-unique sort keys; should be encoded as a project convention (e.g., AGENTS.md § Paginated list endpoints MUST terminate ORDER BY with a unique-key column).
4. **One-shot flag clear via unconditional SQL UPDATE (Story 8.4 fix-up `af07752`)** — `UPDATE users SET force_2fa_enrollment = FALSE WHERE id = :id AND force_2fa_enrollment IS TRUE` instead of `if user.force_2fa_enrollment: user.force_2fa_enrollment = False`. Race-safe: only clears when actually set; the cached ORM row staleness becomes irrelevant. Pattern reusable for any one-shot flag clear after an external state change (force-enroll, force-disable, force-reset).
5. **Pre-claim validation gate to preserve single-use tokens on recoverable errors (Story 8.5 fix-up `cd6354a`)** — password validates BEFORE GETDEL claim; weak-password 422 leaves the token claimable by retry. Pattern reusable for any single-use token + multi-step consume flow (password-reset, invite-redeem, magic-link-login, etc.).
6. **Absolute-URL resolution for out-of-band token delivery (Story 8.5 fix-up applied to Story 8.6 verbatim)** — `new URL(relative_path, window.location.origin).toString()`. Pattern propagated from `ResetLinkDisplayModal.tsx:45-51` to `InviteTokenDisplayModal.tsx:45-51` without re-discovery. **The fix-up itself became reusable code that the next story copy-pasted.**
7. **NEW backend sub-module package convention (`auth/password_reset/` from Story 8.5, mirroring `auth/totp/` from Story 7.2)** — 5-file split (`__init__.py` + `service.py` + `router.py` + `admin_router.py` + `schemas.py`) when a new concern crosses both admin-side and public-side endpoints with shared service primitives. Pre-existing modules stay in their original location; minimal-diff principle holds.
8. **Cleartext-token-once modal pattern propagation (Story 6.3 → 8.5 → 8.6)** — Dialog + readonly Input + clipboard-copy button + Done button + `useState<boolean>(false)` copied slot. 4 modals across 3 stories, same shape, zero divergence. Pattern is now load-bearing UX convention for any one-time-cleartext surface (future: API-key-display, OAuth-secret-display, recovery-code-display).
9. **Forward-guard test pattern for deliberate exclusions (Story 8.2 FR5-ADMIN-4 negative AC)** — vitest V10 + Playwright Test 4 + Test 5 (later renamed to "active-state regression guard" in Story 8.6) act as regression guards for INTENTIONAL non-features (no bulk operations) AND for transitional states (Invites tab disabled until Story 8.6 ships). The test name evolves with the surface state; the assertion shifts from "stays disabled" to "is now an active link" — the test isn't deleted, it's promoted. **Reusable for any "deliberate exclusion" or "deferred feature" with a clear ship-trigger.**
10. **Audit-action forward-declaration at epic-N's closing commit registers epic-(N+1)'s actions** — Story 7.5's closing commit added 6 E8 action names to `app/core/audit.py` per the forward-declaration convention. Story 8.4 + 8.5 emit them WITHOUT touching the registry. AC-13 zero-touch on `app/core/audit.py` holds for the entire epic. Scales linearly with epic count; defends against merge conflicts on the registry file.
