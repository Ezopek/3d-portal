Amelia (Developer): "Single curated commit `893bb97` on `main`. No follow-up fix commit needed (unlike Init 21's two-day PROFILE-PUBLISH-FIX tail). The slice was bounded *and* complete first time."

### W2 — Leak fence asserted structurally, not just by output shape (the load-bearing safety win)

Murat (Test Architect): "The biggest test-discipline win of this slice. NFR22-LEAK-FENCE-1 / Decision AQ has five clauses. AC-10 is the obvious one — *the serialized response matches the allowlist, no `args`/`kwargs`/`result` keys*. That's an output-shape test. **But AC-8 is the structural one** — *no unbounded `KEYS` in the snapshot path*. Output-shape tests can't catch a `KEYS` call that happens to return the same data; you need to assert at the *call surface*."

Murat (Test Architect): "The implementation chose to wrap `fakeredis.aioredis.FakeRedis` in a `_NoKeysRedis` proxy that **raises on any `.keys(...)` invocation** and inject it via `app.dependency_overrides[get_queue_conn]`. **Any code path that calls `KEYS` fails the test loudly.** That is the bounded-SCAN-not-KEYS contract asserted at the seam where it lives, not at the DTO. This generalizes — the same pattern could pin any future 'must use bounded SCAN' contract on a fakeredis test."

Winston (Architect): "And it pre-emptively defended against the exact arq footgun the discovery spec named: `ArqRedis.queued_jobs()` deserializes pickled args; `all_job_results()` issues unbounded `KEYS arq:result:*` (`arq/connections.py:192-208`). Neither is called by the handler. The contract is enforced by the test, not by reviewer vigilance."

Amelia (Developer): "Output allowlist + structural KEYS-forbidden + path/secret negative test. Three orthogonal fences, each catching a different defect class. That's the leak-fence model maturing."

### W3 — Magic-constant discipline at AC time (not "feels right")

Murat (Test Architect): "Three numeric constants in this story — the FE polling interval, the `recent[]` SCAN hard cap, and the `SCAN COUNT` batch hint. Per `[[feedback_scp_pre_enumeration_phase]]` § C, each is declared in `constants.py` (backend) / FE constant (frontend) with a contract-pointing docstring:"

- **Polling interval** — points to UC1/UC2 watchability ("operator's own UC2 cites a 5-minute wait, so few-second resolution is ample") AND the homelab single-Redis load bound (3 `zcard` + 2 small SCANs per poll). Arbitrary-but-bounded product default; marked as such; pinned by perf telemetry or operator preference if either materializes.
- **`recent[]` SCAN hard cap (~50)** — "enough to spot a cluster of failures in the 1h window without an unbounded fetch." Arbitrary display/perf default; marked as such; revisit when G-LEDGER lands.
- **`SCAN COUNT` (small)** — Redis cursor batch hint, not a result cap; small to avoid long single calls on the homelab Redis.
- **`interval_s`** — **NOT a console-chosen constant** — read from the worker's actual `health_check_interval` (3600s today) and surfaced verbatim so coarse granularity is honest.

Murat (Test Architect): "TB-016 lesson held: do not justify a numeric budget by 'feels right'. Every constant in this story can be defended against the contract it points to. And `interval_s` is explicitly *not* a console choice — that's the discipline preventing a future maintainer from drifting it away from arq's actual behavior."

## §3 What we'd do differently — root causes

Amelia (Developer): "Single-story epic, but it has real lessons. The first one is the L4 prediction from Init 21 coming due."

### L1 — AdminTabs ripple under-regeneration in the dev-story session (the Init 21 L4 prediction materializing)

Sally (UX): "The Init 21 retro § L4 named *'`AdminTabs` is functioning correctly but its layout response to number-of-tabs is currently absorbed by the visual-baseline contract rather than designed. The cost is bounded today and rising linearly per tab.'* Init 22 added the 6th admin tab (Queues). The dev-story session regenerated only the desktop-light project's AdminTabs baselines (14 PNGs on the rippled admin specs). The controller `check-all.sh` run surfaced the gap — **46 screenshots failed** because the ripple actually spans all 4 projects (desktop/mobile × light/dark), not just desktop-light."

Sally (UX): "The controller's fix was mechanical (`npm run test:visual -- --update-snapshots`) and the re-ran green gate proved the widened set was correct. **But the under-regeneration is a real session discipline gap.** The story spec says (AC-22): *'New Playwright baselines across the 4 projects (`desktop-light/dark`, `mobile-light/dark`)'* — and *'The new admin tab ripples the `AdminTabs` visual baselines.'* The dev session applied the 4-project rule to the *new* baselines but applied a 1-project assumption to the *ripple*."

Murat (Test Architect): "The structural fact: the AdminTabs ripple's project scope is currently implicit. AC-22's '4 projects' wording attaches to the new baselines. The ripple-scope expectation lives only in the dev-session's mental model. **A check-all run is the only mechanical enforcement.** That worked here — the controller's gate caught it. But it cost a re-run."

**Root cause (named):** the AdminTabs ripple project-scope is implicit in the spec; the dev-session can plausibly under-apply it; the only mechanical check is the final visual-suite gate.

**Forward-applicable lesson:** any future admin-tab addition (or any chrome-ripple addition) should make the *ripple project scope* explicit in the AC ("the new tab ripples N PNGs across **all 4 projects**") and the dev-session should re-affirm the ripple matches the spec before declaring visual baselines done. This is a candidate addition to `[[reference_web_routetree_regen]]` — promoting from "ripples AdminTabs visual baselines" to "ripples AdminTabs visual baselines **on all configured projects**, run `git status apps/web/tests/visual/__snapshots__` and reconcile against the active project list before commit". See A1.

Amelia (Developer): "Init 21 L4 was filed as *'CANDIDATE for a future UX slice, not E33's job'*. After Init 22 it remains: not Init 22's job. But the candidate is now *more* warranted, not less."

**Root cause (named):** same as Init 21 L4 — `AdminTabs` layout response to *number-of-tabs* is currently absorbed by the visual-baseline contract rather than designed. Cost is bounded today and rising linearly per tab; defect surface (under-application of the ripple) is starting to manifest.
