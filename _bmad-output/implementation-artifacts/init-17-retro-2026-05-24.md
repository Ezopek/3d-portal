**Forward-applicable:** subagent dispatch decision matrix per story:

### §3.2 Semaphore primitive needed 3 Codex rounds — sync-error edge cases under-tested
Story 27.1 (acquireShareBlob semaphore) shipped CLEAN on Codex round-1 verdict, but two subsequent Codex re-reviews surfaced P3+P2 sync-error edge cases:
- **Round-2 P3**: sync `fetch()` throw leaked the just-acquired slot. Fix: try/catch + release in catch.
- **Round-3 P2**: re-throwing sync error bypassed AnonymousImage's `.catch()` handler + leaked `_pending` increment. Fix: convert sync throw to rejected Promise + decrement `_pending`.

Both edge cases are PRODUCTION-RARE (browsers rarely throw sync from `fetch()`; URL-parser failures are the realistic-but-uncommon trigger). But the LEAK PERMANENCE made them worth fixing — once `_concurrentFetches` or `_pending` gets stuck, every subsequent share fetch is impacted until page reload.

**Lesson:** for ANY new concurrency primitive (semaphore, lock, refcount), the test plan MUST include a "sync-throw" branch:
- Mock `fetch()` to throw synchronously (not return rejected promise).
- Acquire → assert pre-throw state.
- Trigger throw.
- Assert post-throw state matches pre-acquire state (counter, queue, pending — all clean).

Init 17 Story 27.1 lacked this test path. Codex caught both gaps via cynical review. **Forward-applicable:** add "sync-error edge case" to concurrency-primitive test checklist. Codify in `feedback_worker_single_flight` memory entry (extend with FE-side semaphore equivalent).

### §3.3 Story 28.2 round-3 — round-2 fix introduced a different P2 (dead-zone taps)
Bundle Codex (`codex review --commit df7cfa0`) returned 1×P2 on the round-2 chrome-toggle gate: `start.stripOrigin && chromeVisible` was too broad — it suppressed chrome toggle for ANY touch starting inside the strip's vertical bounds, including padded/empty areas where no thumb exists to defer to. Result: dead-zone taps on visible strip margins that should have hidden the chrome overlay but did nothing.

**Round-3 fix** (commit `71b3dda`): introduced a SECOND `thumbOrigin` flag computed at touchStart from `e.target.closest("[data-thumb-idx]")`. The chrome-toggle gate now checks `start.thumbOrigin` only (narrow signal — true only when the touch landed on an actual thumb button). The navigate-suppression path still uses the original coords-based `stripOrigin` (broad signal — covers hidden-strip area too). Two-flag design separates the two concerns cleanly.

**Lesson:** "guard against X" fixes can introduce "guard against everything X-adjacent" regressions when the guard predicate is too coarse. For interaction guards, the predicate should narrow to the specific element type the deferral targets, not the broader containing region.

**Forward-applicable:** when adding a touch/mouse interaction guard, ask "what is the exact element this guard exists to protect?" and gate on `closest('[<that-element-selector>]')` rather than a coord/region test. Region tests are appropriate only for direction-based behaviors (drag suppression, scroll-vs-swipe). The retro originally framed Story 28.2 as round-2-CLEAN; this addendum corrects that — Story 28.2 needed THREE rounds for full closure (round-1 ship → round-2 chrome-toggle race fix → round-3 dead-zone narrow).

**Round-4 addendum (Codex review of `71b3dda`, 2026-05-24):** Codex returned 1×P2 "Keep hidden-strip taps able to restore the chrome" — DISMISSED as technically incorrect after rigor-not-performance review per `[[feedback_codex_review_invocation]]`. Codex's premise: thumb tap in hidden strip would set `thumbOrigin=true` and short-circuit chrome restore. Reality: strip has `pointer-events-none` in hidden state (ImageFullscreenViewer.tsx:333) and thumb buttons do NOT opt back in with `pointer-events-auto` (line 354) — per CSS spec the parent's `pointer-events: none` blocks hit-test on children too, so a hidden-strip-area touch's `e.target` is the viewer root and `target.closest("[data-thumb-idx]")` returns null → `thumbOrigin === false` → chrome-toggle gate is bypassed → `setChromeVisible((v) => !v)` fires → chrome restored. The scenario Codex described cannot physically occur with the current CSS. **Lesson on Codex (gpt-5.4-mini) routing under interaction-guard reviews:** mini consistently spotted real bugs in rounds 2 + 3 but missed the CSS-propagation interaction in round 4 → trust-but-verify pattern holds. Three rounds of real fix-up + one round of dismissed-false-positive is acceptable Codex performance for a 3-flag interaction state machine; the four-round chain still saved net time vs an hour of manual edge-case enumeration on my end.

### §3.4 H2 backfill philosophy — compact stubs over exhaustive paste
Story 29.1 (TB-039) backfilled Init 11-15 H2 sections to canonical docs. Two approaches considered:
- **Exhaustive paste** — copy each Init's FR/NFR/Decisions verbatim from source SCP into canonical docs. ~200+ LOC per Init per doc = 12 × 200 = 2400+ LOC total. High redundancy with source SCP.
- **Compact stubs** (chosen) — ~15-30 LOC per Init per doc: status summary, source SCP pointer, brief FR/NFR list (where applicable), cross-references. ~360 LOC total across all 12 H2 sections.

Compact stubs honor the "adopt" intent (canonical docs catalog Init existence) without bloating redundancy. Future readers know where to FIND detail (source SCP); they don't need every detail recapitulated in canonical doc.

**Forward-applicable:** when an Init's authoritative spec lives in an SCP file (which is git-immutable as artifact), canonical doc H2 should be a CATALOG ENTRY pointing to SCP, not a duplicate of SCP content. Memory entry candidate? Possibly an extension to `feedback_docs_hygiene` codifying this pattern. Defer until 3rd backfill event to confirm the pattern holds.
