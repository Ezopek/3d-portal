## §3 What we'd do differently — root causes

### L1 — Cache topology was under-specified at Story 30.2 spec time (the root cause of the 7-round chain)
All six Story 30.2 P1/P2 follow-ups (rounds 2-7) live in the same problem space: **resolve-cache-and-mutation-coherence**. The spec (`30-2-conditional-render-member-share-view-info-bar.md`) prescribed concrete `staleTime` / `gcTime` / queryKey values but did not enumerate the FOUR coupled invariants that those values must jointly satisfy:

1. **Revocation/expiry must surface promptly** on next visit → force fresh GET, no staleness.
2. **Admin mutations on the same model must propagate immediately** → ride canonical cache so `useReplaceTags`/`useUpdateModel`/`useDeleteModel` invalidations reach the share view.
3. **Probe and render must be atomic** → race between two fetches surfaces wrong UX (errors.network instead of share-expired); one cache entry, one source of truth.
4. **Share-seeded cache must not contaminate subsequent /catalog/<id> visit** → unmount cleanup + sync evict on info-bar nav.

Invariants 1 + 4 want a private cache (isolated from catalog). Invariants 2 + 3 want the canonical cache. The spec landed on a configuration that satisfied 1 but broke 2 + 3 + 4; rounds 2-7 oscillated through the design space until a satisfying configuration was found.

In hindsight, an enumeration phase that named those four invariants up front would have collapsed rounds 3 + 4 + 5 into a single design choice ("canonical queryKey + force-fresh observer override"). Rounds 2 (staleTime), 6 (unmount cleanup), and 7 (sync evict on info-bar nav) were the genuinely new contracts that emerged from the actual cache semantics; even an expert SCP author would likely catch those only after sitting with the data flow for a while.

Concretely, the spec authored a **buggy line** at AC-2: *"`staleTime: 5 * 60 * 1000` (5 min — matches AuthContext meQuery staleTime)"* — that's the round-1 revocation-lag bug printed inline in the spec. The justification (matching meQuery) is a category error: auth state changes when the same user acts; share-token state changes when ANOTHER user revokes. Different invariants → different staleTime budgets. The spec author copy-paste-with-justification looked correct because the justification was nominally about "same kind of thing", but the contracts diverge.

**Root cause class:** spec-level cache-coherence enumeration miss, parallel in shape to the Init 12 use-case-enumeration miss that surfaced Init 18 itself. Both are "the thing you didn't list as something to think about" failures.

### L2 — "Reuse existing useModel + CatalogDetailBody" was the wrong default
Reuse was the right INTENT (avoid forking the catalog hook) but the wrong DEFAULT (inherit catalog's 30s staleTime). The share path has fundamentally different cache invariants from the catalog browsing path:

| Concern | `/catalog/$id` | `/share/<token>` |
|---|---|---|
| Staleness budget | 30s OK (stable model browsing) | 0s required (revocation must surface) |
| Retry policy | Default backoff OK | None — 404 is decisive (revoked/deleted) |
| Cache propagation | Mutations propagate to all viewers | Mutations propagate to all viewers (same model) |
| Cache eviction on route exit | None (stays in cache for 30s) | Required (don't contaminate next route) |

Three of four columns differ. Pure "reuse" of `useModel` would have propagated all four catalog defaults; the right pattern is *"share the queryKey, override the observer config"* — what `useShareModelProbe` ends up doing at round-5. Spec author could have arrived there directly if the table above had been authored in §AC-2 instead of the staleTime-copy-from-meQuery sentence.

### L3 — The round-1 staleTime was a spec-class miss, not a dev-execution miss
This is worth calling out separately because dev-execution miss is "developer wrote the wrong code"; spec-class miss is "spec told the developer to write the wrong code". The spec text at `30-2-...md` AC-2 line: *"`staleTime: 5 * 60 * 1000` (5 min — matches AuthContext meQuery staleTime)"*. The developer (autonomous mode, this agent) implemented exactly what the spec prescribed. Round-1 Codex caught it as P1.

Implication: post-Init-18 retros should ALSO grep the spec for "magic constants with justifications that don't actually justify them" — if the spec says "X because Y" but Y doesn't entail X for the relevant contract, that's a smell.

### L4 — Open-redirect validator missed backslash on first pass
Story 30.1 round-1 `_isSafeReturnPath` rejected `/` (single), `//` (protocol-relative), `javascript:` (scheme), control chars. Backslash `\\evil.com/path` was missed because the rejection list was built from "things I've seen in OWASP cheat sheets" rather than "all path-separator characters Chrome interprets". Codex on gpt-5.5 caught it; round-2 fix-up landed in 60 minutes.

**Why this wasn't a process collapse:** the Codex gpt-5.5 routing decision (NFR-SECURITY adjacency → escalate to heavyweight model) is exactly what the routing policy is for. The gpt-5.4-mini default would likely have CLEAN'd this. The escalation paid for itself.

**Forward-applicable:** when authoring URL/path validators, enumerate ALL platform-specific path separators (`/`, `\`, URL-encoded variants `%2f`, `%5c`) + ALL scheme indicators (`javascript:`, `data:`, `vbscript:`, `file:`) + ALL control-byte ranges, before testing against a small allowlist. The negative space is too large to test by example; allowlist-first is the right shape (`_isSafeReturnPath` already does this — must start with `/` AND not start with `//` AND not contain control chars). Backslash skipped past this because backslash starts the string with `/` if you wrote the regex wrong. Round-2 adds the explicit `/\\/.test(value) → false`.

## §4 Story 30.2 deep dive — narrative cause-effect chain

Reading the chain bottom-up (each round caused by the prior round's structural choice):

- **Round-1 cause:** spec prescribed `staleTime: 5min` for the resolve query (copy-justified from AuthContext) without checking the revocation contract.
  - **Effect (round-2 P1):** revocations don't surface for up to 5 min on revisits.
  - **Effect (round-2 P2 — independent):** trusting resolve-200 means `CatalogDetailBody` fetch can race against between-request model deletion.

- **Round-2 cause:** probe added via reused `useModel` hook (one-line `enabled` flag addition) — inherits the catalog hook's 30s staleTime + default retry backoff.
  - **Effect (round-3 P2):** if `/catalog/<id>` was visited recently, the catalog cache short-circuits the probe → revoked model still renders as valid. AND default retry backoff delays the 404 surface → revoked share takes seconds to flip to share-expired UX.

- **Round-3 cause:** probe uses share-specific queryKey `['share','model-probe',id]` — isolates from catalog cache, but disconnects from canonical `['sot','models',id]` cache too.
  - **Effect (round-4 P2 — independent):** probe and `CatalogDetailBody` still fire two separate fetches against the same endpoint with different queryKeys → between-fetches deletion shows generic errors.network in `CatalogDetailBody`'s render rather than share-expired in `MemberShareView`'s decision branch.

- **Round-4 cause:** `CatalogDetailRender({detail})` extraction means MemberShareView passes probe data directly. Atomic. ONE network round-trip. But the share-specific queryKey is still detached from canonical.
  - **Effect (round-5 P2):** admin mutations (`useReplaceTags`/`useUpdateModel`/etc.) invalidate canonical key but not share key → admin editing a model from `/share/<token>` sees stale data until hard reload.

- **Round-5 cause:** probe switched to canonical queryKey `['sot','models',id]` while keeping observer-level force-fresh override (`staleTime: 0` + `refetchOnMount: "always"` + `retry: false`). Mutations propagate. Force-fresh GET on every mount.
  - **Effect (round-6 P2):** successful share visit now seeds the canonical cache → subsequent `/catalog/<id>` visit within 30s reads share-fetched snapshot, masks post-share edits/deletes by other users.

- **Round-6 cause:** `useEffect` cleanup calls `qc.removeQueries({queryKey:["sot","models",id]})` on `MemberShareView` unmount.
  - **Effect (round-7 P1):** info-bar `<Link>` mounts `/catalog/$id` route observer BEFORE `MemberShareView`'s unmount cleanup fires → new observer subscribes to share-populated cache before evict → sticks for 30s. (`removeQueries` doesn't invalidate already-mounted observers — it only clears the cache entry; the existing observer keeps its `data` reference.)

- **Round-7 cause:** info-bar's "open in catalog" is now a `<button>` that synchronously calls `qc.removeQueries` THEN `navigate({ to: "/catalog/$id" })`. Cache emptied before catalog mount. Unmount cleanup stays in place for browser-back / direct-URL paths.
  - **Effect:** **chain closed.** Two documented residual trade-offs (browser-back path skips sync evict; warm-cache stale-while-revalidate flash). Both bounded + rare. Operator-acceptable.

The recurring pattern: each round's fix introduces the next coupling. The first three rounds (2/3/4) were about getting the probe RIGHT (staleness + retry + atomicity). Rounds 5/6/7 were about getting the cache topology RIGHT (propagation + eviction + eviction timing). These are two related but distinct design surfaces.
