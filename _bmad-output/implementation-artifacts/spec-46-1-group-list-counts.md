## Intent

**Problem:** Admins have no screen to see how tags are organized into facet groups or how many models use each tag — the only net-new admin surface in Initiative 25 is still missing.

**Approach:** Add a new admin-only, read-only screen (`/admin/tag-groups`) that lists every tag group with its tags and per-tag model counts, plus a groupless section, by reusing the existing public `GET /tag-groups` endpoint and its frontend hook. No backend changes; no write actions (rename/merge/move land in Story 46.2, duplicate detection in 46.3).

## Boundaries & Constraints

**Always:**
- Reuse `useTagGroups()` (`apps/web/src/modules/catalog/hooks/useTagGroups.ts`) and the existing `TagGroupsResponse`/`TagGroupRead`/`TagReadWithCount` types (`apps/web/src/lib/api-types.ts`) — do not add a new hook or new backend endpoint.
- Render every group from `groups[]` in `position` order with its `tags[]` (each showing `name_en`/`name_pl` per locale and `model_count`), plus a distinct "groupless"/ungrouped section for `groupless[]`. Empty groups render with an empty-state line, not hidden.
- Follow the existing admin-screen pattern exactly: route file `apps/web/src/routes/admin/tag-groups.tsx` mirroring `queues.tsx`'s auth-gate dance (`useAuth()`; `null` while loading/unauthenticated; `<Navigate to="/" replace />` for non-admins); page component `apps/web/src/modules/admin/TagGroupsPage.tsx`; register a new tab in `AdminTabs.tsx`.
- All new UI uses Tailwind semantic tokens only (`bg-card`, `text-foreground`, `text-muted-foreground`, `border-border`, etc.) — no raw hex, no `dark:` variants.
- Add i18n keys to both `apps/web/src/locales/en.json` and `pl.json` under `modules.admin.tagGroups.*` (plus one `admin.tabs.tagGroups` tab-label key), with a parity test mirroring `profile-library-i18n.test.ts`.

**Block If:**
- The `GET /tag-groups` response shape on `main` no longer matches `TagGroupsResponse`/`TagGroupRead`/`TagReadWithCount` as documented in `apps/web/src/lib/api-types.ts:90-107` — HALT rather than guess a new contract.

**Never:**
- Do not add rename, merge, move, create, reorder, or delete affordances — this story is list-only (46.2/46.3 scope).
- Do not gate the reused `GET /tag-groups` call behind a new admin-only backend permission — it intentionally stays the shared authenticated-read endpoint also used by the public catalog sidebar; only the frontend route is admin-gated.
- Do not hand-edit `routeTree.gen.ts` — it is regenerated by the TanStack Router codegen (dev/build script), not authored directly.

## I/O & Edge-Case Matrix

| Scenario | Input / State | Expected Output / Behavior | Error Handling |
|----------|--------------|---------------------------|----------------|
| Happy path | `groups` has ≥1 group, each with ≥1 tag | Groups render in `position` order; each tag shows name + `model_count` | No error expected |
| Empty group | A group's `tags` array is `[]` | Group header still renders with an inline empty-state message (e.g. "No tags in this group") | No error expected |
| Groupless tags present | `groupless` has ≥1 tag | A separate "Ungrouped" section renders below the groups, same tag-row styling | No error expected |
| No groupless tags | `groupless` is `[]` | The "Ungrouped" section is omitted entirely (not rendered empty) | No error expected |
| Loading | Query in-flight | Skeleton placeholder shown (mirrors `QueuesPage`'s `animate-pulse` pattern) | No error expected |
| Fetch failure | `useTagGroups()` returns `isError` | Destructive-styled error banner with a retry action (mirrors `QueuesPage`) | Retry re-triggers the query |
| Non-admin visits route | Authenticated non-admin user navigates to `/admin/tag-groups` | Redirected to `/` | No screen content rendered |

- Given an authenticated admin, when they open `/admin/tag-groups`, then they see all tag groups (in `position` order) with their tags and per-tag `model_count`, plus an "Ungrouped" section for any groupless tags.
