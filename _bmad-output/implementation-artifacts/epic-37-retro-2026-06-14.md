## Summary

Epic 37 was a focused operator-quality slice: the backend and frontend mutation hook for deleting model files already existed, but the admin FilesTab lacked the visible delete affordance. The epic closed that UI gap without changing backend semantics or expanding member permissions.

Delivered story/artifact:

- `37-1-stl-file-delete-ui` — delete action for admin-visible STL/source/3MF file rows, confirmation dialog, existing `useDeleteFile(modelId)` reuse, stale expanded/modal state cleanup, i18n, unit coverage, visual baseline refresh, full gate, merge, deploy, and runtime smoke.

## What went well

### 1. The slice stayed narrow

The implementation reused the existing file-delete contract:

- frontend hook: `useDeleteFile(modelId)`;
- backend endpoint: `DELETE /api/admin/models/{model_id}/files/{file_id}`;
- existing query invalidation behavior;
- no backend endpoint changes;
- no estimate data purge;
- no member exposure.

That kept the story as a UI affordance fix rather than a storage/retention rewrite.

### 2. TDD caught the real gap cleanly

The RED tests failed for the intended reason: FilesTab had no delete buttons/confirmation path. After the minimal implementation, the targeted story test passed with **26 FilesTab tests**, covering:

- admin sees delete actions for STL/source/3MF rows;
- non-admin does not see delete;
- confirmation is required before the mutation;
- the expected DELETE request is issued;
- expanded STL/viewer state is cleaned up when the deleted file was active.

### 3. Existing UI patterns were reused

The story followed the PhotosTab safety pattern instead of inventing a new destructive-action style. That made the behavior predictable and kept destructive file actions consistent across model detail tabs.

### 4. Visual regression did its job

The first full closeout gate failed only on visual regression, which was expected: adding a delete affordance changes FilesTab row actions and shared visual surfaces. Baselines were refreshed only after checking the changed snapshot scope, then the full gate passed.

## What was painful / surprising

### 1. Snapshot churn needed manual skepticism

The Playwright baseline update touched **36 snapshots**, which looked too broad for a small row-action change. The controller paused and checked the scope instead of blindly committing all visual churn. The final decision was that the affected snapshots shared the relevant catalog/detail/share surfaces and were consistent with the UI delta.

**Action:** continue treating wide visual baseline updates as suspicious until diff scope is explained.

### 2. Locale structure was easy to misread

A first edit path assumed nested locale JSON and briefly introduced a nested `catalog` object. The project actually uses flat locale keys.

**Action:** when adding i18n keys, inspect nearby keys first and preserve the flat-key convention (`catalog.actions.*`).

### 3. Shell quoting remains a footgun for TSX patches

Template strings/backticks and apostrophes were damaged by nested SSH/shell quoting during test insertion. The safer workflow was to generate a script locally or via a quoted heredoc and execute it on `.170`, rather than fighting inline shell edits.

**Action:** for non-trivial TSX/JSON edits over SSH, prefer script/heredoc writes with single-quoted delimiters or file-based prompting; avoid inline quoting heroics.

### 4. Reviewer false positives still require controller judgment

Aider returned `REQUEST_CHANGES` because it claimed `catalog.actions.delete` was missing. Direct repo verification showed the key existed in both EN/PL, and targeted i18n/ConfirmDialog/hook tests passed.

**Action:** keep using Aider as routine review input, but verify every finding against the repo before treating it as a blocker.

---

## Decisions carried forward

1. **FilesTab destructive actions remain admin-only.** Member surfaces must not receive file-delete affordances unless a future story explicitly defines a safe member contract.
2. **Use existing destructive-action primitives.** Prefer shared confirmation/dialog/mutation patterns over bespoke delete UI.
3. **Do not couple file deletion to estimate-history deletion by default.** Storage cleanup remains the backend endpoint's responsibility; estimate/history retention needs a separate product/data-retention decision.
4. **Wide visual baseline refreshes need a scope explanation.** Do not commit snapshot churn just because `--update-snapshots` produced it.
5. **Flat locale keys are the convention.** Use keys like `catalog.actions.deleteFile`, not nested JSON objects.
