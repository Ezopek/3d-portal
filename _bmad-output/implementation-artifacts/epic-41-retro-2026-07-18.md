## What went well

### 1. Safe migration sequencing was the epic's spine

The single most valuable outcome was recognizing — and resisting — the epic sketch's forward-only destructive `0018`. Two independent constraints were reasoned through against the actual code, not guessed:

- **Green-before-merge gate (41.1):** removing `Category` from the ORM breaks pytest *collection* (≈8 backend imports + dozens of fixtures constructing `Category`/`category_id`), so 41.1 had to be additive-only. The test/dev schema is built from the ORM via `init_schema()` → `create_all()`, so the additive parts are live in tests with no migration.
- **Prod deployability (41.2):** the suite never runs Alembic, so the green gate did **not** force the drop into 41.2 — but prod does run `alembic upgrade head` on any deploy during the multi-week E41→E47 window. A forward-only destructive migration on `main` is therefore a deploy landmine with no rollback. The additive/destructive split keeps "main deployable at every commit" and ORM↔migration parity throughout.

Both migrations are additive and fully reversible; the one irreversible operation was isolated and deferred to the exact point where the ORM and app references disappear with it.

### 2. Native + external review layering caught real gaps

Each story went through native BMAD adversarial review, and 41.2 and 41.3 additionally through an independent Aider review — all APPROVE. The layering earned its keep repeatedly:
- **41.2** surfaced a genuine test-coverage gap (the `tag.group_id` FK SET-NULL semantics were asserted only by source inspection), closed with an executable `PRAGMA foreign_key_list(tag)` assertion.
- **41.3** surfaced five test-quality/ergonomics findings — most notably an error-atomicity test that injected its fault in the *group* phase, making the "no orphaned tags" assertion vacuously true. All five were resolved (fault moved into the tag phase with genuine non-orphan verification; real-`IntegrityError` race branches now covered; a truthful DB-queried count added to the admin script), and a focused re-review confirmed each fix without regression.

The lanes were complementary, not redundant — native review owned AC/guardrail verification, the external lane an independent adversarial pass.

### 3. Deterministic, executable guardrails instead of prose promises

Deferrals and invariants were not left as comments to trust. `test_migration_0018` asserts `category` + `model.category_id` **still exist** after `upgrade(head)` — the destructive deferral is executably proven, so any accidental early drop fails a test. `test_seed_taxonomy_no_model_side_effects` asserts the seed touches no `Model`/`ModelTag`/`Category` rows. Index/FK names (`uq_tag_group_slug`, `fk_tag_group_id`) were pinned to match the ORM so a future `compare_metadata` drift guard has a stable target. Determinism (NFR25-DETERMINISM-1): full suite run 3× with identical pass counts on all three stories.

### 4. Reused the existing seed contract instead of inventing one (41.3)

41.3 mirrored the repo's one existing idempotent seed (`seed_admin`) verbatim — create-if-absent by unique key, never update/delete, `IntegrityError`→`rollback` tolerance — rather than designing a new mechanism. That kept the no-clobber update policy (admin edits win over re-seeds) consistent with established behavior and made the review's job a comparison, not a novel audit. The one deliberate divergence — **not** wiring the seed into the lifespan — was the correct call: auto-seeding a fixed taxonomy every boot would fight the admin governance HANDOFF §9 establishes.

### 5. Slug-as-slug discipline

Both sprint-status keys (`…-category-removal`, `…-drop-category`) and the 41.3 key (`…-optional-…`) name work the stories deliberately did **not** do (or a status that no longer applies). The team treated the keys as slugs and documented each divergence in Dev Notes + operator questions, rather than letting a stale key drive scope. Operator-accepted precedent from 41.1 carried cleanly through 41.2 and 41.3.

## Challenges / growth areas

### 1. Epic sketch encoded an unsafe sequencing

`epics.md` §E41 and `NFR25-SCHEMA-MIGRATION-1` literally map the destructive drop into 41.2 with a raising `downgrade`. Executing the sketch verbatim would have shipped a deploy landmine. Detection depended on the implementer reasoning about the deploy ritual (`docs/operations.md:234`) during story creation — it was not pre-flagged in planning. **Action for E42+:** planning-level migrations should be classified additive vs. destructive up front, with destructive DDL explicitly coupled to its ORM-removal story.

### 2. A premature closeout had to be corrected mid-epic

E41 was briefly closed `done` with 41.3 deferred to E42 (`DEFER_41_3`) before the operator reversed it. The native completion check had in fact flagged the epic as *not complete* (a non-`done` story remained) and offered a partial-retro path — that signal was initially overridden, then honored. **Lesson:** trust the completion check. An epic with a non-`done` story is not complete; "defer the last story to the next epic" is a scope decision that needs an explicit operator call *before* a retro declares closure, not a default the retro reaches for. Cost this time was low (artifact-only churn), but the same reflex on a code epic would ship a half-finished epic behind a "done" label.

### 3. ORM↔migration parity is still only partially guarded

`test_migration_0018` pins the specific new objects, but there is still no full `compare_metadata`/autogenerate diff test asserting the entire ORM matches the migration head. This has been carried since 41.1. The E42 cut-over migration is the natural moment to add a single autogenerate-diff parity test over the finished facet schema. Tracked in `deferred-work.md`.

### 4. Two `[Review][Defer]` items on the seed remain open by design

41.3's review left two low-severity items deferred (the bare `except IntegrityError` mirrors `seed_admin` and can in principle mask a non-slug integrity error; idempotency keyed on `slug` only means a DB-side *slug* rename would recreate a duplicate). Both are outside the documented governance model and spec-mandated by AC #1's "mirror `seed_admin`" requirement, so they were accepted rather than fixed. Worth revisiting if the seed mechanism is ever generalized.

## Key takeaways

1. **Reason migrations against the real deploy ritual, not the sketch.** The additive/destructive split — and coupling destructive DDL to its ORM removal — is the reusable pattern for the whole E41→E47 initiative.
2. **Make deferrals executable.** A test that proves the drop did *not* happen (and that the seed touched no models) beats a comment that promises it later.
3. **Reuse the established contract for repeat mechanisms.** Mirroring `seed_admin` verbatim gave 41.3 a known-good idempotency + no-clobber policy and made review a comparison, not a novel audit.
4. **Layer native + independent review** where the change is irreversible-adjacent; the lanes catch different classes of gap (SET-NULL semantics; a vacuously-true error-atomicity test).
5. **Trust the completion check.** An epic with a non-`done` story is not done — "defer the last story" is an explicit scope call, not a closure shortcut.
