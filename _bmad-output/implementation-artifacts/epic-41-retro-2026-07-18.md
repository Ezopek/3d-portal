## What went well

### 1. Safe migration sequencing was the epic's spine

The single most valuable outcome was recognizing — and resisting — the epic sketch's forward-only destructive `0018`. Two independent constraints were reasoned through against the actual code, not guessed:

- **Green-before-merge gate (41.1):** removing `Category` from the ORM breaks pytest *collection* (≈8 backend imports + dozens of fixtures constructing `Category`/`category_id`), so 41.1 had to be additive-only. The test/dev schema is built from the ORM via `init_schema()` → `create_all()`, so the additive parts are live in tests with no migration.
- **Prod deployability (41.2):** the suite never runs Alembic, so the green gate did **not** force the drop into 41.2 — but prod does run `alembic upgrade head` on any deploy during the multi-week E41→E47 window. A forward-only destructive migration on `main` is therefore a deploy landmine with no rollback. The additive/destructive split keeps "main deployable at every commit" and ORM↔migration parity throughout.

Both migrations are additive and fully reversible; the one irreversible operation was isolated and deferred to the exact point where the ORM and app references disappear with it.

### 2. Native + external review layering caught real gaps

Each story went through native BMAD adversarial review, and 41.2 additionally through an independent Aider review — both APPROVE. The layering earned its keep: reviews surfaced a genuine test-coverage gap (the `tag.group_id` FK SET-NULL semantics were asserted only by source inspection), which was closed with an executable `PRAGMA foreign_key_list(tag)` assertion. The two review lanes were complementary, not redundant — native review owned AC/guardrail verification, the external lane an independent adversarial pass.

### 3. Deterministic, executable guardrails instead of prose promises

The deferral was not left as a comment to trust. `test_migration_0018` asserts `category` + `model.category_id` **still exist** after `upgrade(head)` — the deferral is executably proven, so any accidental early drop fails a test. Index/FK names (`uq_tag_group_slug`, `fk_tag_group_id`) were pinned to match the ORM so a future `compare_metadata` drift guard has a stable target. Determinism (NFR25-DETERMINISM-1): full suite run 3× with identical pass counts on both stories.

### 4. Slug-as-slug discipline

Both sprint-status keys (`…-category-removal`, `…-drop-category`) name work the stories deliberately did **not** do. The team treated the keys as slugs and documented the divergence in each story's Dev Notes + operator questions, rather than letting a stale key drive scope. Operator-accepted precedent from 41.1 carried cleanly into 41.2.

## Challenges / growth areas

### 1. Epic sketch encoded an unsafe sequencing

`epics.md` §E41 and `NFR25-SCHEMA-MIGRATION-1` literally map the destructive drop into 41.2 with a raising `downgrade`. Executing the sketch verbatim would have shipped a deploy landmine. Detection depended on the implementer reasoning about the deploy ritual (`docs/operations.md:234`) during story creation — it was not pre-flagged in planning. **Action for E42+:** planning-level migrations should be classified additive vs. destructive up front, with destructive DDL explicitly coupled to its ORM-removal story.

### 2. ORM↔migration parity is still only partially guarded

`test_migration_0018` pins the specific new objects, but there is still no full `compare_metadata`/autogenerate diff test asserting the entire ORM matches the migration head. This has been carried since 41.1. The E42 cut-over migration is the natural moment to add a single autogenerate-diff parity test over the finished facet schema. Tracked in `deferred-work.md`.

### 3. Optional-but-tracked story creates a status-vocabulary edge

41.3 is "optional" in the epic but `backlog` in sprint-status, so the native retrospective's completion check counts it as pending. Closeout resolves this by keeping the canonical `backlog` status + an inline E42-linking comment rather than inventing a non-canonical "deferred" status. Worth a convention note for future optional trailing stories (see Native-workflow note below).

## Key takeaways

1. **Reason migrations against the real deploy ritual, not the sketch.** The additive/destructive split — and coupling destructive DDL to its ORM removal — is the reusable pattern for the whole E41→E47 initiative.
2. **Make deferrals executable.** A test that proves the drop did *not* happen beats a comment that promises it later.
3. **Layer native + independent review** where the change is irreversible-adjacent; the lanes catch different classes of gap.
4. **Optional trailing stories deferred to the next epic stay canonical `backlog` + a linking comment** — don't invent status vocabulary to express "deferred."
