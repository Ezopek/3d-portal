## What went well

### 1. A code-grounded dependency audit stopped an impossible sequence before code was written

42.3's pre-ready audit didn't take the epic sketch on faith — it verified, against live code at `a05a3a5`, that `CategorySummary` is embedded in the **retained** `ModelDetail`/`ShareModelView` (`schemas.py:18,167`), that `create_model`/`update_model` hard-validate `select(Category)` (`admin_service.py:161-167,220-230`), that `Model.category_id` is NOT NULL (`_entities.py:110`), and that `.190` = `main` HEAD auto-deploy with live FE/hydrate `GET /api/categories` consumers. Each coupling made "remove category in E42" a **production regression, not a test failure**. The audit converted a plausible-looking plan into a provably-unsafe one *before* a single line was written — the cheapest possible place to catch it.

### 2. Correct-course *before* implementation, not after a failed merge

The re-sequencing happened at the story-creation gate, so nothing had to be un-built. The additive stories (42.1/42.2) were already `done` and correct; only the destructive stories moved. E42 was re-scoped, not rolled back — the epic stayed green and deployable throughout.

### 3. Honest OpenAPI capability tags — auth metadata that doesn't lie

The sharpest design call in 42.4: an admin-only route must **not** advertise itself as an agent capability just to satisfy a module-scoped invariant. Rather than tag the tightened `POST /api/admin/tags` `agent-write` (to keep `test_openapi_agent_surface.py::test_every_sot_admin_route_has_agent_write_tag` green the lazy way), the story stood up a **dedicated `sot-admin-governance` router**, moved the route there unchanged (stable operation ID `admin_create_tag_api_admin_tags_post`), and kept the old router a truthful agent-write surface. Capability metadata and actual auth now agree; the agent runbook is not misled. This was a controller tightening beyond the validator's "retain `agent-write`, it's non-blocking" (I3) resolution — the stricter, more honest choice.

### 4. The explicit-null PATCH contract (D-NULLSEM-1) — caught by adversarial validation, not in production

Independent validation found that a naive "mirror `update_category`" would blind-`setattr` an explicit `slug/name_en/position: null` onto a NOT NULL column → a nonsensical **409 `slug_conflict`** (or a raw **500** for `group_position`, since `update_tag` has no IntegrityError guard). The fix: `@field_validator`s reject an *explicit* null on NOT NULL patch fields → **422** at the Pydantic layer (Pydantic v2 skips validators for omitted defaults, so *omitted* stays untouched), leaving only the genuinely-nullable `name_pl` (clear) and `group_id` (groupless). Service bodies stayed identical to the live idiom; the wrong-status path never shipped.

### 5. Deterministic, executable guardrails — carried from E41

Full backend suite 3× identical (`1761p/3s`); the FK SET-NULL delete proven by a real seed→delete→re-query survival test (not source inspection); the auth matrix seeds real admin `User` rows so the `audit_log.actor_user_id` FK is genuinely exercised on the admin path; read-after-write verified through the real 42.2 HTTP round-trip. Invariants are tests, not comments.

### 6. Live verification accounted for CSRF-before-auth

The `.190` anonymous-write probe was sent **with `X-Portal-Client: web`** so the client/CSRF gate passes and the request actually **reaches the auth layer**, returning `401 missing_access` for all four write operations. A naive anonymous probe would have been rejected earlier by the client gate and proven nothing about the auth boundary. Verifying default-deny required modeling the middleware order correctly — the boundary is confirmed *at the layer that enforces it*.

## Challenges / growth areas

### 1. Sprint-planning encoded the unsafe sequence — again

This is the second consecutive initiative epic whose **sketch mis-ordered destructive work ahead of its prerequisites** (E41's forward-only `0018`; E42's 42.3/42.5 ahead of the FE/hydrate cutover). E41 retro action item #4 was written to prevent exactly this, and it still reached a story spec — detection again depended on the create-story dependency audit rather than on planning. The correct-course fixed it *and* discharged action item #4 by assigning classification/coupling to the atomic 47.5 story, so the guard now has a concrete owner. But the recurrence says the additive-vs-destructive classification needs to happen **at epic decomposition**, not be rediscovered per story.

### 2. Reviewer disagreement required evidence-based triage, not ritual compliance

Aider returned `REQUEST_CHANGES` on 42.4. Two failure modes were available: obey it (ritual fixes that would have worsened correct code) or dismiss it (ignore a review). The controller did neither — it triaged each finding against the actual transaction boundary and the ratified contract, documented *why* each was an invalid assumption (atomic tx → no orphan audit; intentional groupless ordering; ratified 400), and made no change. This is the right outcome, but it depends on a controller with enough context to adjudicate; a less-grounded operator could have been steered into unnecessary churn. The lesson worth keeping: **a reviewer's REQUEST_CHANGES is an input to triage, not a command** — resolve it with code-grounded evidence either way.

### 3. Pre-existing green-gate debt lived on `main` and surfaced at merge time

42.1 couldn't merge until TB-054 (three hex tokens + eleven stale visual baselines, both predating the 42.1 branch) was cleared. It was handled correctly (scoped `bmad-quick-dev`, render-neutral proof, dual review, isolated commit) — but a backend-only story being blocked by unrelated web-gate hygiene is friction worth noting. The full `check-all` gate is doing its job (it refuses to merge onto a red main), yet the debt should have been swept before the initiative, not during it.

### 4. Two informational Minors deferred to E46 (by design)

42.4's native review left two pre-acknowledged Minors: (a) `detached_tag_ids` in the delete audit is unbounded by group membership (bounded in practice by the small `tag` table; `detached_tag_count` carried alongside); (b) moving a groupless tag into a group without an explicit `group_position` reuses the tag's stale `0` (per-spec `omitted = untouched`; harmless because the 42.2 read tie-breaks by slug). Neither is a defect at current scale; both are correctly E46's concern when the admin screen sets positions on move.

## Key takeaways

1. **Audit dependencies against live code before, not after.** 42.3's create-story dependency audit turned a plausible epic sketch into a provably-unsafe sequence at the cheapest possible moment. Reason about the real deploy ritual (`.190` = `main` HEAD auto-deploy), not the plan's ordering.
2. **Correct-course before implementation preserves green.** Re-scoping at the story gate meant nothing built had to be un-built; the additive stories were already correct and the epic stayed deployable.
3. **Make auth metadata honest.** A dedicated `sot-admin-governance` router beat tagging an admin-only route `agent-write` to satisfy a test — capability metadata must not lie to the agent runbook.
4. **Reject explicit nulls at the schema layer.** D-NULLSEM-1 kept a wrong-status PATCH path (409/500) out of production; adversarial validation, not users, found it.
5. **Verify boundaries at the layer that enforces them.** The `.190` default-deny probe had to pass the CSRF/client gate (`X-Portal-Client: web`) to actually reach — and confirm — the auth boundary.
6. **A reviewer's REQUEST_CHANGES is triage input, not a command.** Invalid findings were dismissed with code-grounded evidence (atomic tx, intentional ordering, ratified 400), not ritual fixes — and the reasoning was written down.
