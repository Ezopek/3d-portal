## Decisions formalized at closeout

### E43 is complete on the additive scope — and, unlike E42, carries no relocated keys of its own

The completion check sees 43-1/43-2/43-3 all `done` with no non-`done` E43 story. The destructive category work was never numbered inside E43 in the corrected plan — the correct-course re-scoped the *sketches* to additive-only and pointed the deletions at existing 47.4/47.5 stories, so E43 owns exactly what it delivered. `epic-43` → `done`; `epic-43-retrospective` → `done`. This is the E41-retro completion discipline satisfied trivially (no force-flip, no superseded-key reconciliation).

### The category surface stays a zero-code compatibility bridge

E43 added the tag surface **alongside** the category surface without touching it. The still-live category types/hook/field/URL param are the bridge; they stay consumed until `CatalogList` (44.3), `ModelHero`/detail (45.2), and `AddModelForm` (45.3) migrate and the terminal 47.4/47.5 cutover retires them. Removing any of them in E43 would have desynchronised `api-types.ts` from the live wire (a dishonest contract denying a field the server still sends) and broken production consumer compilation.

### Destructive category retirement stays deferred and un-granted

The FE `CategoryNode`/`CategoryTree` type + `useCategoriesTree` hook deletion is 47.4 (reversible-by-revert, gated on zero remaining consumers). The `CategorySummary` / `ModelDetail.category` / `ModelSummary.category_id` drop is the atomic 47.5 commit coupled to the backend ORM + `0019_drop_category` removal, behind a **DEFERRED destructive-go gate** (verified pre-`0019` `.190` DB backup required). **Not granted by this closeout.**

### `tag_ids` `UUID_RE` is a deliberate canonical-only subset

43.3's `UUID_RE` accepts only the canonical 8-4-4-4-12 hyphenated form — narrower than pydantic `uuid.UUID` (which also accepts hyphenless/braced/urn). The drop is one-directional and safe: a repo sweep confirmed all real `tag_ids` are minted canonical, so no non-canonical id reaches `validateSearch`. Documented inline; flagged for E44 only if a non-canonical id source ever appears.

## What went well

### 1. The create-story dependency audit caught a third stale-destructive sketch before code

Story 43.1's pre-ready audit did not take the epic sketch on faith. It verified against live code that the backend `TagRead` ships `group_id`+`group_position` only (no `group`; 42.2 D-SHAPE-1), that `CategorySummary` is embedded in the retained `ModelDetail`/`ShareModelView`, and that six production FE consumers (`ModelHero`, `AddModelForm`, `CatalogList`, `CategoryTreeSidebar`, `useCategoriesTree`, `share/$token`) still import the category types. Each made "remove category in E43" a compile-time production break, not a test failure. The plausible-looking sketch became provably-unsafe *before* a line was written.

### 2. Correct-course before implementation kept the epic additive and green throughout

Because the fix landed at the story-creation gate, nothing had to be un-built. The correct-course was a MINOR planning re-scope (no new story IDs, no FR/NFR change, no architecture-decision reversal) that merely *aligned* the stale E43 sketches with the already-approved E42→E47 relocation. E43 was re-scoped, never rolled back — every story stayed deployable.

### 3. Type-layer TDD used the gate that actually fails

All three stories are type-heavy, and the team was precise about which gate is the real RED. A type-only `import type` of a not-yet-exported symbol, or an `expectTypeOf(...).toEqualTypeOf<…>()` mismatch, fails `tsc -b` (which compiles `*.test.ts`) but passes vitest-run (esbuild erases `import type`/`expectTypeOf` at runtime). 43.2 went further and picked `model_count` — the *only* structural difference between `TagRead` and `TagListItem` — as the framework-independent RED, explicitly noting that `toEqualTypeOf`'s DeepBrand strict-equality (not mere assignability, since the two arrays *are* mutually assignable) is what discriminates the shapes. RED was genuine, not masked.

### 4. Additive-only diffs, category bridge untouched by construction

Each story shipped the smallest correct diff and proved the bridge intact mechanically: 43.2's `git diff -- useCategoriesTree.ts` empty; 43.3's `category_id`/`q`/`status`/`source`/`sort`/`page` branches byte-for-byte; no `api-types.ts`, backend, migration, locale, route-tree, or visual-baseline file touched outside the additive surface. The "preserve the bridge" constraint was enforced by explicit fenced-file byte-equality guards, not by intent.

### 5. Serialization verified against the *resolved* dependency, not the declared range

43.3 proved `@tanstack/react-router` search serialization (`tag_ids`→URL-encoded JSON array, `tag_match=any` bare, `untagged=true` boolean, defaults omitted) against the actually-installed `1.169.2`, and made the normalized-state-vs-serialization-vs-address-bar distinction explicit in the AC so it did not over-claim a physical URL scrub that this no-navigation story never performs. Verifying the resolved version caught what a declared-range check would have missed.

### 6. Clean completion check — no closure reflex needed

Unlike E42 (which had to justify two `backlog` superseded keys), E43 closes on an unambiguous 3-of-3 `done`. The correct-course that pointed deletions at 47.4/47.5 kept those as *pre-existing* owners, so no E43 key had to be force-flipped or reconciled. The completion-check discipline held with nothing to override.
