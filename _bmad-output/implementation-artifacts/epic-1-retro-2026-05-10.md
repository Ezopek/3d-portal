## What Worked Well

### Phase 0 dry-run as a real discovery story (axis 1)

Story 1.1's spec assumed binary outcome (HAPPY-PATH or FALLBACK-PATH = issue #299). Reality was a **third category** — HTTP 413 Payload Too Large at the chunk-upload step, *upstream of* the `assemble` endpoint where #299 fires. Root cause: GlitchTip server returned the public proxy URL on the chunk-upload config endpoint, and the public nginx proxy capped body at 1 MB. Story 1.1 stopped at this unknown unknown, surfaced it as `phase0-result.md`, and required operator decision A/B/C.

- **Operator decision:** Option B — bump `client_max_body_size` to 50 MB on chunk-upload nginx location (configs repo, deployed via that repo's `sync.sh`).
- **Re-run #2 outcome:** HAPPY-PATH — chunk upload + artifact bundle, debug IDs uploaded successfully, release registered.

Two axes of value:
1. **The dry-run gate worked as designed.** "Stop at unknown unknowns" was upheld by an autonomous AI agent — the agent did not guess A/B/C, it surfaced the choice. Phase 0's primary value was proving the discovery mechanism, not gating.
2. **Worktree isolation was load-bearing.** `git worktree add --detach ../3d-portal-phase0 HEAD` kept dry-run lockfile + plugin-install artifacts off `main`. Two iterations of Phase 0 (re-run #1 → operator gate → re-run #2) cost zero `main` pollution.

### Architecture's "predecessor as documented manual recovery" pattern materialized (axis 2)

Story 1.6 did NOT delete `infra/scripts/upload-sourcemaps.sh` after the plugin became active. Instead, the script got a structural reframe per FR25 + FR26:

- Header block reframed as documented manual recovery (purpose, when-to-use, prerequisites, exit codes, example, recovery context).
- `--help` / `-h` flag re-prints the header content (single source of truth for docs).
- RELEASE expression aligned with `apps/web/src/release.ts` (FR26 — drift-impossible: both pipelines compute `${pkg.version}+${git_short_sha}` from the same package.json + `git rev-parse --short HEAD`).
- Documented exit-code map (0 success / 1 generic / 2 unreachable / 3 auth).

This pattern was **promoted to project-context.md as a repo-wide rule** (Story 3.4):
> "Every replacement keeps its predecessor as documented manual recovery for one release cycle."

This is the largest architectural artifact of Epic 1 — applies to any future swap of an active mechanism (queue, auth, rendering, etc.).

### State-tracking discipline for cross-story file edits (axis 3)

Four files in Epic 1 were modified three times each:

- `apps/web/vite.config.ts` — Stories 1.3, 1.4, 1.5
- `apps/web/Dockerfile` — Stories 1.3, 1.4, 1.5
- `infra/docker-compose.yml` — Stories 1.3, 1.5, 1.6
- `infra/scripts/deploy.sh` — Stories 1.3, 1.5, 1.6

Each affected story's Dev Notes carried a "Files being touched — current state (post-N) and what changes" section explicitly listing:
- Current state coming in (what previous stories landed)
- What this story changes
- What is preserved (and why)

Delta-on-delta reasoning held up across all 3 cycles per file. **Worth preserving as an explicit pattern for any epic that touches the same file multiple times.**

### TDD red-red-green sequence (Story 1.2)

Story 1.2 documented two distinct red phases:
1. **Red 1:** ImportError (`release.ts` missing; new vitest test fails to load).
2. **Red 2:** ReferenceError (`__PKG_VERSION__` not defined — vitest config did not yet have the `define` block).
3. **Green:** After mirroring `define` block from `vite.config.ts` to `vitest.config.ts`, all 3 tests pass.

Each red→green transition produced a verifiable, distinct failure mode. The two-step red sequence surfaced a real spec gap (vitest needed its own `define`) that a single-shot test would have papered over. This is the canonical implementation of project-context's "don't skip red phases — they catch test-vs-impl inversions" rule.

## What Didn't Work / Friction Points

### Friction 1 — 12h "shipped but not yet fixed" window (axis 1, primary)

Between commit `9e69e62` (Story 1.6 deploy) and commit `40344de` (catch-up fix), ~12 hours elapsed. During that window production carried known defects:

- **FR4 latent bug** (Story 1.5 P1): empty/missing `SENTRY_AUTH_TOKEN` would let Sentry plugin warn-and-skip rather than hard-fail. Production token was always valid → bug never fired, but the discipline failed silently.
- **Manual fallback could "succeed" without maps** (Story 1.6 P1): if operator ran `upload-sourcemaps.sh` after a fresh build, plugin's `filesToDeleteAfterUpload` had already removed `*.map` files; CLI would upload `.js` only.
- **Exit-code mapping unreachable under `set -e`** (Story 1.6 P2): documented exit-code semantics (0/1/2/3) were never enforced in the script body.

**Diagnosis:** the autonomous overnight batch shipped 6 stories successfully, but Codex reviews ran in a separate operator-driven session the following morning. The fix commit `40344de` followed. The window between the last Epic 1 deploy and the catch-up fix was ~12h. Pattern viable, but "done" status across stories was prematurely set on dev-commit, not on review-fix completion.

**This is the central friction Epic 1 surfaces and Epic 2 already implicitly fixed by running per-story review-fix cycles in-band.**

### Friction 2 — Codex caught spec↔impl drift, missed spec↔reality drift (axis 2)

Three production defects, only one caught by review:

| Defect | Codex review caught | Why / why not |
|---|---|---|
| FR4 not enforced (latent) | ✅ Story 1.5 review HIGH | Spec-vs-impl: AC5 said FR4 enforced; impl did not enforce when token empty/missing |
| `sourcemapPathTransform` missing | ❌ All 5 reviews missed | Spec-vs-reality: no review traced symbolicated frame paths against NFR-R1 regex; the gap was not in the diff, it was in the absence of a check |
| GlitchTip homelab volume mount | ❌ N/A (out-of-repo) | Out of review surface entirely |

Codex review is excellent at catching contract gaps in the diff. It cannot catch "deployed bundle does not actually symbolicate" — that requires end-to-end smoke (Story 3.1's verify ritual). **Two of three production defects were invisible to any plausible review of the diff.**

This is the strongest empirical case for the project-context rule (added in Story 3.4):
> Run `verify-symbolication.sh` after every deploy until CI auto-verify lands.

Epic 1 is the canonical proof that Codex review is **necessary but not sufficient** — verify ritual closes the spec↔reality gap.

### Friction 3 — Pre-existing issues escalating without promotion (axis 3, backwards-validates Epic 2 retro)

The "pre-existing issue flagged in N stories" pattern Epic 2 retro called out was **already present in Epic 1, just without a mechanism to surface it**:

| Issue | First flagged | Stories that re-flagged in Epic 1 | Status at Epic 1 close | Resolution |
|---|---|---|---|---|

| arq-worker `WorkerSettings.redis_settings` classmethod bug (commit `a63481c`) | Story 1.4 dev notes | 3/6 (1.4, 1.5, 1.6) | Open | Fixed in commit `bcabd75` between Epic 1 and Epic 3 |

### Friction 4 — Conditional stories (1.4, 1.5) gated on Phase 0 outcome (axis 4)

Stories 1.4 and 1.5 were authored as full-detail specs in `epics.md` BEFORE Phase 0 had run. Phase 0's outcome could have been fallback-path → 1.4 + 1.5 closed as won't-ship. This worked out (re-run #2 was happy-path), but the upfront authoring cost would have been wasted under fallback-path.

**Pattern observation:** "ungated story authoring" — authoring a follow-up story whose existence depends on a gate-story's outcome. Cost is wasted if gate fails. The mitigation here was Phase 0's binary HAPPY/FALLBACK contract, but reality was a third category requiring operator gate, which is when the cost manifested differently (Phase 0 took two runs + operator decision time, but 1.4/1.5 specs stayed valid).

### Friction 5 — Mixed code-review formats across stories (axis 5)

Story 1.2 review prompt used `bmad-code-review` skill. Stories 1.4-1.6 reviews explicitly opted out: "DO NOT use bmad-code-review skill — direct lean review". The format drifted toward lean direct-prompt over the course of Epic 1.

**Both formats produced equally precise findings.** Lean format had less ceremony, shorter output. `bmad-code-review` skill format had more orchestration but no measurable advantage for ad-hoc per-story reviews. **Lean wins on signal-to-noise for autonomous review sessions.**

## Decisions Made in Retrospective (Project Lead deferred to Amelia)

### A1 — Batch close-out rule (highest-impact decision)

**Adopted:** story `done` status requires review-pass (Codex APPROVE OR all P1/P2 findings fixed). Sprint-status's `review` status is the holding-pattern between dev-commit and review-fix-commit. **A batch (autonomous or otherwise) does NOT close on the last deploy — it closes on the catch-up fix-commit after all reviews complete.**

- Epic 1's `40344de` was the de-facto implementation of this rule. The rule's value was the timing: the rule formalizes that the 12h window must close before the next epic begins, not "before someone notices".
- **Implementation:** documented in project-context.md (cross-tool rule, applies to Claude AND Codex sessions). NOT a skill customize — this is sprint-status / status-tracking discipline, not orchestration.
- **Operational shape:** when an autonomous overnight batch ships N stories, the next-day operator session must run the review pass + catch-up fix commit BEFORE marking any story `done`. The intermediate `review` status holds the loop open.

### A2 — "Current state (post-N)" pattern as story-create discovery rule

**Adopted:** customize `bmad-create-story` skill to enforce: when a story modifies a file already touched by a previous story in the same epic, Dev Notes MUST include a "Files being touched — current state (post-N) and what changes" section listing current state, what changes, what is preserved.

- Implementation: customize entry in `_bmad/custom/bmad-create-story.toml` (or its user variant).
- **Backwards-validation:** Epic 1 stories 1.3 / 1.4 / 1.5 (vite.config.ts edited 3 times) and 1.3 / 1.5 / 1.6 (docker-compose.yml + deploy.sh) are the canonical examples this rule formalizes.
- **Success criteria:** next epic touching a file 3+ times has explicit state-tracking blocks in stories 2 and 3.

### A3 — Third-category gate-stories pattern

**Adopted as template note (lighter touch than A1/A2).** Add to bmad-create-story templates a note: *"Gate / experimental / dry-run stories should plan three outcome categories (HAPPY / FALLBACK / THIRD-CATEGORY → operator decision), not binary."*

- Implementation: short addition to bmad-create-story story-template prose. No skill orchestration change.
- **Trigger condition:** any story whose stated AC includes "if X happens, follow-up stories close as won't-ship; if Y happens, proceed."
- **Backwards-validation:** Phase 0 (Story 1.1) third-category outcome is the canonical example.

### A4 — Code-review format

**Adopted:** lean direct-prompt format (Stories 1.4-1.6 pattern) for ad-hoc per-story Codex reviews. `bmad-code-review` skill remains valid for user-invoked workflow reviews (e.g., end-of-epic comprehensive reviews).

- No new artifact; this is a usage convention.
- The 5 Epic 1 reviews (2 with skill, 3 lean) established that lean is faster + equally precise for per-story reviews.

### A5 — QD-3 (CardCarousel) priority

**Decision:** retain Epic 2 retro sequencing (QD-1 → QD-2 → QD-3). However, **flag in this retro that CardCarousel now has 9 flags total** (5 in Epic 1 + 4 in Epic 2) — well beyond the 3-flag threshold. Re-evaluate priority after QD-1 ships; if CardCarousel still hasn't been touched after QD-1 + QD-2, escalate ahead of any further work.

- Sequencing argument from Epic 2 retro (QD-1 deploy-critical workaround pays interest; QD-2 zero-change atomic; QD-3 needs diagnosis) still holds.
- 9-flag count is a STATE we're noting, not an immediate re-prioritization.

## Key Takeaways

1. **Autonomous overnight batch is a viable workflow pattern, but its close-out is the catch-up fix-commit, not the last deploy.** Epic 1's `40344de` was the de-facto implementation; Epic 1 retro formalizes it as a project-context rule.

2. **Codex review catches spec↔implementation drift; verify ritual catches build↔runtime drift.** Epic 1 shipped 1 latent bug Codex caught (FR4) AND 2 production-affecting bugs Codex did NOT catch (sourcemapPathTransform, GlitchTip volume mount) — both surfaced only by Story 3.1's verify ritual. Both quality gates required.

3. **Phase 0 dry-run gate's value was proving discovery is non-binary.** Original spec assumed HAPPY/FALLBACK; reality was a third category requiring operator decision A/B/C. "Stop at unknown unknowns" upheld by autonomous AI agent. Pattern: gate stories must plan three outcome categories.

4. **Architecture's "predecessor as documented manual recovery" pattern materialized in Epic 1's Story 1.6** and was promoted to a repo-wide rule (Story 3.4). Epic 1 is the canonical example.

5. **Pre-existing issues without a promotion mechanism escalate silently across an epic.** CardCarousel × 3 was re-flagged 5 times in Epic 1 alone. Epic 2 retro's "3-flag → candidate" threshold is backwards-validated by Epic 1's data.

6. **State-tracking discipline ("current state post-N") for cross-story file edits is worth preserving** — Epic 1's vite.config.ts ×3 / Dockerfile ×3 / docker-compose.yml ×3 / deploy.sh ×3 surface proved delta-on-delta reasoning scales when each story explicitly tracks state coming in + what changes + what is preserved.
