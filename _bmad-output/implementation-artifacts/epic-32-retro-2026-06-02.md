## §2 What went well

### W1 — The clean dependency chain held end-to-end; each seam was honored, not bypassed
The architecture was authored as a strict chain (resolver → worker → cache → recompute → override → FE) with explicit seams between stories: 32.2's `GcodeSink` no-op default became 32.3's parser sink; 32.1's `NoopOverrideProvider` became 32.5's real provider; 32.4's `affected_keys` caller-supplied parameter became 32.5's classifier output; 32.6 CALLED the engine modules (`git diff main` 0 lines on resolver/recompute/estimate_store/models) rather than editing them. Every story's AC-9 scope fence (NO routes / NO Alembic / NO config slot / NO reshape, as appropriate) was grep-asserted and held. This is the payoff of authoring the seam at spec time: six stories composed without a single retro-grade integration surprise.

### W2 — Review caught the runtime-invariant class spec inspection structurally cannot
Three of the six stories (32.3, 32.4, 32.6) had independent review surface a **real** blocker that no amount of spec enumeration would have caught by inspection — each is a runtime invariant:
- 32.3: non-finite required numerics passing the parser; missing `EstimateRecord` failed/fresh invariants; a TOCTOU race in the fresh-no-op.
- 32.4: `invalidate()` using one hash for both stale-marking and re-enqueue (the AC-5 Critical).
- 32.6: the production read path mutating the real bundle store on a content miss.
In every case the fix landed under TDD with a regression test **proven red pre-fix / green post-fix**, and re-review APPROVE'd the fixed tree. The "trust-but-verify review; act on real findings under TDD" pattern (carried from Init 17/18/19) held for a data-integrity-heavy epic.

### W3 — The "never a silent zero" doctrine is load-bearing across the whole estimate pipeline
From the worker's typed `SliceFailureReason` taxonomy (32.2) through the parser's `None`-not-`0` failure classification (32.3) through the recompute engine's raise-on-non-finite-price guard (32.4) to the FE's finite-number-guard em-dash rendering and honest per-state display (32.6), a missing/garbled/degraded estimate is **always** surfaced as a typed failure or an explicit state — never a fabricated `0`. For an owner-facing cost figure this is exactly the right invariant, and it was applied uniformly rather than per-story.

### W4 — The R1 self-DoS guard was designed in, not retrofitted
Cost-only changes (a Spoolman price tick) recompute by **arithmetic in-place** (`filament_g × price_per_gram`, <1s, no enqueue, no Orca subprocess); only slice-affecting changes (geometry, mapped vol-speed/temp/density, bundle re-tune, Orca upgrade) re-enqueue. This is the difference between "a price edit on `.190` quietly recomputes a cached number" and "a price edit triggers a re-slice storm". The guard is enforced at the dispatch chokepoint (32.4) and the classifier (32.5), and `spoolman_cost_only` *raises* if a caller tries to smuggle a new `bundle_hash` through the cost-only path.

### W5 — The settings/env/compose drift gate earned its keep on 32.1
`check-settings-env-compose.py` failed the 32.1 pre-push because three new Settings fields were in `env.example` but not in both compose env blocks. This is the mechanical class that silently breaks `.190` at runtime (a Settings-validation fail on first start — exactly what bit the configs-side worker in 32.2's `979f4a8` hotfix). The gate caught it at author time. Keep it.

### W6 — Honest deferral discipline: every fence is recorded in deferred-work.md with promotion triggers
SPOOL-EVT-1 (32.5), EST-RECOMPUTE-1 + EST-INGEST-1 (32.6), and SW-DEPLOY-1 (32.3) are all recorded in `deferred-work.md` with enough context to promote without re-deriving — including explicit promotion triggers and, crucially, the **deploy entanglement notes** (which deferred items re-open the SW-DEPLOY-1 window). 32.6's Blocker 2 was specifically "you deferred AC-1b but didn't write it down" — and the fix was to write it down. The epic does not leave undocumented holes.

## §3 What we'd do differently — root causes

### L1 — SW-DEPLOY-1 is a structural deploy gap that bit on 32.3 and silently threatens every slicer-touching deploy
The base `deploy.sh` rebuilds/restarts only the base stack; the slicer-worker **overlay** lives configs-side and is not touched. So any story that lands `app.modules.slicer.*` code (or bumps the `portal-api` base image the worker layers on) deploys "green" while the worker silently runs **old modules** — the failure only surfaces when a slice/estimate job runs, well after deploy is reported done. This hit 32.3 (caught only because the controller ran an in-container import smoke) and is, by construction, a live risk for 32.4 and 32.5 (both land worker-executed code paths). **Root cause:** the deploy automation has no awareness of the configs-side overlay's dependency on the `portal-api` image. **This is not a process miss — it is unfixed infra tech debt** (crosses the HC2 repo↔configs boundary, so it needs coordinated work). Until automated, every slicer-touching deploy MUST be followed by the manual overlay rebuild + in-container import/Orca/parser-or-resolve-override smoke. **See § §5 — this is the recommended next slice.**

### L2 — Story 32.1's first shallow implementation was off-contract and had to be reworked from scratch
The initial 32.1 dev pass shipped a flat single-PLA profile with fabricated AC numbering — REQUEST_CHANGES'd as off-contract, then reworked under strict TDD to the actual Decision AH contract (recursive inheritance merge, canonical hash, append-only store). **Root cause:** the first pass treated the resolver as simpler than the architecture decision specified — it built to an imagined contract rather than the authored one. The rework was correct and the gate caught it before merge, but the lesson is that Decision-class stories (AH/AI/AJ here) need the dev pass to read the architecture decision as the binding contract, not the story title. No new process artifact needed — the review gate did its job — but worth flagging that the first-pass cost was real.

### L3 — The 12 AC-8 caplog failures on 32.5 are a recurring test-harness footgun, not a logic bug
32.5's full-backend run surfaced 12 failures, **all** AC-8 observability assertions, because `configure_logging` evicts pytest's caplog root handler suite-wide (`root.handlers[:] = [handler]` via the client fixture lifespan), leaving `caplog.records` empty after any app-touching test. One test was even vacuously false-passing. This same root cause hit Init 19 (Story 31.x used the `_CaptureHandler` pattern in `test_spools.py`) and bit again here. **Root cause:** there is no shared, discoverable "use `_CaptureHandler`, not caplog, for log assertions in app-touching tests" convention captured where a dev pass will see it. **Forward-applicable lesson:** this is now the second initiative where the same caplog-eviction footgun cost a fix pass — worth a one-line testing-convention note (candidate, see § §6) so the third occurrence is avoided rather than re-diagnosed.

### L4 — Native Codex review was not available this initiative; all reviews ran on delegates/fallbacks
Per the story rows, Gemini CLI was unavailable on this host and native Codex was not relied upon; reviews ran on the Hermes gpt-5.5 delegate and Claude Sonnet 4.6 via `feature-dev:code-reviewer`. Each story labeled its reviewer identity honestly in the closeout. Per AGENTS.md § Autonomous development mode, native Codex is contractually mandatory only for NFR-prefixed security gates with countersignature — none of the Init 20 stories are NFR-SECURITY gates (the one auth-adjacent surface, 32.6's `/api/estimates`, is a read-only authenticated route with `_PUBLIC_ROUTES` untouched). So labeled-fallback was admissible and the audit trail is unambiguous. **This continues the Init 19 § L4 finding** that the native-Codex MCP path is intermittently unavailable; the operational workaround (spec-tagged routing + labeled fallback) keeps working. Carry-forward: the reviewer-availability instability is now N≥2 initiatives running — it is the steady state, not an anomaly.

### L5 — Story 32.6 shipped a display surface that has no live data source yet (by design, but worth naming)
32.6's read seam resolves by a **supplied** `stl_hash`; there is no catalog→`stl_hash` linkage (EST-INGEST-1) and no live Spoolman event source (SPOOL-EVT-1). So today the estimate display is a correct, honest **shell** — it renders every state truthfully but, for real catalog parts, there is nothing to render until ingestion lands. This is the right MVP boundary (the SCP is explicit: per-STL only, supplied hashes, ingestion is a separate story), but it means **the epic's user-visible value is gated on a follow-up**. This is not a defect — it is the deliberate consequence of slicing the MVP at "primitives + display shell" — but the retro should name it plainly so the next-slice decision is made with eyes open (see § §5).
