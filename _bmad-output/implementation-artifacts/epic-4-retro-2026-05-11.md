## Decisions

### D1 — Spec drift correction in same session is canonical

**Pattern:** Story 4.1 found that the planning docs (prd/architecture/epics) described auth as "Authorization: Bearer $(cat agent.token)" while the actual code (apps/api/app/modules/sot/admin_router.py + bootstrap_agent.py) uses cookie-based auth. The drift was caught during create-story phase, BEFORE any code was written.

**Resolution chosen:** rewrite the planning docs to match code reality in the same session, before authoring the story file. Three planning-artifact commits + one story-file write, all consistent.

**Codification:** when create-story analysis reveals planning docs contradict code reality, the correct response is to fix the planning docs first (Path A from Story 4.1's pre-flight) rather than ship a story file that perpetuates the drift OR write code that matches the (wrong) plan. Spec ↔ reality wins over spec ↔ spec consistency.

**Memory candidate (already captured in feedback_collaboration_division.md autonomous-mode update + this retro):** treat strict textual rules + grep-heuristic mechanizations as the contract; if Codex flags a strict reading violation that the heuristic would have allowed, apply the strict tightening (per Story 4.3's P2 NFR8 finding).

### D2 — Doc-only commits that change API-image content need explicit deploy

**Pattern:** Story 4.5's final commit (691e4f0) was a `docs(agents)` change to `docs/agents-add-model-runbook.md`. By project memory `feedback_auto_deploy_dev.md`, doc-only commits skip auto-deploy. BUT Story 4.2 wired the runbook into the API image via Dockerfile COPY — so the live `/agent-runbook` would serve stale content until the next non-doc commit triggered a redeploy.

**Resolution chosen:** manually run `infra/scripts/deploy.sh` after the doc-only commit. Documented in the commit message body so future operators know to do the same.

**Codification:** when a doc-only file is COPY'd into a service image as static content, doc-only commits to that file have a hidden code-deploy dependency. Until the auto-deploy rule is broadened (or a smarter rule is added that watches Dockerfile COPY paths), the manual-deploy-after-doc-only-commit-on-runbook is the documented workaround.

**Action item:** **AI-1** — consider extending `feedback_auto_deploy_dev.md` rule to: "doc-only commits skip auto-deploy EXCEPT for paths COPY'd into a service image". Alternative: deploy.sh detects "did the COPY'd source change since last deploy" and warns. Defer until second incident — Ezop's call.

### D3 — Symbolication verify false-negative on partial-image deploys is known-tolerable

**Pattern:** Story 4.5 deploy (691e4f0 — runbook content change only) triggered API image rebuild but NOT web image rebuild. `verify-symbolication.sh` polls GlitchTip for an event tagged with the new release identity (`0.1.0+691e4f0`); since the web bundle wasn't rebuilt, no web-side event with the new release tag arrives within the 30 s budget → timeout → FAILED state marker.

**Resolution chosen:** accept as known false-negative. Verify is non-fatal per NFR-R3; the runbook fingerprint verify (also wired in deploy.sh post-Story-4.2) DID pass and is the relevant signal for the runbook deploy. The false-negative state marker (`infra/.last-verify` FAILED) gets overwritten on the next code-deploy that does rebuild the web bundle.

**Action item:** **AI-2** — when deploy.sh rebuilds API but skips web rebuild (cached), `verify-symbolication.sh` should either (a) skip itself, or (b) accept release-tag mismatches if the web bundle's mtime predates the deploy. Defer until the noise becomes operationally annoying.

### D4 — Two-pass smoke-test (strict-runbook fail + adapt success) is the right shape

**Pattern:** Story 4.5 ran the runbook auth section literally (`cat ~/.config/3d-portal/agent.password`) — failed because operator's file is at `agent.token` JSON. Adapted via `jq` against the actual file — succeeded. Both attempts captured in the transcript.

**Why this beats either pure-strict OR pure-adaptive:** pure-strict would block at step 1 + produce a transcript that says "runbook is broken" without showing what the agent's adaptation produces. Pure-adaptive would silently work around the gap and the transcript would never surface R-1 as a documented blocker. The two-pass shape catches BOTH the gap AND validates that an agent CAN adapt.

**Codification:** future smoke-tests / acceptance runs follow the same shape — strict-runbook attempt FIRST, adapt only if strict fails, document both outcomes inline in the transcript.

### D5 — Worked-example IDs must be verified against the live source at runbook-write time

**Pattern:** Story 4.1 pinned Printables 661995 as "Stanford Bunny — long-lived public model safe to pin". Codex (in 4.1 review) flagged the legacy mutation signature as P2-deferred but did NOT verify the model identity. Story 4.5 smoke-test discovered 661995 is actually a firearm accessory.

**Resolution chosen:** swap to Printables 1000 (Prusa MK3 LED Lamp by Prusa) — verified live during smoke-test, neutral, topical, small files. R-3 fix in commit 691e4f0.

**Action item:** **AI-3** — when adding worked examples that pin external IDs (Printables, GitHub, Wikipedia, etc.), curl/fetch the actual content during runbook-write and verify the example matches the prose. Don't rely on memory of "what model X is".

## Process observations

### What worked well

- **Codex review on every commit set** — even the smallest-scope stories (4.4 doc-only) got high-yield findings. Cost is ~3-5 min per review against a ~10-15 min initial dev pass; ROI is overwhelming.
- **Decision-doc-first for irreversible cleanup (Story 4.4 / Decision H)** — captured the "DROP" recommendation with all three audit inputs + alternatives + follow-up plan in 148 lines (then 179 after Codex fix-up). Future archaeologist gets the "why" in one read.
- **Two-pass smoke-test (Story 4.5)** — caught + documented the R-1 BLOCKER while still completing the end-to-end flow. See D4.
- **In-band fix-up commits** — every Codex review fed directly into the next commit. No "deferred fix-up" debt accumulated. Each story closed with `done` status reflecting reality.
- **Autonomous mode handshake (Ezop's directive at session start)** — single product-question pause for Story 4.1 spec drift Path A vs B, plus the Story 4.5 file-format BLOCKER that was self-resolved by R-1 fix. Otherwise zero per-story handshakes.

### What could improve

- **Pre-story planning-artifact verification** — Story 4.1 spec drift could have been caught at planning time (during the original epics.md authoring), not at create-story time. The `bmad-create-story` skill currently doesn't have an explicit "verify auth claims against code" step. Could add one. Defer until second incident of plan-vs-code drift.
- **Worked-example verification at write time** — see D5 + AI-3. Single-shot fix when authoring; recurring discipline if not codified.
- **Doc-only-but-API-image deploy ergonomics** — see D2 + AI-1. Manual workaround documented in commit messages; could be automated.

### Loop quality

- **Same-session loop quality:** very high. Every story closed `done` in the same session it was created. No `review` holding-pattern carryover between stories. The `bmad-create-story → bmad-dev-story → Codex → fix-up → done → next` loop ran 5x without breaks.
- **Cross-session loop quality:** N/A this epic — entirely same-session.
