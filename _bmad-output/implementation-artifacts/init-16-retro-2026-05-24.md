## §3 What we'd do differently / new debt

### §3.1 Story 22.3 needed 3 Codex rounds — operator-mobile-edge-case territory under-tested by subagent
The fullscreen image viewer landed clean on round-1 except for touch-interaction edge cases that surfaced in rounds 2/3:
- **Round-2 P1**: viewer strip's eager-mount of all thumbs bypassed Story 22.2 round-2 LazyAnonymousImage mitigation on /share/ (60+ image carousels would re-introduce 429). Fix: optional `renderThumb` prop.
- **Round-2 P2**: viewer touch handler at root captured strip-area swipes, breaking native horizontal strip scroll. Fix: stripRef.contains(e.target) guard.
- **Round-3 P2**: round-2 guard turned the hidden-strip area into invisible dead zone (couldn't tap-to-restore-chrome from bottom area when chrome hidden). Fix: `pointer-events-none` on hidden strip.
- **Round-3 P3 (DEFERRED)**: round-3 fix means hidden-strip-DRAGS bypass round-2 guard via fall-through. Filed as TB-043 — subjective interaction nuance; user can't see strip when hidden so both behaviors equivalent.

**Lesson:** mobile-touch UI components need explicit gesture-matrix testing upfront (visible-strip × hidden-strip × tap × drag × swipe = 4-8 cells). Designer spec covered visual + i18n exhaustively but didn't enumerate gesture-state-matrix combinatorics. **Forward action**: when authoring touch-component spec, include a gesture-matrix table in Dev Notes (rows = surface state, columns = gesture types, cells = expected behavior). Even a 4×4 sketch would have caught most of the round-2/3 nuances at spec-time. Codify if more touch components surface in Init 17+.

### §3.2 Codex MCP Playwright hang (Story 22.3 round-3 retry)
First Codex review on commit `cbbed23` hung ~1h30m on `mcp: playwright/browser_run_code_unsafe started` — 14 stale playwright procs orphaned. Codex was trying to spawn a real browser to verify the touch interaction. Operator noticed delay + I killed the hung review + restarted. Retry returned clean P3 in normal ~3 min wall.

**Lesson:** Codex review with Playwright MCP available can hang in WSL2 / constrained env. Mitigation: limit Codex MCP access on suspicion-prone reviews, OR add a watchdog (~10 min timeout) on background Codex tasks to auto-kill + retry. **Forward action**: monitor Codex reviews running >15min and surface to operator OR auto-restart after 20min. Not yet a memory-worthy pattern (1 occurrence) — re-evaluate if recurs.

### §3.3 Per-story pause within epic vs per-epic budget gate — operator-suggested retro topic
Operator 2026-05-24 (verbatim): *"warto w tak razie rozpocząć pracę nad epiciem a pauzować co story? Ale to zostawiam wam do przegadania tego na spokojnie potem"*.

Current rule ([[feedback_autonomous_sleep_on_budget]]): pause at the start of an epic when remaining budget won't fit the whole epic without crossing thresholds. The rule worked correctly mid-Init-16 (paused before Epic 22 start with 49% budget used + 4 stories of Epic 22 estimated 32-42%).

Operator's per-story alternative: start the epic immediately, pause after each story (or every 2 stories). Pros: faster end-to-end if budget refresh happens MID-epic (no upfront pause). Cons: harder to predict mid-epic budget overshoot; partial-epic state on disk = harder for operator to mentally model.

**Recommendation (deferred to Init 17+ retro for cross-cross-confirmation):** keep current per-epic-pause rule as DEFAULT, but allow operator-override via "lecisz autonomicznie + start epic now even if budget tight" instruction. The /loop / cron-based autonomous modes can adopt the per-story-pause if operator explicitly invokes that path. Don't change the default — it worked here. Track for one more init cycle before committing.

**Memory candidate (not yet written):** `feedback_per_story_pause_optional.md` — if operator opts in to per-story-pause AND budget is borderline (not catastrophic), prefer per-story pause to allow opportunistic mid-epic continuation.
