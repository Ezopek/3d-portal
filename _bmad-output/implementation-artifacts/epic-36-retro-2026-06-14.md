## What went well

### 1. Scope boundaries held

The initiative stayed read-only where it was supposed to:

- no on-demand slicing / G-ENQUEUE,
- no queue mutation,
- no slicer-worker path,
- no Alembic migration,
- no exposure of raw Orca/bundle internals to member DTOs or UI.

This was important because E36 sits on top of sensitive profile publishing internals. The member surface consumes published sidecars and estimate records; it does not become a second admin/profile API by accident.

### 2. Backend leak fences were explicit early

Story 36.1 forced a purpose-built member DTO and negative tests against internal fields (`bundle_hash`, raw Orca refs, chain bodies, sidecar paths). That made 36.3 safer: frontend could treat the member endpoint as product-shaped data, not scrub admin-shaped data in the browser.

### 3. `offer_id` estimate path avoided member 422s

36.2 realized the intended contract: member-selected offers resolve to estimate / `not_computed` / 404 for invalid offer, not resolver 422s. The backend branch also avoided live resolver and queue code, preserving the read-only deploy story.

### 4. UX gate was worth it — and operator correction still improved it

The initial UX gate correctly located the picker in the FilesTab estimate context and kept selection ephemeral. However, the first 36.3 implementation rendered too much visual weight. The operator correction produced 36.4, which made the picker fit the existing material/quality row as a compact inline select.

The lesson is not "skip UX". The lesson is: UX gate should define placement/state semantics, but the first visual density pass must still be checked against the live page rhythm.

## What was painful / surprising

### 1. Story status drift inside story files

`sprint-status.yaml` correctly marked 36.1 and 36.2 done, but the story file headers still said `Status: review`. This creates avoidable ambiguity for future agents: sprint-status is the SoT, but stale headers make humans and agents double-check unnecessarily.

**Action:** after a story is ff-merged and status synced, update the story header to `Status: done` in the same closeout/docs pass.

### 2. First member picker was visually too loud

36.3 used a larger picker/card/radio-style surface. Functionally correct, but visually overweight for the member FilesTab. 36.4 fixed this by integrating into the existing estimate selector row.

**Action:** for future small member controls, default to inline controls inside existing bars before introducing standalone panels/cards.

### 3. Backend/API story split worked, but frontend had to absorb vocabulary

The backend introduced `not_computed`, `profile_selection_context`, published offer DTOs, and offer-mode reads. Frontend needed careful naming to avoid making `not_computed` look like an error or zero-valued estimate.

**Action:** any new estimate status must include a frontend rendering contract in the story ACs, not only API schema tests.

### 4. Backlog capture happened mid-epic

`37-1-stl-file-delete-ui` was captured during/after E36 because operator hands-on workflow exposed a missing UI affordance. This was handled correctly as backlog capture, not scope creep into E36.

**Action:** keep this pattern: capture adjacent operator pain immediately, but do not silently expand the current epic unless explicitly promoted.

## Decisions carried forward

1. **Member-facing profile offer selection stays read-only.** No request-estimate CTA until a separate G-ENQUEUE story is approved.
2. **Compact inline controls are the default for small member estimate options.** Standalone panels require a stronger UX reason.
3. **Do not show printer names in member offer option text.** 36.4 intentionally reduced member-facing option content to `portal_label` only.
4. **Story file status headers must be reconciled with sprint-status at closeout.** Sprint-status remains authoritative, but stale headers are process debt.
