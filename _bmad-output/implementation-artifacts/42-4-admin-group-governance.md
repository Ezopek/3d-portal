## Story

As a **portal admin governing the facet taxonomy (and the E46 admin tag/group screen that will consume this API)**,
I want **admin-only write endpoints to create / rename+reorder / delete `TagGroup` facet groups (`POST/PATCH/DELETE /api/admin/tag-groups`), to move a tag into or out of a group via `PATCH` of the tag's `group_id`, and to keep tag creation admin-only — each write emitting a correct audit row and each group deletion leaving its tags groupless (never deleted)**,
so that **Initiative 25's curated facet taxonomy has the governance write-surface the admin screen (E46) consumes, without any category coupling, without rebuilding the existing `POST /tags/merge`, and without touching the destructive category-retirement work relocated to E47**.

Traces: **FR25-TAX-2** (tag creation admin-only), **FR25-ADMIN-1** (tag/group governance — the API half; the screen is E46); architecture.md § Initiative 25 **Decision AU** (`SET NULL` on `tag.group_id`, groupless is first-class) + **Decision AW** ("New `POST/PATCH/DELETE /api/admin/tag-groups` … move-to-group … Admin group-governance routes stay `current_admin`-gated"). Epic **E42** (additive backend contract; **closes after this story**), SCP `sprint-change-proposal-2026-07-19-e42-deferred-coupled-cutover.md` §3.2/§4.

## Acceptance Criteria

1. **New admin-only taxonomy-governance router (`POST /api/admin/tag-groups` — create).** A **new** router module (`apps/api/app/modules/sot/tag_group_admin_router.py`, prefix `/api/admin`, registered in `apps/api/app/router.py`) exposes the three tag-group routes and owns the relocated admin-only `POST /api/admin/tags` route (AC #5). `POST /api/admin/tag-groups` uses the contract below. Request schema `TagGroupCreate = {slug: str (min_length 1), name_en: str (min_length 1), name_pl: str | None = None, position: int = 0}`. Returns **201 + `TagGroupSummary` = `{id, slug, name_en, name_pl, position}`** (a new flat read schema in `sot/schemas.py`, mirroring `CategorySummary` — NOT the 42.2 `TagGroupRead`, which embeds `tags[]` and is for the read endpoint only). **409 on slug conflict** (`TagGroup.slug` is globally unique via the shipped `uq_tag_group_slug` index — Decision D-AUTH-GOV / D-UNIQ-1). 422 on other Pydantic validation. **RESOLVED (D-ROUTER-1):** the governance routes go on a **dedicated router, NOT on `sot/admin_router.py`**, because `test_openapi_agent_surface.py::test_every_sot_admin_route_has_agent_write_tag` asserts **every** route on `sot.admin_router.router` carries `tags=['agent-write']` (the AI-ingestion write surface). Governance is admin curation, not agent ingestion (FR25-ADMIN-1), and is `current_admin`-gated — tagging it `agent-write` would be a false auth signal to the agent runbook. A separate router keeps the agent-write invariant intact and gives governance a clean admin-only home. Every new route carries a non-empty `summary` + `description` (OpenAPI honesty, story convention). **Correction (Independent Validation M1):** this is **not** enforced by `test_openapi_agent_surface.py` for the governance routes — `_iter_target_operations` scopes by **tag** (`TARGET_ROUTER_TAGS = {admin, sot-admin, sot-read}`), **not** by path, and the governance router's `sot-admin-governance` tag is outside that set, so the new routes fall outside the summary/description gate. Supply them anyway (honesty); do not rely on the test to catch a missing one.

2. **`PATCH /api/admin/tag-groups/{group_id}` — rename + reorder.** Request schema `TagGroupPatch = {slug?, name_en?, name_pl?, position?}` with `model_config = ConfigDict(extra="forbid")` (unknown keys → 422, mirroring `CategoryPatch`/`TagPatch`). **Partial update via `model_dump(exclude_unset=True)`**: only supplied fields change. **RESOLVED (D-NULLSEM-1 — Independent Validation C1):** `name_pl` is the **only** nullable column, so an **explicit `name_pl: null` clears** it (present in `exclude_unset` data), an omitted `name_pl` is untouched. `slug`/`name_en`/`position` are **NOT NULL** columns — an **explicit null on any of them is a `422`** (rejected at the Pydantic layer, see Task 2), NOT applied. This is the critical divergence from a naive "mirror `update_category`": `update_category` (`admin_service.py:1238-1257`) does a blind `setattr` over `exclude_unset` data and only catches `IntegrityError → slug_conflict`, so an explicit `slug/name_en/position: null` would set a NOT NULL column to null → IntegrityError → **wrong 409 `slug_conflict`** (nonsensical for `position`). Rejecting null in the schema means null never reaches `setattr`, the service body stays identical to `update_category`, and the only remaining IntegrityError path is a genuine slug race → 409. **No-op** (empty `{}` body) → **200 + unchanged `TagGroupSummary`** (no error) **and still writes one `tag_group.update` audit row with `before == after`** (unconditional audit, mirroring `update_category`; consistent with AC #6 "every write emits a row"). `position` is updatable (reorder). **404** if the group id does not exist. **409** if a new `slug` collides with another group. Returns 200 + `TagGroupSummary`. Audit `tag_group.update` records the **full fixed field snapshot** `{slug, name_en, name_pl, position}` for both `before` and `after` — **not** a changed-only diff (verified: `update_category`/`update_tag` snapshot the full field set, bounded and non-secret, D-AUDIT-2).

3. **`DELETE /api/admin/tag-groups/{group_id}` — delete, tags survive as groupless.** Returns **204** on success, **404** if the group id does not exist (**not** idempotent-204-on-missing — matches the `delete_category`/`delete_tag` repo convention: missing → 404). **RESOLVED (D-SETNULL-1):** deleting a group **must not delete or block on its tags** — the shipped FK `fk_tag_group_id` is `ON DELETE SET NULL` (Decision AU; `_entities.py:87-90`; migration `0018`), and `PRAGMA foreign_keys = ON` is set on every SQLite connection (`session.py:19`) so SET NULL fires on SQLite **and** Postgres. After delete, every member tag has `group_id = NULL` → it is a **groupless** tag and surfaces in `GET /api/tag-groups` `groupless[]` (read-after-write consistency, AC #8). The service **reads the member tag ids before the delete** (for the audit snapshot) and relies on the DB-level SET NULL for the nulling — no manual per-tag `UPDATE`, no ORM cascade relationship (none is defined; the FK column alone drives it). There is **no 409 "group in use"** — unlike category delete, a non-empty group deletes cleanly.

4. **Move a tag into / out of a group via the existing tag `PATCH` (`PATCH /api/admin/tags/{tag_id}`).** **RESOLVED (D-MOVE-1):** move-to-group **extends the existing** `PATCH /api/admin/tags/{tag_id}` endpoint (reuse per the SCP + Decision AW "move-to-group (`PATCH` a tag's `group_id`)") — **no dedicated route is built**. Add `group_id: uuid.UUID | None = None` and `group_position: int | None = None` to `TagPatch` (`admin_schemas.py`). Semantics (via `exclude_unset`): `group_id = <uuid>` → validate the group exists (**400 "tag group not found"** if absent) and assign; `group_id = null` → tag becomes **groupless** (`group_id` is the one intentionally-nullable FK); `group_id` omitted → unchanged; `group_position = <int>` → set intra-group order (omitted → unchanged). **`group_position` is a NOT NULL column** — an **explicit `group_position: null` is a `422`** (rejected at the Pydantic layer, Task 2), NOT applied. This matters especially because `update_tag` (`admin_service.py:1000-1016`) has **no** IntegrityError guard, so an unrejected `group_position: null` would raise a **raw IntegrityError → 500** (D-NULLSEM-1 / Independent Validation C1). This endpoint stays on `sot/admin_router.py` and therefore keeps its existing **admin-or-agent** auth + `agent-write` tag (the task scopes admin-only to *creation*, not *move*; a move only reorganizes already-curated tags). The existing `tag.update` audit row (`update_tag`) is extended so `group_id` / `group_position` are always present in its full before/after snapshot.

5. **Tag creation is tightened to admin-only and relocated to the governance router (`POST /api/admin/tags`).** **RESOLVED (D-ADMINONLY-1 — deliberate security tightening, FR25-TAX-2):** move the existing `admin_create_tag` route registration (same method/path, function name, request/response/status semantics, and reused `create_tag` service) from `sot/admin_router.py` to the new governance router and gate it with canonical **`current_admin`** (`app/core/auth/dependencies.py:76`; admin → pass, else **403 `admin_required`**). The route is tagged `sot-admin-governance`, **not** `agent-write`; an admin-only operation must not advertise itself as an agent capability merely to satisfy a module-scoped test. FastAPI's generated operation ID remains stable by preserving function name + method/path; add an explicit operation-id assertion if the current OpenAPI proves otherwise. **What changes:** a **member** was already 403'd; the real delta is the **`agent` service role loses tag-create** (now 403). This is the intended FR25-TAX-2 posture (curated vocabulary — the AI ingester selects existing tags, it does not mint new ones). **Unchanged:** `PATCH /tags/{id}` (rename + AC #4 move), `DELETE /tags/{id}`, and `POST /tags/merge` remain on `sot/admin_router.py` with existing admin-or-agent auth + `agent-write`. Code-first audit found no live consumer mints tags as agent; every existing create test authenticates as admin.

6. **Every write emits a correct audit row (repo conventions).** **RESOLVED (D-AUDIT-1):** add `"tag_group"` to `KNOWN_ENTITY_TYPES` in `apps/api/app/core/audit.py` (currently absent — the frozenset has `"tag"` but not `"tag_group"`; `record_event` raises `ValueError` on an unknown `entity_type`, and the direct-`AuditLog` `_audit_entity` helper writes `entity_type` verbatim, so an unregistered type would drift the Slice-2 enum-tightening guard). Action names follow the dotted `<entity>.<verb>` convention already used for categories/tags: **`tag_group.create` / `tag_group.update` / `tag_group.delete`** (entity_type `"tag_group"`, `entity_id = TagGroup.id`), and the tag **move** reuses **`tag.update`** (entity_type `"tag"`, `entity_id = Tag.id`). Rows are written **in the same transaction** as the mutation via `_audit_entity(session, ...)` (the `admin_service.py` idiom), not the out-of-band `record_event`. **Audit detail allowlist (no secret/PII leakage — D-AUDIT-2):** before/after JSON contains **only** `{slug, name_en, name_pl, position}` for groups, `{group_id, group_position, slug, name_en, name_pl}` for the tag move, and for delete `{slug, name_en, detached_tag_ids: [uuid…], detached_tag_count: int}` — UUIDs / slugs / display names / ints only. No user credentials, tokens, emails, or free-form user text ever enter an audit row here (none of these fields carry such data; `name_en`/`name_pl`/`slug` are non-secret taxonomy labels, consistent with existing `category.*` / `tag.*` audit snapshots). NFR25-LEAKFENCE-1 governs the anonymous `ShareModelView` DTO, **not** audit rows — no share DTO is touched here.

7. **Auth boundary + route-enforcement coverage.** The three new group routes plus relocated tag-create route are **`current_admin`-gated and outside `_PUBLIC_ROUTES`** (no `main.py:_PUBLIC_ROUTES` edit — TB-055: admin routes were never anonymous). Auth-boundary tests assert, for `POST`/`PATCH`/`DELETE /api/admin/tag-groups` **and** the tightened `POST /api/admin/tags`: anonymous → **401**, member → **403**, **agent → 403**, admin → **2xx**. The `test_route_enforcement_gate.py` enumeration keys off the live route table and self-heals for the new authenticated routes (no allowlist entry needed). `test_openapi_agent_surface.py` invariants hold: the new governance routes are **not** on `sot.admin_router.router` (so no `agent-write` requirement / no "extra" flag), `POST /tags` is absent from the `agent-write` set and present under `sot-admin-governance`, and every new route has `summary` + `description`.

8. **Read-after-write consistency with the 42.2 read API (`GET /api/tag-groups`).** After each governance write, `GET /api/tag-groups` (shipped in 42.2) reflects it under its documented ordering (groups `(position, slug)`; intra-group tags `(group_position, slug)`; `group_id IS NULL` → `groupless[]`): (a) a created group appears as an **empty group** (`tags: []`) at its sort position; (b) a `PATCH position` re-sorts groups; (c) a tag moved to group G appears under G at its `(group_position, slug)` slot and leaves `groupless`/its old group; (d) `group_id: null` returns the tag to `groupless`; (e) after `DELETE`, the group is gone from `groups` and its former members appear in `groupless`. At least one read-after-write test per case exercises the real HTTP round-trip (write endpoint → `GET /api/tag-groups`).

9. **Concurrency / uniqueness / transaction integrity.** Slug uniqueness is enforced at the **DB layer** (`uq_tag_group_slug` unique index) — the service catches `IntegrityError` on `flush()` and raises `ValueError("slug_conflict")` → 409 (mirrors `create_category`/`update_category`), so a create/rename race cannot double-insert a slug (the pre-check is a nicety, the constraint is the guarantee). Each service function **owns its own transaction** (`session.commit()` with the audit row inside the same tx — the `admin_service.py` atomicity idiom). `position` and `group_position` are **plain non-unique ints** — duplicate positions are **allowed** and deterministically tie-broken by `slug` at read time (42.2 sorts `(position, slug)` / `(group_position, slug)`); no positional uniqueness constraint is added.

10. **Ordering + tie behavior (no renumber cascade).** Setting a group `position` or a tag `group_position` writes exactly that one value — the story does **not** renumber siblings (no gap-filling / reindex cascade). Ties on equal `position` are resolved by `slug` in the 42.2 read (verified in AC #8 tests). This keeps writes single-row and idempotent; contiguous reordering (if E46 wants it) is a client concern that issues N single-field `PATCH`es.

11. **E41-retro `tag.group_id` index action item — discharged code-first (no migration).** **RESOLVED (D-INDEX-1):** evaluated the E41-retro action item ("Evaluate an index on `tag.group_id` when group-scoped queries appear"; owner re-pointed to 42.4). **Discharge = no index, no migration.** Rationale, code-first: (a) the 42.2 read (`list_tag_groups`) loads **all** tags in one `select(Tag)` and buckets by `group_id` **in Python** — there is **no `WHERE group_id = ?` query** anywhere; (b) this story's move is a single-row `UPDATE tag … WHERE id = <pk>` (PK-indexed); (c) group **delete**'s only `group_id`-keyed access is the FK `ON DELETE SET NULL` child scan, over the `tag` table (dozens–low-hundreds of rows, admin-rare) — a seq scan here is negligible and does not justify an index; (d) no admin governance query filters tags by group. The triggering condition ("tags in group X ordered by group_position" as a hot query) **never materialized**. Adding an index is also doubly out of scope: **42.4 touches no migration**, and the next Alembic slot **`0019` is reserved** for the destructive category cutover (47.5). Sources do not approve a speculative index (Ponytail minimal-diff; PRD/architecture are silent — no index requirement). **Action:** flip the `action_items` entry (epic 41, `tag.group_id` index) to `done` with a truthful discharge note; if a future hot per-group query appears (e.g. E46 server-side group filtering), re-open as a fresh triage item.

12. **Gates + no-leak scope fences.** `cd apps/api && uv run ruff check . && uv run ruff format --check .` clean; focused suites (`test_sot_admin_tag_groups.py` new, plus the extended tag/auth-boundary/audit tests) green; then the **full backend suite 3× consecutive identical** pass counts (NFR25-DETERMINISM-1); after code reviews, the repo **full `check-all`** (16/16, incl. web visual) green — the 42.1/42.2 gate shape. **Hard scope fences (no leak):** **no** `Category`/`Model.category_id` ORM/route/schema/DTO change, **no** `0019`/any migration, **no** `ModelCreate/Patch`/`ShareModelView` change, **no** frontend (`apps/web/*` — the admin screen is E46), **no** rebuild of `POST /tags/merge` or of tag rename/delete. E42 epic-closeout + retrospective are the **controller's** to run after this story is `done` (this story does not flip `epic-42`).

### Findings & resolutions

- **Critical C1 — PATCH explicit-null on NOT NULL fields → wrong 409/500 (adversarial #1).** `update_category` (`:1238-1257`) blind-`setattr`s `exclude_unset` data and only maps `IntegrityError → slug_conflict`; `update_tag` (`:1000-1016`) has no IntegrityError guard at all. So a naive "mirror `update_category`" would turn an explicit `slug/name_en/position: null` into a nonsensical **409**, and `group_position: null` into a raw **500**. The story specified null semantics only for the genuinely-nullable `name_pl`/`group_id`. **Resolved:** D-NULLSEM-1 — `@field_validator` rejects explicit null on the NOT NULL patch fields → **422**; service bodies stay identical to the live idiom; 422 tests added (AC #2, #4; Task 2, #4, #7).
- **Important I1 — "changed-only before/after" audit claim was factually wrong (adversarial #2/#6).** `update_category`/`update_tag` snapshot the **full** fixed field set, not a diff. **Resolved:** corrected AC #2, Task 4, Dev Notes, and D-NULLSEM-1 to specify a full bounded snapshot (`{slug,name_en,name_pl,position}` for groups; `{slug,name_en,name_pl,group_id,group_position}` for the tag move).
- **Important I2 — empty `{}` PATCH audit behavior was ambiguous (adversarial #2).** **Resolved:** made explicit — empty body → 200 **and** one `tag_group.update` row with `before == after` (unconditional audit, mirroring `update_category`); test added.
- **Important I3 — `agent-write` on the admin-only `POST /tags` is a latent contract-honesty inconsistency (adversarial #3).** Not a blocker: the security decision is settled by FR25-TAX-2 + controller directive, and `agent-write` is read by no consumer (only `test_openapi_agent_surface.py`) so nothing is actively misled. **Resolved:** documented that the tag is retained solely for the structural router invariant and is not an auth signal; authoritative contract is the `agent → 403` test; E46 follow-up recommended to disambiguate/relocate.
- **Minor M1 — summary/description enforcement rationale was inaccurate (adversarial #4).** `_iter_target_operations` scopes by **tag**, not path; the `sot-admin-governance` tag is outside the gate, so summary/description on the new routes is convention, not test-enforced. **Resolved:** corrected AC #1.
- **Minor M2 — `detached_tag_ids` payload bound (adversarial #6).** Acceptable at this cardinality (a group's membership is a small subset of a dozens–low-hundreds `tag` table; `detached_tag_count` already carried). Noted; no cap needed.
