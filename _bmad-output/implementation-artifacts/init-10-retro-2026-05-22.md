## Forward-applicable recommendations (operator-actionable)

### Recommended memory additions (Claude-owned, operator-blessable)

1. **`feedback_scp_pre_enumeration_phase.md`** — Before drafting SCP for an initiative, perform an enumeration pass: (a) grep codebase for already-shipped endpoints/components matching planned scope, (b) read sprint-status.yaml for closed initiatives covering the same surface, (c) read SoT for active feature inventory. 10-15 min upfront cost; saves story-mid scope-cuts. Threshold for skipping: <2 stories, all in obviously-greenfield surface.

2. **`feedback_security_vector_enumeration.md`** — For any story with NFR-SECURITY-* tag, the SCP §C MUST contain a "Threat vectors enumerated" sub-section listing: cookie-sending mechanisms (fetch, img, a-href, link rel=preload, script src, websocket upgrade), auth-state-consultation points (any /api/auth/* call), browser-default-credentials behaviors per element type. Anti-pattern: discovering vectors via Codex review iterations.

3. **`feedback_voice_heavy_dedicated_grilling.md`** — When a brief item involves decision-shape question (who does the work? what product persona changes? what user-facing surface emerges?), it gets DEDICATED AskUserQuestion grilling with 3-4 calibrating sub-questions, NOT one radio in a multi-item options grid. Test: would the operator answer this with a different question shape than the others in the grid?

### Recommended runbook / architecture additions (operator-decision)

4. **AGENTS.md "Scope boundaries"** (Winston): *Infrastructure concerns (logging pipeline, OTEL collector config, deployment topology, reverse-proxy rules) belong to `~/repos/configs`. 3d-portal initiatives scope only app-layer concerns. Recon subagent assesses split; >50% infra-side → carve out to `configs/`.*

5. **architecture.md "Frontend auth patterns" micro-rule** (Winston): *Component-level authorization checks (role, permission, ownership) MUST NOT redirect when authentication state is unknown or anonymous; defer to shell-level AuthGate for unauthenticated cases, only act on authenticated-but-unauthorized.*

6. **AGENTS.md § Test Discipline addition** (Murat): *After fix-up of any test in hang/timeout/deadlock class — mandatory full pytest re-run before claim "fixed". Masked tests have non-zero prior.*

### Recommended template additions (Claude-owned, low-friction)

7. **bmad-create-story checklist** (Amelia + Winston): "Verify against SoT + sprint-status for already-shipped endpoints" as a gate before SCP `## H2` writing.

8. **Architecture-decision template** (Winston): For each schema migration with dual-field (e.g. `body_pl + body_en`), enumerate read-path × write-path × {legacy, new} = minimum 4 combinations in the Decision body.

9. **SCP estimation template** (John): "Existing-coverage audit" subsection — `git log --oneline | grep -i <feature>` results per story before assigning effort estimate.

### Recommended test infrastructure (Murat)

10. **Credentialless surface test class** (`apps/api/tests/fixtures/anonymous_client.py` + playwright network-probe assertions) for share-token / anonymous endpoints. ~50-100 LOC. Asymmetric cost-benefit: single cookie-leak regression = public session exposure.

11. **NFR10-DETERMINISM-1 triage label gate** — before running `--update-snapshots`, classify each failure: `stale-baseline` (regen OK) / `deterministic-fail` (real bug, do NOT regen) / `flake-candidate` (need 3× run probe). Prevents masking real UX regressions behind blanket regen.

## Zażalenia (grievances surfaceable to operator)

Three operator-decision items:

1. **SCP-drafting time vs scope-rework cost**: today autonomous-mode ITCM drafts SCPs fast (~5-10 min for Init 10). Recommendations above would add ~10-15 min per story for enumeration phases. For 13-story initiative, that's +2-3h upfront vs current. **Operator decision needed**: is the velocity tradeoff (slower drafting, fewer mid-init scope cuts) acceptable? Today the rework cost is hidden in Codex round-2 fix-ups and mid-init clarifications — it's there, just not on the drafting clock.

2. **Voice-heavy elicitation pattern overhaul**: today AskUserQuestion grilling uses options-grid for all items. John's framing — "what work is operator trying to delegate?" vs "what feature should I build?" — would require separate dedicated grilling for decision-shape items. **Operator decision needed**: tolerate longer grilling step (3-4 sub-questions per voice-heavy item) for better intent-capture, or accept current "options grid + mid-init correction" model?

3. **Security work convergence model**: Codex-rigor-based + operator-blessing has worked (Story 16.3 closed correctly in 2 rounds). Murat proposes maszynowo-enforced credentialless test fixture. Cost: 50-100 LOC + maintenance. **Operator decision needed**: is the additional CI cost justified by removing the "lagging indicator" risk, or is Codex sufficient for this project's scale?
