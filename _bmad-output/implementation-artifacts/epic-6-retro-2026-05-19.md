## What went well

1. **Init-0 share-service / share-admin precedent paid massive dividends across the epic.** Every backend module mirrored an existing Init-0 file: `ShareService` shape → `InviteService`, `share/admin_router.py` → `invite/admin_router.py`, `share/admin_router.py::_service()` factory copied verbatim into `invite/router.py`. Caller-owned audit emission (service is pure state mutation) was the dominant pattern, replicated cleanly across 6.3 / 6.4. New ground was added only where genuinely new (Redis sliding-window primitive in 6.6, soft-alert threshold mechanic in 6.7). Estimated time saved across the epic: substantial — most stories were "look up the precedent file, mirror its structure" rather than designing from scratch.

2. **Verbatim AC-6 / AC-7 test-name checklist gave dev agents a hard stop-condition per task.** Every story spec from 6.2 onward listed 20-30 named tests with exact assertion shape. TDD red phase started by writing failing test files matching the checklist names. Dev agent had no ambiguity about "done": when all named tests passed AND `infra/scripts/check-all.sh` was 10/10 green, the story was reviewable. 6.2 (19 named tests), 6.3 (32), 6.4 (23 — some collapsed when assertion-set overlapped), 6.5 (26), 6.6 (31), 6.7 (38) all hit or exceeded their checklist count.

3. **CAS predicate pattern bootstrapped in 6.2 generalized cleanly.** Codex review on Story 6.2 introduced `revoke()` as a conditional UPDATE with `WHERE id=:id AND used_at IS NULL AND revoked_at IS NULL` + `rowcount` inspection, replacing the original read-then-write that lost the consume↔revoke race. Same pattern reused implicitly in 6.4's `IntegrityError` race compensation for duplicate-email under concurrent register requests. The pattern (read state, attempt conditional write, handle compensation if rowcount == 0) is now a project convention.

4. **`feat → codex fix-up` commit pair convention made every story's defect surface traceable.** Every story shipped as exactly one `feat` commit + zero-or-more `fix(api)` commits with subject containing "codex fix-up". Retro can reconstruct the dev→Codex→fix loop just by reading `git log --oneline`. No detective work needed. Three fix-ups on 6.6 → immediately legible as "this story had a complex iteration cycle."

5. **`infra/scripts/check-all.sh` as one-shot 10-stage quality gate.** Established pre-epic, used as the canonical "is this ready for review?" signal across all 7 stories. No per-stage ad-hoc verification; one command, ten stages, all green or none. Range-based deploy-skip gate (also pre-epic) meant the rapid feat→fix-up cadence on `main` didn't thrash `.190` deploys — doc-only commits skipped automatically.

6. **Pre-flight grep discipline emerged organically in Story 6.6.** Dev agent documented (lines 1262-1280 of story spec) a pre-implementation grep scan to find existing tests that hit `/api/auth/login` more than 3 times within a single test (the rate-limit threshold) — those would silently start failing once the middleware mounted. Pattern not in the spec; dev agent introduced it. Worth promoting to a convention for any future cross-cutting middleware story.

7. **Same-day execution cadence kept context warm.** All 7 stories shipped on 2026-05-19 between roughly 03:-- and 13:53. Cross-story context (fixture patterns, conftest semantics, audit-action vocabulary) stayed in the agent session memory. No re-discovery of "how do I mint a member JWT for a test" between Story 6.5 and 6.7 — same fixture template was copy-pasted forward.

8. **The 3-round security review on 6.6 caught a real production-grade DoS surface.** Without Codex's three iterative passes, the rate-limit middleware would have shipped with:
   - Bucket collapse: all users sharing one bucket because nginx wasn't forwarding X-Real-IP correctly across the 2-hop proxy chain.
   - LAN spoofing: any caller on the trusted network could forge X-Real-IP via direct port-8090 access and pick a fresh bucket per request.
   - ZADD member collision: sub-millisecond bursts collapsed to single ZSET entries, under-counting actual rate.

   The eventual fix moved the trust boundary upstream into nginx via the `real_ip` module + CIDR allowlist — a more principled position than trying to sanitize inside FastAPI. This is exactly the class of defect that backend pytest cannot catch; Codex's "cross-system threat-modeling" review layer is now demonstrably load-bearing.

9. **Story 6.5 shipped clean on first review — minimal-diff strategy validated.** Two-line dependency swap + one new dependency export = no fix-up commit. Reinforces the convention that small, surgical PRs reduce review surface.

## What surprised us

1. **Codex review intercepts 86% of stories — the dev→Codex loop is the *primary* defect-catching layer, not pytest.** 6 of 7 stories had at least one P1/P2 finding caught at Codex review. Backend pytest catches logic; Codex catches security + cross-system wiring (compose env not propagated × 2 stories, `uv.lock` stale, OpenAPI contract too wide, IntegrityError race not handled, X-Real-IP trust boundary, ZADD member shape, logging filter pass-through fields, lazy-formatting collision). The partition is reliable enough that future story specs should explicitly call out "what Codex will catch that pytest won't" as a pre-flight reminder.

2. **fakeredis event-loop binding was rediscovered THREE separate times.** Stories 6.3, 6.4, and 6.6 each independently solved "fakeredis is bound to a different event loop" with slightly different workarounds (`c.portal.call(...)`, async-closures + `c.portal.call(_do)`, `TestClient(app).__enter__()` inside the fixture factory). The solution exists in the codebase by Story 6.6 but isn't documented anywhere that a future dev agent would discover it pre-implementation.

3. **Story 6.6 spec literally contained the wrong claim that broke AC-5.** Spec said "install rate-limit middleware AFTER `install_csrf_middleware`" — which, given Starlette's LIFO `add_middleware` semantics, would make rate-limit OUTERMOST, directly violating AC-5's `test_login_csrf_rejection_does_not_burn_rate_limit` invariant. Dev Notes also incorrectly claimed `@app.middleware("http")` decorator wraps as outermost. Dev agent had to invert spec at implementation time. This is doc-induced rework — the spec actively misled.

4. **Planning-doc drift is structural, not authorial.** Architecture.md / epics.md / prd.md were written before Init 0 code reality settled (UUID PKs vs INTEGER, `user` table singular vs `users`, `UserRole` enum vs `Role`, `KNOWN_ENTITY_TYPES` semantics, `/api/admin/share` vs `/api/share/` path, frontend file-based routing convention). No single author or session is at fault; drift accumulated because docs froze planning-time assumptions and code drifted toward correct conventions during Init 0 implementation. 17 distinct drift items surfaced across 6 stories — see catalog below.

5. **The `pyproject.toml` → `uv.lock` regeneration step is invisible.** Story 6.4 added `zxcvbn` as a new dependency; the lockfile was not regenerated. Codex caught it. Backend pytest, ruff, alembic — none of them care about lockfile staleness. Only the eventual `docker compose build` on `.190` would have surfaced it. **This is a class-of-defect candidate** for `check-all.sh` enhancement.

6. **Compose-env wiring was forgotten TWICE (6.6 and 6.7 — identical Codex finding).** Both stories added new `Settings` fields in `apps/api/app/core/config.py`. Both stories forgot to wire them into `infra/docker-compose.yml`'s `services.api.environment` block. Codex flagged both; both required identical fix-up commits. Story 6.7 should have benefited from Story 6.6's lesson — but the dev agent did not re-grep compose for "did I do this last time?" before shipping. **The lesson from one story did not propagate to the next within the same session.** Worth a process-level note.

7. **Story 6.7's Dev Agent Record stayed empty (template placeholders).** The story commits shipped (`12ba359` + `54af50a`), the code is in production, the tests pass. But the story file `_bmad-output/implementation-artifacts/6-7-per-member-share-token-cap.md` § "Dev Agent Record" still contains `{{agent_model_name_version}}` placeholders. This is itself a process drift — the story file is supposed to be the durable record of the dev experience. For 6-7, the retro had to reconstruct from commit messages. Worth a process action item.

8. **fakeredis cross-test state pollution under the global `FakeServer` registry.** Story 6.6 dev agent discovered that `fakeredis.aioredis.FakeRedis()` (no explicit `server=` kwarg) shares a global registry across the test session. Once a counter incremented, it stayed incremented for subsequent tests. Worked around by passing `server=fakeredis.FakeServer()` per instantiation. Not in the spec; dev agent debugged it live. Should be in a conftest fixture or AGENTS.md testing note.

9. **The nginx 2-hop trust-boundary topology was NOT in any spec.** Story 6.6 spec wrote rate-limit by IP; reality is browser → edge nginx (192.168.2.180) → port 8090 web nginx → API. X-Real-IP overwrites at each hop unless explicitly configured. Architecture.md Decision G said "key by client IP" but didn't model the topology. Codex's two follow-up rounds (#2 X-Real-IP overwrite at second hop; #3 LAN spoofing via direct port-8090) caught what architecture missed. **Threat-modeling against the actual infra topology should be a pre-implementation step for any IP-keyed rate-limit / auth surface.**

## What we'd do differently

1. **Run `bmad-correct-course` BEFORE the next Init 5 epic starts.** 17 doc-drift items are accumulated; the next story-creation session (Story 7.1) will re-discover at least 4 of them (alembic path, KNOWN_ENTITY_TYPES semantics, Role→UserRole, frontend routing convention). Single `docs(bmad)` commit on `main` patching the four major drift surfaces clears the deck. Doc-only, gitignored target (`_bmad-output/planning-artifacts/`), no deploy.

2. **Document the fakeredis testing pattern in a project-knowledge note or conftest fixture.** Three rediscoveries of "fakeredis is bound to a different event loop" + one rediscovery of "fakeredis global FakeServer pollution" = 4 hours of dev-agent time across the epic, all spent solving the same problem. A 10-line conftest helper or a 30-line note in `AGENTS.md` § Testing would have eliminated this entirely. Promote.

3. **Add a `check-all.sh` stage that diffs Settings ↔ env.example ↔ docker-compose.yml api environment.** Class-of-defect catcher for the 6.6 + 6.7 repeat finding. Stage scans `apps/api/app/core/config.py` for `Settings` field names, asserts each appears in `infra/env.example` (with at least a commented-out default), and asserts the same set is forwarded to `services.api.environment` in `infra/docker-compose.yml`. Single shell stage; catches "I added a Settings field and forgot the compose wiring" before deploy. Same stage could check `pyproject.toml` ↔ `uv.lock` consistency via `uv lock --check`.

4. **Spec authors must fact-check pseudocode against current framework source.** Story 6.6's spec pseudocode about Starlette `add_middleware` ordering was wrong; Story 6.6's Dev Notes also wrong about `@app.middleware("http")` decorator ordering. Both took dev-agent time to discover at implementation. A pre-spec lint pass that grep'd for framework-specific claims (`Starlette`, `Pydantic`, `Alembic`, `SQLModel`, `pyotp`) and required a citation to current upstream source would catch this class.

5. **Story 6.7's Dev Agent Record needs back-fill.** The story file should be the durable record. For 6-7, the retro had to reconstruct from commit messages and sprint-status notes. Either (a) back-fill the story file post-hoc from commit log, or (b) accept that the retro doc + sprint-status notes are the canonical record and remove the empty placeholder section from the story file. (a) is preferred for long-term traceability.

6. **For any new IP-keyed surface, threat-model the proxy topology pre-spec.** Story 6.6 had the dev agent and Codex reverse-engineer the 2-hop nginx topology across three iteration rounds. The topology is documented elsewhere in the repo (deploy scripts, nginx confs) but was not consulted during spec authoring. Adding "topology diagram + trust boundary marked" as a required artifact for any IP-trust story would prevent re-discovery.

7. **Promote per-file `client` fixture to conftest.** Stories 6.3, 6.4, 6.5, 6.7 each mint admin/member JWTs + seed users + swap fakeredis in their own per-file `client` fixture. Story 6.5 spec explicitly flagged the ≥3 files threshold for conftest promotion. By Epic 6 close, **four+ files** need it. Promote in Story 7.1's spec or as a separate `chore(api)` commit before Story 7.1 starts.

8. **Pre-flight `pyproject.toml` / `uv.lock` sanity check.** Codex caught the stale `uv.lock` on 6.4. Should be one shell line in `check-all.sh`: `uv lock --check` (fails if lockfile out of date with pyproject.toml).

## Procedural lessons (for future BMAD initiatives)

1. **The dev → Codex review loop is the primary defect-catching layer.** Pytest catches logic; Codex catches security + cross-system wiring. Future story specs should explicitly list "Codex coverage targets" (compose env, lockfile, OpenAPI contract width, trust-boundary headers, IntegrityError compensation paths) as pre-flight reminders.

2. **Doc-drift accumulates structurally, not by individual fault.** When code precedent stabilizes during Init N implementation, planning artifacts authored before Init N freeze a fiction. The correction mechanism (bmad-correct-course) exists; it must be invoked before the next initiative's stories start, not deferred indefinitely.

3. **Same-session lesson propagation is unreliable.** The 6.6 → 6.7 compose-env wiring repeat shows that one story's Codex fix-up doesn't automatically prevent the same omission in the next story of the same session. The lesson must be encoded in tooling (a `check-all.sh` stage) or in spec template (a pre-flight checklist) to durably transfer.

4. **Spec literal accuracy matters more than spec completeness.** A 100-line story spec with one wrong claim about framework behavior costs more dev-agent time than a 50-line spec that explicitly says "verify Starlette middleware ordering against current source before mounting". Pseudocode in specs should be marked "verify against source" rather than treated as authoritative.

5. **For IP-keyed / trust-boundary work, the topology diagram is non-optional.** Story 6.6 demonstrated that without an explicit topology + trust-boundary annotation, even a thorough spec misses cross-hop-overwrite and direct-port-spoofing surfaces. Mandate a topology + boundary section in any future spec touching authentication / rate-limit / IP-trust.

6. **Test-name verbatim checklists in AC-6/AC-7 are high-leverage.** Every Epic 6 story used them; every Epic 6 story landed with a tight TDD red phase. Continue.
