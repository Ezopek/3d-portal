## What surprised us

1. **Codex intercept rate climbed from 86% (E6) to 100% (E7) — 14 fix-ups across 6 stories.** Epic 6 had 7 fix-ups for 7 stories; Epic 7 had 14 fix-ups for 6 stories. The fix-up complexity also climbed: Story 7.3 alone needed three sequential fix-ups (`7188b13` event-loop offload + atomic GETDEL → `2ffb290` partial-token restore-on-fail → `16b34d5` commit-guard preventing post-commit restore from minting duplicate sessions). **The dev → Codex loop is now the primary security-hardening layer for concurrency-sensitive code; pytest does not catch any of these classes.**

2. **Concurrency-and-race-condition findings dominate Epic 7's fix-up cycle.** Recurring shapes: (a) `partial_token` atomic claim — Story 7.3 needed `redis.execute_command("GETDEL")` after code verify; (b) `recovery_codes` single-use under concurrent verify — Story 7.3 needed conditional UPDATE `WHERE used_at IS NULL`; (c) regenerate-vs-cancel race — Story 7.5 needed AbortController plumb + 2-batch-window-on-concurrent-regen flagged as architectural-scope-creep (deferred to bmad-correct-course backlog); (d) restore-on-fail commit-guard — Story 7.3's `16b34d5` prevented post-commit exception from triggering partial-token restore that would mint a second RefreshToken family. **The dual-write surfaces (Redis + DB + audit emission) are concurrency-correct only with explicit CAS/atomic primitives.**

3. **Production incident on Story 7.1 was avoidable but the response was correct.** The fail-fast-in-production Fernet validator was the textbook secure default — but the operator deploy-gate ("provision `TOTP_FERNET_KEY` before merge") was not enforced by tooling, only by the sprint-status note. Production crash → incident → relax. **The lesson: a fail-fast validator must be paired with a provisioning verification stage in deploy.sh (or pre-merge gate in `check-all.sh`).** Story 7.2 ACs already encode this for subsequent stories via the `_assert_fernet_key_configured()` endpoint-init gate, but the systemic class — "any new production secret MUST land with a deploy-pre-flight verification" — is not yet codified.

4. **The drill artifact required THREE iterations to pass — and each failure revealed real production behavior.**
   - **Iter-1 failure (Step 1 audit assertion):** clock skew between dev box (`~/repos/3d-portal`) and `.190` + audit-write lag. Drill assumed `at >= STEP_START_ISO` would catch a row written 2-3s later. Fixed via 10s-past start lower bound + 15× 1s retry loop in `assert_audit_row`.
   - **Iter-2 failure (Step 5 HTTP 429):** the 9 login-scope endpoints share a single 5-failures-per-60s budget (login + verify + regenerate + disable all rate-limited under the same key). Drill at native cadence exhausted the budget at Step 5. Fixed via `wait_login_window()` 13s inter-request pacing → drill duration 20s → 120s.
   - **Iter-3 success:** all 8 binding steps PASS; 9 expected audit rows verified; binding lifecycle assertions all match.
   **Each "failure" was actually correct production behavior surfacing in the drill** — clock skew is real, the rate-limit IS active across all 4 endpoints, the budget IS shared as designed. The drill became binding evidence that the production semantics are correct.

5. **Decision F architecture contradiction was caught at create-story time, not at code-review time.** Architecture.md §1554 Choice-block said "middleware.py"; §1557 Cascading-block said "inline in login()". Story 7.4 create-story session explicitly identified the contradiction and resolved it (Cascading wins, per perf-budget rationale). Compare to Epic 6 Story 6.6 where the spec literally contained the wrong claim about Starlette middleware ordering — that was a doc-induced rework. **Story 7.4's create-story stage was more architecturally rigorous than Story 6.6's** — the Epic 6 retro lesson about spec literal accuracy was applied.

6. **The 2-FA disable filter binding (`invalidated_at IS NULL` only, no `used_at IS NULL` predicate) was deliberate, not a bug.** The drill's Step 7 assertion `invalidated_count=9` is binding evidence: 8 regen-batch active + 1 consumed-but-not-invalidated enroll-batch code. Codex P2 fix-up `f325efa` had tightened the REGEN UPDATE to active-only (correctly — regen should only invalidate currently-active codes). The analogous tightening on DISABLE was NOT applied, intentionally, because disable's semantics are "invalidate the entire batch including any consumed-not-yet-invalidated code, so the consumed code transitions to a dual-stamped `used_at` + `invalidated_at` state". **This regen-vs-disable predicate asymmetry is shipped as-designed but is NOT documented in architecture.md Decision E §1527-1528.** Drill artifact's runbook gap R-5 (P2) flags this as fragile to future maintainer "cleanup" PRs.

7. **The pragmatic-design ruling on forced-enrollment cookies stood up to E7 closure.** Story 7.4 issues cookies BEFORE the forced-enrollment branch fires (the user IS password-authenticated; forced-enrollment redirects to `/settings/2fa` carrying the original `next` URL). Architecture.md Decision F's "pre-cookie-issue" wording was treated as loose ordering language, not a hard constraint. **No Codex finding flagged this as insufficient.** The strict no-cookies + new-endpoint-variants alternative was explicitly DEFERRED to a future story if needed; Michał's sole-admin trust model + the drill's Step 8 assertion (`totp_enroll_required=false` post-disable) verified end-to-end. Pragmatic suffices for homelab scale.

8. **Story 7.5's two P2 findings shipped as separate fix-up commits — and the second one (predicate restoration) is itself a deferred architectural fix.** `91d91b6` addressed concurrent-regenerate + cancel UX + asyncio.to_thread offload. `f325efa` restored the active-only predicate on UPDATE that the first fix-up had inadvertently broadened. **Two layered fix-ups on the same story is a new pattern** — Epic 6 had stories with multiple fix-ups (6.6 had 3) but they were all serialized findings from one review pass. Story 7.5's two fix-ups came from a re-review after `91d91b6` shipped — meaning Codex caught a regression introduced by its own previous fix. The dev → Codex loop iterates to fixed-point.

9. **Story 7.6's drill is the first artifact in the new NFR5-OBS-2 category — and it sets the precedent for future drills.** Format mirrors `agent-runbook-smoke-2026-05-11.md` (7 binding H2 sections). The drill artifact is gitignored (under `_bmad-output/`) so the binding evidence lives operator-locally; the script that produces it is committed (`infra/scripts/2fa-recovery-drill.sh`). **Future operational drills (E10 cutover smoke, backup-recovery drill, etc.) should follow this split** — committed reproducible script + gitignored artifact.

## What we'd do differently

1. **Codify "secret provisioning gate" in `infra/scripts/deploy.sh` or `check-all.sh`.** Story 7.1's production incident was avoidable if a pre-deploy verification stage had checked that every secret env-var named in `infra/env.example` (or in a manifest) is actually set in `.190`'s `infra/.env`. The same class will repeat in Epic 8 (`PASSWORD_RESET_TTL_SECONDS` not strictly a secret, but new env-var) and Epic 10 (cutover). A single shell stage scanning `infra/env.example` for `^[A-Z_]+=$` patterns (un-set secrets) and asserting each has a non-empty value in `.190` `infra/.env` would close this loophole. Estimated implementation effort: small; coverage value: high.

2. **Document the regen-vs-disable predicate asymmetry in architecture.md Decision E.** The drill's R-5 (P2) gap flags this. One paragraph of architecture text explicitly noting "disable's UPDATE filter is `invalidated_at IS NULL` only — DOES NOT include `used_at IS NULL` predicate — because disable's semantics are to invalidate the entire batch including any consumed-but-not-yet-invalidated codes, transitioning them to a dual-stamped state". This is binding behavior shipped in code, verified by drill — future maintainers WILL "fix" this by adding the `used_at IS NULL` predicate if architecture doesn't document the intent. Owner: bmad-correct-course session before Epic 8 starts.

3. **Promote the concurrency-pattern catalog to AGENTS.md or `docs/testing-fakeredis.md`.** Epic 6 retro item §2 said the same thing about fakeredis. Epic 7 expanded the pattern set: (a) atomic Redis GETDEL after verify, (b) conditional SQL UPDATE with rowcount inspection, (c) restore-on-fail with commit-guard, (d) asyncio.to_thread offload for CPU-bound bcrypt. **None of these are in a discoverable place.** A 30-50 line `docs/concurrency-patterns.md` cataloging the 4-5 patterns + code citations would prevent re-discovery in Epic 8 (admin endpoints will hit similar surfaces) and Epic 9 (audit verification will need to reproduce them).

4. **Add a "Codex re-review after fix-up commits" verification step to the dev→Codex loop.** Story 7.5's two-layered fix-ups (the second fix-up addressed a regression introduced by the first) show that one Codex pass on the dev commit + one fix-up commit is NOT a fixed-point closure. **The loop should iterate until Codex re-review is clean** — which Story 7.5 ultimately did (re-review on `f325efa` clean). Codifying this expectation (Codex re-review on every fix-up commit, not just on the dev commit) would prevent the case where a fix-up ships with an unflagged regression. Already practiced organically in 7.5; promote to convention.

5. **Tighten the drill's `wait_login_window()` cadence based on actual rate-limit budget.** Drill's R-1 (P2) gap captured this: the drill runs slow (~120s) because of conservative 13s spacing. A more aggressive pacing strategy (track actual sliding-window timestamps in script state, pace exactly to budget edges) would reduce drill duration without sacrificing reliability. Optional future enhancement; current 120s is acceptable for operator-facing drill UX.

6. **Re-run the drill in operator-manual mode (`DRILL_TOTP_CODE_PROVIDER=manual` with real authenticator app) before Initiative 5 closure.** Drill's R-2 (P2) gap: pyotp synthesis can't catch QR-code provisioning URI shape bugs or real-clock-skew edge cases. Owner: Michał, opportunistic. Not blocking Epic 8 or Epic 9; produces a complementary artifact (`2fa-recovery-drill-YYYY-MM-DD-manual.md`).

## Breakthrough moments / patterns reused

1. **`_assert_fernet_key_configured()` endpoint-init gate (Story 7.2)** — moved the hard-fail from app boot (Story 7.1 production incident) to the actual load-bearing endpoint. Pattern: defer hard-fail to the call site that actually consumes the secret, not to app initialization.
2. **`redis.execute_command("GETDEL")` atomic claim after verify (Story 7.3)** — replaces SETNX-lock with atomic claim semantics; pattern reusable for any "verify-then-consume" Redis-backed token flow.
3. **Conditional UPDATE `WHERE col IS NULL` for single-use semantics (Story 7.3)** — `UPDATE recovery_codes SET used_at = NOW() WHERE id = ? AND used_at IS NULL RETURNING id` (or equivalent `result.rowcount` check). Same pattern as Epic 6's CAS predicate, now applied to recovery-code lifecycle.
4. **`asyncio.to_thread(bcrypt.checkpw, ...)` offload (Story 7.3)** — keeps event loop non-blocking under bcrypt cost. Reusable for any CPU-bound verify in async handlers.
5. **Atomic single-session.commit covering UPDATE + INSERT + audit (Stories 7.2, 7.5)** — all dual-state transitions land in one commit; no half-state visible to concurrent readers. Replicates Epic 6 Story 6.2's atomicity pattern at a larger scope (multi-row INSERT + UPDATE).
6. **Restore-on-fail commit-guard (Story 7.3 `16b34d5`)** — when restore-on-fail path runs after partial commit, check `session.in_transaction()` before restoring Redis state. Prevents duplicate-session mint.
7. **Drill artifact 7-section format (Story 7.6)** — mirrors `agent-runbook-smoke-2026-05-11.md` precedent: Preconditions + Step-by-step transcript + Audit row map + Runbook gaps + Cleanup + NFR coverage + Recommendations. Now the binding template for future NFR5-OBS-2 artifacts (E10 cutover smoke, backup-recovery drill).
8. **`X-Portal-Client: drill` request-ID propagation (Story 7.6)** — every HTTP call carries `X-Request-ID` for audit + GlitchTip log correlation. Reusable in Story 9.2 audit-scenario reproducers.

## Significant discoveries (impact on Epic 8 plan)

⚠ **EPIC UPDATE: RECOMMENDED but NOT REQUIRED** — significant discoveries surfaced, but Epic 8 plan as documented in epics.md §1737-1831 is structurally sound. Recommended adjustments are scope refinements, not story splits/merges/descopes.

1. **Story 8.4 (admin 2FA overrides) inherits Story 7.5's pattern directly — but with a different auth contract.** Story 7.5 disable requires user-password + user-TOTP re-auth (defense against XSS-stolen recovery code). Story 8.4 force-disable bypasses re-auth (operator-recovery scenario: user is locked out). The admin-vs-user pivot is well-modeled in epics §1799. **However:** Story 7.5 surfaced two layered Codex fix-ups (concurrent-regenerate + active-only predicate restore); Story 8.4's admin-side mirror needs the SAME concurrency hardening from day one. Spec template should explicitly cite Stories 7.5's atomic-single-commit + active-only predicate patterns.

2. **Story 8.1 (alembic 0014 + LastActiveMiddleware) is the natural carrier for the missing `check-all.sh` gates from Epic 6.** Bundle into 8.1's chore commit: (a) Settings ↔ env.example ↔ docker-compose.yml diff stage; (b) `uv lock --check` stage; (c) optionally, `infra/.env` secret-provisioning verification stage. These three stages would have caught Story 7.1's production incident pre-deploy.

3. **Story 8.5 (admin-issued password-reset) reuses Story 6.6's `register` rate-limit scope — but the rate-limit budget was sized for legitimate registration cadence, not for legitimate admin-driven reset-link issuance.** Three resets/60s per IP may be sufficient for a single-tenant homelab (Michał + maybe one other operator) but tight for a "I locked myself out, please reset" support flow. Recommend explicit verification scenario in Story 9.2 audit smoke matrix.

4. **The drill template (Story 7.6) is now the binding format for future NFR5-OBS-2 artifacts.** Epic 10 cutover smoke (story 10.x) and any backup-recovery drill (deferred backlog) should follow the same 7-H2-section format. Worth documenting as a convention.

5. **The regen-vs-disable predicate asymmetry (item #2 in drift catalog) is binding production behavior that future maintainers WILL "fix" without documentation defending the intent.** Specifically vulnerable: Story 8.4 admin force-disable mirror, Story 8.5 password-reset (which doesn't touch recovery_codes but could prompt revisiting Decision E). Architecture.md update is the strongest defense.
