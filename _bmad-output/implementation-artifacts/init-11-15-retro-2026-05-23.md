## §3 Cross-cutting challenges (worth fixing for next batch)

### C1 — Codex round-2 + round-3 fix-up volume was high
Across 14 base commits, ~7 needed round-2 fix-ups + 1 needed round-3 (Story 18.4 admin_token helper class). Examples of caught issues:
- Story 18.3 round-2: FE source allowlists out of sync with backend enum (5 files)
- Story 19.1 round-2: settings/env/compose gate missed (`check-settings-env-compose.py` not run pre-merge)
- Story 19.5 round-2: shared blob cache StrictMode refcount + revocation invalidation
- Story 19.6 round-2: P1 preview path never reached + P2 stale renders on STL replace + P2 FE consumer drift
- Story 19.7 round-2: bundle code-split regression + size-confirm gate bypass
- Story 20.2 round-2: modal state reset + scrollable

**Root cause:** ITCM autonomous mode optimizes for shipping forward; pre-merge "did I run all the gates?" checklist isn't explicit. Each round-2 P1/P2 was something a quick pre-commit verification pass would have caught.

**Forward fix:** add `feedback_pre_merge_gate_checklist.md` listing the gates that should run before EVERY commit:
1. `check-settings-env-compose.py` if config.py touched
2. `npm run build` if FE route/component file touched (catches bundle-split issues)
3. Visual verification per [[feedback_frontend_visual_verification]] if FE component RENDERED (current memory)
4. `alembic check` if SQLModel model touched
5. Codex review intent classified (heavy security vs routine) BEFORE invocation, not after

### C2 — Blob cache class issues are subtle + cross-context
Story 19.5 shared blob cache fix introduced TWO new latent regressions (StrictMode refcount + revocation invalidation) that Codex round-2 caught. Both come from the "shared-state across React lifecycle" class:
- StrictMode double-mount makes per-effect ownership tracking fragile
- Module-level caches have no React-aware invalidation hook

**Forward fix:** document `feedback_shared_cache_in_react.md` — when adding any module-level cache that React components depend on, MUST address: (a) StrictMode double-mount refcount correctness, (b) explicit invalidation API for components to clear cache on session-state-change, (c) memory cap to prevent unbounded growth. Codex caught both for Story 19.5 — codify the pattern so the next module-level cache author starts from the right base.

### C3 — Lazy code-split discipline was tribal knowledge
Story 19.7 round-1 imported `Viewer3DInline` directly from default export, defeating the lazy() pattern set up in `viewer3d/index.ts`. Codex round-2 caught — `npm run build` warning ("dynamically and statically imported"). The lazy/Suspense pattern is established in the codebase (catalog detail uses it) but new route file added direct import.

**Forward fix:** document `feedback_lazy_import_discipline.md` — when adding a NEW consumer of a heavy component (Three.js, large form, etc.), check if the component has a `lazy()` barrel export. If yes, import from barrel + wrap in `<Suspense>`. If no, propose adding the barrel pattern.

Alternative: ESLint rule that flags direct imports of files known to have lazy exports (advanced; out of scope for memory codification).

### C4 — Single-flight race patterns shipped in 19.6 + would-have-been-needed for thumbnail backfill
Story 19.6 worker idempotency check via row count (>= 4 stl_preview rows) doesn't address two concurrent dispatchers both passing the check pre-resolve. Filed as TB-034.

Mid-batch we also discovered that operator's first thumbnail backfill run only completed 188/590 — likely arq-worker queue race / job killed / silent error. The `--inline` mode worked deterministically.

**Pattern:** for any "lazy dispatch on first hit" worker job, the dispatcher MUST either:
- Atomic claim via Redis SETNX with TTL (single-flight lock)
- OR: dispatcher is idempotent at row level (worker idempotency check is post-fetch, not pre-fetch)

**Forward fix:** `feedback_worker_single_flight.md` — codify the pattern (Redis SETNX recipe) + when it's needed (any "lazy on first share-access" / "auto-render on upload" / similar). Apply during next promotion of TB-034.

### C5 — Visual verification debt accumulated (3 share-view stories + 1 catalog story)
NFR12-FRONTEND-VISUAL-1 was specified in SCP — every UI story carries visual verification gate per [[feedback_frontend_visual_verification]]. But no existing visual specs cover `/share/$token` or `/catalog` toolbar slot. Adding new specs is its own scope (API stubs + token fixtures + new baseline PNGs). Stories 19.5, 19.7, 20.1, 20.2 all shipped with "operator manual verify required" instead of automated visual baseline.

**Forward fix:** before next initiative that touches /share/* or /catalog toolbar, ship a precursor "visual scaffolding" story that adds the missing visual specs + API stubs + token fixtures. One-time cost; unblocks future UI stories from manual verification debt.

Alternative: explicit operator acceptance gate replaces visual baseline for these surfaces (pragmatic; loses regression protection).
