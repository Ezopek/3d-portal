## §3 What we'd do differently — root causes

### L1 — Story 31.1 round-7 lock-ownership race was a real runtime bug the spec couldn't catch by inspection
The Story 31.1 spec at AC-9 said: *"Single-poller SETNX leader-election survives concurrent worker contention"*. The dev shipped `b932a9c` with `SETNX(_LOCK_KEY, b"1", ex=90s)` + unconditional `DEL(_LOCK_KEY)` in `finally`. The acceptance criterion (one poller at a time) is satisfied **on the happy path**. The bug is on the TTL-expiry path: if a poll runs longer than 90s (Spoolman degraded latency, large catalog load, GC pause), Redis expires the lock, a second worker SETNX's a NEW lock with a NEW lease, and the first worker's `finally` runs an unconditional `DEL` against the new worker's lock — violating the AC-9 invariant by literally deleting the second worker's lock.

The Hermes gpt-5.5 delegate reviewer (Story 31.1 round-7) caught it. The fix in `82b4e21` uses per-refresh `secrets.token_hex(16)` token + `WATCH/MULTI/EXEC` compare-and-delete release (Lua `EVAL` would have been a one-call equivalent, but `fakeredis 2.35` lacks the optional `lupa` runtime, so transactions are the safest primitive supported by both production `redis-py 5.3` and the test harness without an extra dependency — rationale documented inline). A new regression test `test_service_refresh_summary_lock_release_is_ownership_safe_under_ttl_expiry_race` hijacks the lock value mid-poll and asserts the foreign token survives the worker's `finally`; verified red against pre-fix code, green after.

**Root cause class:** runtime-invariant bug that spec-level enumeration cannot catch by inspection — the lease-expiry race only surfaces when the test harness can simulate "lock value foreign-rewritten mid-poll", which the original test plan did not include. Codex review (and the regression test it motivated) is the correct catching mechanism. **No process change needed** — the cache-coherence-enumeration table at the architecture level + Codex review at the code level worked as designed. The pattern is what saves the next equivalent bug.
