# Portal as Source of Truth — Design Spec

**Date:** 2026-05-04
**Status:** Draft, pending user review
**Author:** Claude Opus 4.7 (brainstorming session with Michał)
**Scope:** Atomic — replaces only the source-of-truth layer. Future modules (print queue, filament cost, structured notes UI, print requests) are out of scope but the data model is shaped to receive them without further refactor.

---

## 1. Goal, non-goals, glossary, architecture

### 1.1 Goal

Move the source-of-truth (SoT) for the 3D model catalog from the Nextcloud-synced
folder (`/mnt/c/Users/ezope/Nextcloud/3d_modelowanie/`) to the portal's database
and storage volume on `.190`. After cutover, the portal is the **only** writer
(all mutations go through the API), and the Nextcloud folder becomes a
**passive mirror** retained for multi-device sync and version-history safety
net.

### 1.2 Goals (in scope, atomic)

- Metadata schema in the portal DB (Postgres-ready, SQLite for now), full
  refactor "done properly" (categories/tags normalized; files, prints, notes,
  external links as relations).
- File storage on `.190` with UUID-based layout
  (`/mnt/raid/3d-portal-content/models/<uuid>/files/<file-uuid>.<ext>`).
- Portal API rich enough that an agent (and the UI) can do everything that is
  currently done by editing `_index/index.json` directly.
- One-shot, idempotent **migration script** that imports the existing 89 models
  + binaries into the new SoT.
- Optional **reverse-sync** script (`hydrate-local-tree`) that reconstructs a
  human-readable directory of `*.stl` files (and optionally images/prints) on a
  client machine (the user's WSL/Windows for Orca Slicer; optionally also into
  the Nextcloud folder as a passive mirror).
- **Audit log** for every mutation of a model and its related entities.
- **Service account** for the agent, with its own token, identifiable in the
  audit log.
- **Soft delete** on `model`; hard cascade on `model_file`/`model_print`/
  `model_note`/`model_external_link`/`model_tag`.
- Two-stream **backup**: SQLite dump (frequent) + restic on the file storage.

### 1.3 Non-goals (consciously deferred)

- Print queue / user-driven print requests (separate brainstorm).
- Filament cost calculator / Spoolman integration (separate brainstorm).
- Dedicated UI for multi-kind structured notes (the `model_note` table with
  `kind` field is laid down as foundation; the UI to take advantage of it is a
  separate slice).
- Multi-user features (per-user rating, per-user notes).
- STL revision/version history (column `replaced_by_file_uuid` deferred).
- Full-text search via OpenSearch (architectural slot remains; not implemented).
- DB migration SQLite → Postgres (separate slice; the schema is Postgres-ready).
- Orca-URL deep-linking (`bambu://` etc.) — separate topic.

### 1.4 Glossary

- **Model** — the catalog abstraction = a row in `model` plus its files and
  relations. Identified by UUID.
- **Model file** — one binary file attached to a model
  (STL, image, print photo, source CAD file, archive 3MF).
- **Master / SoT** — the `.190` host's database plus
  `/mnt/raid/3d-portal-content/`.
- **Mirror** — a human-readable file tree on a client machine and/or in the
  Nextcloud folder, generated by reverse-sync from the master. Read-only from
  the system's perspective.
- **Hydrate** — the reverse-sync action: portal → local file tree.
- **Agent** — Claude/Codex/other LLM operating on the catalog, authenticated as
  the service account `agent@portal.local`.

### 1.6 Data flow (mutations)

1. Admin or agent sends a request to `POST /api/admin/models` (or
   `PATCH`/`DELETE`/file upload).
2. API validates → DB transaction (insert/update + audit_log row) → returns 2xx.
3. For a file upload: write to `models/<uuid>/files/.tmp.<file-uuid>` →
   `fsync` → atomic rename → `INSERT model_file` in the same DB transaction.
   Cleanup of `.tmp` on failure.
4. The render worker (already exists) reacts to the catalog refresh signal and
   (re)generates thumbnails for models without their own images — kept as is.
5. Reverse-sync (cron on the client machine, **never** on master): pulls the
   diff from `/api/models` since the last run, downloads files, lays them out
   in a human-readable tree. There is never a push in the other direction.

## 2. Storage layout (filesystem)

### 2.1 Volume and master path

A new volume: **`portal-content`** (rw, master storage of model files).

```
/mnt/raid/3d-portal-content/                # bind mount → portal-content volume
└── models/
    └── <model-uuid>/                       # e.g. 6f1c3a2b-…
        └── files/
            ├── <file-uuid>.stl             # original_name kept in DB only
            ├── <file-uuid>.png
            ├── <file-uuid>.jpg
            ├── <file-uuid>.step
            └── <file-uuid>.3mf
```

The old `catalog-data` volume (the RO Nextcloud rsync target,
`/mnt/raid/3d-portal-data/`) **disappears** after cutover.
The `portal-renders` volume **stays** (the worker still writes pre-rendered
thumbnails for models without own images).
The `portal-state` volume **stays** (the DB).

### 2.2 File-naming conventions on disk

- **Per-model directory:** model UUID, dashed lowercase (standard `8-4-4-4-12`).
- **Per-file:** file UUID + extension preserved from `original_name` (lowercase,
  normalized). The extension is kept on disk so that dev tools, tarballs, and
  restic can identify file types at a glance.
- **No `images/` / `prints/` subdirectories on the FS.** That dimension lives
  only in the DB as `model_file.kind`. Reason: putting kind into the path makes
  it a second SoT — moving a file from `kind=image` to `kind=print` would
  require an `mv` on disk in addition to the DB update.

### 2.3 Atomic writes

Every file upload:

```
1.  generate file_uuid = uuid4()
2.  open(f"models/{model_uuid}/files/.tmp.{file_uuid}{ext}", "wb")
3.  write streamed body, fsync, close
4.  compute sha256 (during write or post-write)
5.  BEGIN tx
6.    INSERT INTO model_file (uuid=file_uuid, model_uuid, kind, original_name,
                              storage_path=f"models/{model_uuid}/files/{file_uuid}{ext}",
                              sha256, size_bytes, mime_type)
7.    INSERT INTO audit_log (...)
8.  COMMIT tx
9.  os.rename(tmp, final)
10. fsync(parent_dir)
```

Crash handling:

- Crash before step 9 → tmp is orphaned, no DB row exists. Cleanup: nightly
  job removes `.tmp.*` files older than 1 hour.
- Crash after step 9 (rename) but before COMMIT — impossible, COMMIT is before
  rename.
- Failed COMMIT → tmp orphaned, same cleanup as above.

For **delete**: `DELETE FROM model_file` first (in the same tx as the audit
row), `os.unlink` after commit. If unlink fails — orphan file on FS, no row in
DB (caught by the GC below).

### 2.4 Garbage collector (orphan cleanup)

Cron on `.190`, once per night:

1. Walk `models/*/files/*` and compare with `SELECT storage_path FROM model_file`.
2. Files on FS with no DB row → delete (with a 24h grace from `mtime` to avoid
   eating an in-progress upload).
3. DB rows with no file on FS → emit a structured log event
   `event=integrity_warning` for human investigation. The DB row is **not**
   deleted (audit_log is for mutations only; integrity warnings live in the
   structured log per the project's logging contract).
4. `.tmp.*` files older than 1 hour → delete.

### 2.5 Sha256 and CAS potential

`model_file.sha256` is computed and stored on every upload. We do not use it as
a layout key today (no deduplication). Benefits we get for free:

- Integrity check (recompute, compare against the stored value on demand).
- Per-model dup detection (UNIQUE `(model_uuid, sha256, kind)`).
- Easy migration to content-addressed storage (CAS) later — the column is
  already there, all that changes is the directory layout and an UPDATE on
  `storage_path`.

### 2.6 Storage limits / quotas

No per-model quotas in v1. Free-space monitoring:

- Health endpoint `/api/admin/health/storage` reports `df` for the
  `portal-content` volume.
- Alert in GlitchTip when free space < 10% (separate ticket; observability
  hook should be wired up now anyway).

### 2.7 Permissions

`portal-content` volume:

- Owner: `portal:portal` (UID/GID of the api container).
- Mode: `0750` on directories, `0640` on files (api reads & writes; the worker
  reads only).
- Worker mounts read-only.
- The web container does not mount it at all (web does not serve files
  directly — everything goes through the api endpoint with an ETag).

### 2.8 Reverse-sync path

Reverse-sync (Section 6) operates **only via the API**. The script never reads
`portal-content` over SSH/NFS — to avoid bypassing auth and audit, and to avoid
depending on volume mounts. It is slightly more expensive (HTTP rather than
rsync) but for 89 models and tens of GB the cost is negligible, and it gives a
deterministic "what the API returns is what you have" guarantee.

## 3. Database schema

Postgres-ready DDL (SQLite at first). All PKs are UUID. All tables have
`created_at` + `updated_at` (except join tables, which have only `created_at`).
Timestamps in UTC.

### 3.1 Enums (Python `Enum`, SQLAlchemy `Enum(native_enum=False)` — works on both SQLite and PG)

```
UserRole          : admin | agent | member
ModelSource       : unknown | printables | thangs | makerworld | cults3d | thingiverse | own | other
ModelStatus       : not_printed | printed | in_progress | broken
ModelFileKind     : stl | image | print | source | archive_3mf
ExternalSource    : printables | thangs | makerworld | cults3d | thingiverse | other
NoteKind          : description | operational | ai_review | other
AuditAction       : model.create | model.update | model.delete | model.restore | model.hard_delete
                  | model_file.upload | model_file.update | model_file.delete
                  | model_tag.add | model_tag.remove
                  | model_note.create | model_note.update | model_note.delete
                  | model_print.create | model_print.update | model_print.delete
                  | model_external_link.create | model_external_link.update | model_external_link.delete
                  | category.create | category.update | category.delete
                  | tag.create | tag.update | tag.delete | tag.merge
```

### 3.2 Tables (pseudo-DDL, lightly elided)

```
user
  id              UUID PK
  email           text NOT NULL UNIQUE
  hashed_password text NOT NULL
  role            UserRole NOT NULL
  display_name    text
  created_at, updated_at

category
  id          UUID PK
  parent_id   UUID NULL REFERENCES category(id) ON DELETE RESTRICT
  slug        text NOT NULL          -- ASCII; used in URL + reverse-sync mirror folder name
  name_en     text NOT NULL
  name_pl     text NULL

  UNIQUE (parent_id, slug)                                    -- slug unique within parent
  UNIQUE INDEX uq_category_root_slug ON slug WHERE parent_id IS NULL
                                                              -- NULL-aware: ANSI SQL treats NULL != NULL,
                                                              -- so the composite UNIQUE above does not catch
                                                              -- two root-level categories with the same slug.
                                                              -- The partial unique index closes that gap on
                                                              -- both SQLite (>=3.8.9) and Postgres.

tag
  id          UUID PK
  slug        text NOT NULL UNIQUE   -- ASCII
  name_en     text NOT NULL
  name_pl     text NULL
  created_at, updated_at

model
  id                  UUID PK
  legacy_id           text NULL UNIQUE         -- "001"… only for migrated rows; drop after audit
  slug                text NOT NULL UNIQUE     -- frozen at create; rename only via explicit endpoint
  name_en             text NOT NULL
  name_pl             text NULL
  category_id         UUID NOT NULL REFERENCES category(id) ON DELETE RESTRICT
  source              ModelSource NOT NULL DEFAULT 'unknown'

model_tag                          -- M2M

  PRIMARY KEY (model_id, tag_id)

model_file
  id            UUID PK

model_print
  id            UUID PK

  note          text NULL
  created_at, updated_at

model_external_link
  id          UUID PK
  model_id    UUID NOT NULL REFERENCES model(id) ON DELETE CASCADE

  url         text NOT NULL
  created_at, updated_at

model_note
  id          UUID PK
  model_id    UUID NOT NULL REFERENCES model(id) ON DELETE CASCADE
  kind        NoteKind NOT NULL
  body        text NOT NULL                      -- markdown
  author_id   UUID NULL REFERENCES user(id) ON DELETE SET NULL
  created_at, updated_at

audit_log
  id            UUID PK
  actor_user_id UUID NULL REFERENCES user(id) ON DELETE SET NULL  -- column name diverges from spec ('actor_id' originally) but matches existing AuditEvent convention; harmless rename if desired in Slice 2
  action        text NOT NULL                                      -- free text in Slice 1B; will be tightened to AuditAction enum in Slice 2 once API mutation catalog is final
  entity_type   text NOT NULL
  entity_id     UUID NULL                                          -- nullable: not every event ties to a single entity (auth.login.fail, refresh_catalog)
  before_json   text NULL                                          -- 'jsonb' on Postgres; SQLite stores TEXT — application stores JSON-encoded strings
  after_json    text NULL
  request_id    text NULL

```

### 3.4 Existing tables (kept, lightly migrated)

`render_selection`, `thumbnail_override` already exist and reference models.
After cutover their `model_id: text` (today holding `"001"`) becomes
`UUID REFERENCES model(id) ON DELETE CASCADE`. The migration script handles
both tables.

### 3.5 Notable design decisions

1. **`model.slug` is frozen at create.** Renaming `name_en` does not change
   slug. URLs and reverse-sync mirror folder names stay stable. An explicit
   slug-change endpoint exists for the rare case.
2. **`model.thumbnail_file_id` is a circular FK.** SQLite is permissive;
   Postgres needs deferred constraint or a two-step insert. In practice:
   `INSERT model (thumbnail_file_id=NULL)` → `INSERT model_file …` →
   `UPDATE model SET thumbnail_file_id=...`. Trivial in the service layer.
3. **`category.slug` UNIQUE per parent**, not globally. Two distinct
   `accessories` subcategories under different parents are fine. Root-level
   uniqueness (parent_id IS NULL) is enforced by an extra partial unique
   index `uq_category_root_slug` because ANSI SQL treats NULL != NULL.
4. **`model_tag.tag_id ON DELETE RESTRICT`** — protects against accidental
   deletion of a tag that is still attached to models. Deleting a tag means
   `DELETE FROM model_tag WHERE tag_id=…` first, then `DELETE FROM tag`.
5. **`model_file.UNIQUE (model_id, sha256, kind)`** — catches the same STL
   uploaded twice for the same model, but does **not** block the same file
   used in two different models (e.g. shared thumbnail).
6. **Soft delete only on `model`.** `model_file`/`model_print`/`model_note`
   hard-cascade — for them the `audit_log` snapshot in `before_json` provides
   forensic trace.
7. **`mime_type` not normalized into a lookup table.** Tiny gain, more code.
8. **`audit_log.before_json` / `after_json` as `jsonb`** — Postgres-native;
   in SQLite stored as TEXT with `json_extract` queryable. Sufficient for
   forensics and agent debugging.
9. **`model_external_link.url` always NOT NULL**, `external_id` may be NULL
   (sometimes only the URL is known, e.g. Cults3D where the ID is encoded in
   the slug).
10. **`category` as a self-FK on `parent_id`** rather than a separate
    `subcategory` table. The current depth is two levels; the cost of
    hierarchy is zero, and it removes schema duplication.

11. **`audit_log.action` is `text` in Slice 1B**, not yet the `AuditAction`
    enum from Section 3.1. Reason: the spec's enum enumerates ~20 actions
    keyed to API mutations that don't exist yet. Slice 1B preserves
    existing free-text kinds (`auth.login.success`, `admin.thumbnail.set`,
    etc.). Slice 2 will refactor to a strict enum once the full API
    mutation catalog is final.

12. **`audit_log.entity_id` is nullable.** Spec section 3.2 originally
    listed it as `NOT NULL`, but real audit events don't always tie to a
    specific entity (login failures, catalog refresh, share-token
    operations on legacy string `model_id`s). Amended to nullable; events
    without a specific entity carry contextual data in `after_json`.

## 4. API surface (day 1)

The bar: an agent (and the UI) must do everything they would otherwise do by
editing `index.json`. No gaps = no temptation to bypass the API.

### 4.1 Auth

```
POST /api/auth/login                      body: {email, password}  → {access_token, expires_in}
POST /api/auth/refresh                    cookie / refresh token   → {access_token, expires_in}
GET  /api/auth/me                         → {id, email, role, display_name}
```

The agent logs in like any user (`role=agent`), JWT for 24h, refresh in place.
Future upgrade slot: a long-lived PAT in `X-Portal-Agent-Token` — not
implemented now.

Every mutation: middleware reads `actor_id` from the JWT and writes to
`audit_log`.

### 4.2 Public read (basic-auth household)

```
GET  /api/models                          ?category=&status=&tag=&q=&include_deleted=false&limit=50&cursor=
GET  /api/models/{id}                     → full model embedding category, tags, files, prints, notes, external_links
GET  /api/models/{id}/files               → list (already exists; kind filter)
GET  /api/models/{id}/files/{file_id}/content   → bytes, ETag, Content-Type, Content-Disposition
GET  /api/categories                      → tree (parent_id materialized)
GET  /api/tags                            ?q=&limit=
```

### 4.3 Admin write (`role in {admin, agent}`)

**Models:**
```
POST   /api/admin/models                          body: {name_en, name_pl?, category_id, source?, status?, rating?, tags?: [tag_id]}
PATCH  /api/admin/models/{id}                     body: any subset + {slug?}  (slug only via this endpoint)
POST   /api/admin/models/{id}/restore             clears deleted_at
DELETE /api/admin/models/{id}                     soft delete
DELETE /api/admin/models/{id}?hard=true           hard delete (admin only; agent forbidden)
PUT    /api/admin/models/{id}/thumbnail           body: {file_id}
```

**Files:**
```
POST   /api/admin/models/{id}/files               multipart: file=<binary>, kind=stl|image|print|source|archive_3mf
PATCH  /api/admin/models/{id}/files/{file_id}     body: {kind?, original_name?}
DELETE /api/admin/models/{id}/files/{file_id}
```

Upload uses `multipart/form-data`. Per-file size limit: 500 MB
(`client_max_body_size 500m` on nginx-180; FastAPI streams to a tempfile).
Sha256 and mime-type computed server-side, never trusted from the body.

**Tags (M2M on model):**
```
PUT    /api/admin/models/{id}/tags                body: {tag_ids: [...]}                # replace
POST   /api/admin/models/{id}/tags                body: {tag_id}                         # add one
DELETE /api/admin/models/{id}/tags/{tag_id}                                              # remove one
```

**Tags (global):**
```
POST   /api/admin/tags                            body: {slug, name_en, name_pl?}
PATCH  /api/admin/tags/{id}                       body: {slug?, name_en?, name_pl?}
DELETE /api/admin/tags/{id}                       (RESTRICT if used)
POST   /api/admin/tags/merge                      body: {from_id, to_id}                 # rewires model_tag
```

**Categories:**
```
POST   /api/admin/categories                      body: {parent_id?, slug, name_en, name_pl?}
PATCH  /api/admin/categories/{id}                 body: {parent_id?, slug?, name_en?, name_pl?}
DELETE /api/admin/categories/{id}                 (RESTRICT if has models or children)
```

**Notes / prints / external links:**
```
POST   /api/admin/models/{id}/notes               body: {kind, body}
PATCH  /api/admin/notes/{note_id}                 body: {kind?, body?}
DELETE /api/admin/notes/{note_id}

POST   /api/admin/models/{id}/prints              body: {printed_at?, note?, photo_file_id?}
PATCH  /api/admin/prints/{print_id}
DELETE /api/admin/prints/{print_id}

POST   /api/admin/models/{id}/external-links      body: {source, external_id?, url}
PATCH  /api/admin/external-links/{link_id}
DELETE /api/admin/external-links/{link_id}
```

**Operations:**
```
GET    /api/admin/audit-log                       ?entity_type=&entity_id=&actor_id=&action=&since=&until=&limit=
GET    /api/admin/health/storage                  → {free_bytes, total_bytes, percent_used}
POST   /api/admin/refresh-catalog                 (kept; now invalidates in-memory caches only — no FS scan)
```

### 4.4 Idempotency

- `POST /api/admin/models/{id}/files` accepts an `Idempotency-Key: <uuid>`
  header. On retry within 24h the server returns the existing file row instead
  of creating a duplicate. Useful for an agent retrying after an upload
  timeout.
- For other mutations the idempotency key is optional but accepted (FastAPI
  middleware backed by Redis).

### 4.5 Consistency

- Every mutation of a single entity = one DB transaction including its
  `audit_log` row.
- Every file operation: tx commit precedes `os.rename` (Section 2.3).
- Race conditions on UNIQUE constraints (e.g. two requests creating a model
  with the same slug) → 409 Conflict; the client retries.

### 4.6 Error model

Standard FastAPI `HTTPException` plus a structured body:

```json
{"error": {"code": "model_not_found", "message": "...", "details": {...}}}
```

About a dozen codes on day 1, documented in the cheatsheet. Every 4xx logs a
warning; every 5xx → Sentry/GlitchTip plus a structured log event
(`event=server_error`). The audit log is mutation-only and does not receive
error rows.

### 4.7 OpenAPI + agent cheatsheet

- FastAPI generates `/api/openapi.json` (already in place).
- A new file `docs/agent/api-cheatsheet.md` lists the most common agent flows:
  1. Add a model with files (3 calls: `POST` model → `POST` files → `PUT`
     thumbnail).
  2. Convert 3MF → STL and archive the original (POST file `kind=stl`,
     POST file `kind=archive_3mf`, optionally DELETE old file).
  3. Move a model to a different category (PATCH model).
  4. Add or edit an operational note (POST/PATCH note with
     `kind=operational`).
  5. Record a printed entry with a photo (POST file `kind=print` →
     POST print with `photo_file_id`).
  6. Deduplicate tags (POST `tags/merge`).

### 4.8 Out-of-scope for day 1 (deliberate)

- OpenSearch-backed search (basic ILIKE only on day 1).
- WebSocket / SSE.
- Bulk endpoints other than `PUT /tags` as a single-bulk pattern. The agent
  loops; for 89 models it's fine.
- Future `queue/`, `prints/` (user-facing), `requests/`, `cost-calculator/`
  modules — slots remain; endpoints not added.

## 5. Migration script

One-shot, **idempotent** transfer of 89 models from `index.json` plus binaries
into the new SoT. Re-runnable after partial failure without producing
duplicates.

### 5.1 Location and invocation

```
scripts/migrate_from_index_json.py
```

Invocation (on `.190` inside the api container, or as a standalone CLI with
DATABASE_URL + STORAGE_ROOT set):

```
python scripts/migrate_from_index_json.py \
    --index    /mnt/raid/3d-portal-data/_index/index.json \
    --src-root /mnt/raid/3d-portal-data \
    --src-3mf  /mnt/raid/3d-portal-staging/3mf-originals \
    --dst-root /mnt/raid/3d-portal-content \
    --report   docs/migration/report.md \
    [--dry-run]
```

`--dry-run`: parses, validates, writes the report; no inserts, no copies.

### 5.2 Pre-flight

1. DB reachable; alembic at the latest revision.
2. The index.json file exists and parses as a JSON array.
3. `--src-root` exists and is readable.
4. `--dst-root` exists, writeable; free space ≥ `du -s --src-root` × 2 + 5 GB.
5. No `migration_complete.flag` on `--dst-root` (otherwise abort, unless
   `--force-rerun`).
6. Snapshot `index.json` → `docs/migration/index-snapshot-YYYY-MM-DD.json`
   (once).

### 5.3 Migration actor

The script inserts (or reuses) a `migration@portal.local` user, `role=admin`,
`hashed_password = "*"` (login disabled). Every audit row gets `actor_id`
pointing at this user and `request_id="migration-batch-<UTC-iso>"`.

### 5.4 Steps (per record, one tx)

```
for record in index.json:
    if model.legacy_id == record["id"] exists:
        continue                                 # idempotent skip

    BEGIN tx
        category_id  = upsert_category_path(record["category"], record.get("subcategory"))
        tag_ids      = [upsert_tag(t) for t in record["tags"]]   # canon + dedup
        model_id     = uuid4()
        INSERT model (
            id=model_id, legacy_id=record["id"],
            slug=unique_slug(record["name_en"], legacy_id=record["id"]),   # see 5.5a
            name_en, name_pl, category_id,
            source=normalize_source(record["source"]),
            status=record["status"],
            rating=record["rating"],
            date_added=record["date_added"],
            thumbnail_file_id=NULL                                # set later
        )
        INSERT model_tag rows
        files = walk_model_dir(src_root / record["path"])
        for f in files:
            kind, original_name = classify(f)
            file_id = uuid4()
            sha = compute_sha256(f)
            UNIQUE check (model_id, sha, kind) — if collision, skip with warn
            copy f → dst_root / f"models/{model_id}/files/{file_id}{ext}"
            INSERT model_file row

        for src_3mf_file in walk_3mf_dir(src_3mf / record["path"]):
            ... copy as kind=archive_3mf

        if record["thumbnail"]:
            UPDATE model SET thumbnail_file_id = match_filename(record["thumbnail"])
        elif first image file exists:
            UPDATE model SET thumbnail_file_id = first image

        for print_entry in record["prints"]:
            photo_file_id = match_filename(print_entry["photo"]) if print_entry.get("photo") else NULL
            INSERT model_print

        for src in ["printables_id", "thangs_id", "makerworld_id"]:
            if record[src]:
                INSERT model_external_link (source=src.removesuffix("_id"), external_id=record[src],
                                            url=build_url(src, record[src]))
        if record.get("source_url") and not matched_above:
            INSERT model_external_link (source="other", external_id=NULL, url=record["source_url"])

        if record["notes"]:
            INSERT model_note (kind="description", body=record["notes"], author_id=NULL)

        INSERT audit_log rows (action=*.create, request_id=batch)
    COMMIT tx
```

### 5.5a Slug uniqueness during migration

`model.slug` is globally UNIQUE. Two models with the same `name_en` (e.g.
"Cube") would naively slugify to the same value and collide. The migration
helper `unique_slug(name_en, legacy_id)` produces:

- `slugify(name_en)` if no collision in the running batch.
- `slugify(name_en) + "-" + legacy_id` if a collision is detected (legacy_id
  is unique by definition since it comes from `index.json`).

For new models created post-cutover via the API the same pattern applies but
the suffix is a short numeric counter (or a 4-char UUID prefix) since
post-cutover models do not have a legacy_id.

### 5.5 Tag dedup (PL/EN pairs)

Today `tags[]` is a mixed list, e.g. `["dragon", "smok", "egg", "jajko"]`. The
script:

1. Loads `scripts/tag_translations.py` — a hand-curated map:
   ```python
   PL_EN = {
     "smok": "dragon", "jajko": "egg", "przegubowy": "articulated",
     "narzedzia": "tools", "uchwyt": "holder", ... # ~50 pairs
   }
   ```
2. For each raw tag:
   - If the key is in `PL_EN` → canonical EN tag, set `name_pl=` from the key.
   - Otherwise the tag is treated as EN, `name_pl=NULL`.
3. After the run the script prints a "Tags requiring manual review" list of
   EN-only tags that may actually be PL but lack a known mapping.

`tag_translations.py` is committed to the repo and can be extended
incrementally. Re-running the migration is **not** the way to extend
translations after cutover — use `POST /api/admin/tags/merge` instead.

### 5.6 File classification

```python
def classify(path: Path) -> tuple[ModelFileKind, str]:
    ext = path.suffix.lower()
    parts = path.parts
    if ext == ".stl":           return ("stl", path.name)
    if "prints" in parts:        return ("print", path.name)
    if "images" in parts or ext in {".png", ".jpg", ".jpeg", ".webp"}:
                                  return ("image", path.name)
    if ext in {".step", ".stp", ".obj", ".f3d", ".fcstd"}:
                                  return ("source", path.name)
    if ext == ".3mf":            return ("archive_3mf", path.name)
    raise UnknownFileKind(path)                                       # → log + skip
```

Unknown files (`*.gcode`, `*.bgcode`, `*.docx`, etc.) → log a warning, **skip**
(not copied). All skips are listed in the final report.

### 5.7 Source normalization

```python
def normalize_source(s: str | None) -> ModelSource:
    if not s or s == "unknown": return ModelSource.unknown
    s = s.lower().strip()
    if "printables" in s: return ModelSource.printables
    if "thangs" in s:     return ModelSource.thangs
    if "makerworld" in s: return ModelSource.makerworld
    if "cults"   in s:    return ModelSource.cults3d
    if "thingiverse" in s:return ModelSource.thingiverse
    if s in ("own", "self"): return ModelSource.own
    return ModelSource.other
```

### 5.8 Verification (post-import)

1. `SELECT COUNT(*) FROM model WHERE legacy_id IS NOT NULL` == count in
   index.json (89). Mismatch → fail, write report.
2. Per model: `SELECT COUNT(*) FROM model_file WHERE model_id=? AND kind='stl'`
   ≥ 1. Mismatch → fail.
3. Sha256 spot check on 5 random files per model — re-read, compare with
   `model_file.sha256`. Mismatch → fail.
4. No orphaned `model_file` rows (left join `model`).
5. Free space on `--dst-root` reported before/after.

Pass → write `migration_complete.flag` (timestamp + report hash).
Fail → details in the report, `sys.exit(1)`. After fixing the cause the
script can be re-run.

### 5.9 Report

`docs/migration/report-YYYY-MM-DD.md` contains:

- Status: success / failed.
- Models imported / skipped (with reason).
- File counts per kind.
- Tag counts (EN-only / EN+PL aliased).
- Categories created.
- Tags flagged for manual review.
- Skipped files (path + reason).
- Free space before/after.

### 5.10 Pre-migration data quality report (DQR)

A separate small script `scripts/data_quality_report.py` runs **before** the
migration as a stand-alone step. It generates
`docs/migration/data-quality-report.md` with two sections:

- **Informational** — fields that are commonly empty for legitimate reasons
  (no `source`, no `notes`, no own `images/`). Listed for awareness, no
  action expected.
- **Actionable** — likely data hygiene issues:
  - Models with `name_en` but no `name_pl` or vice versa.
  - Tags appearing exactly once (likely typos or merge candidates).
  - Tags with no known PL/EN mapping (candidates to extend
    `tag_translations.py` before migration).

DQR does not block migration. The user reads it and decides whether to
clean up some items by hand (or via the agent in the old workflow) before
the cutover, or to import as-is and clean up later through the new API.

### 5.11 Post-migration agent sweep (smoke test)

After 72h of passive monitoring (Section 7) the agent gets its first real
workload through the **new API**:

- For models with a `source_url` but no `external_link` row → POST link
  (extract `printables_id` etc. from the URL).
- For models with an external link but no `description` note → fetch the
  page description from the source service → POST a note with
  `kind=description`.
- Re-tagging from the model description and name → `PUT /api/admin/models/{id}/tags`.
- Optional `name_pl` enrichment for models that lack one.

This doubles as the agent's first end-to-end exercise of the new API; if
the API has gaps, they surface here.

## 6. Reverse-sync (`hydrate-local-tree`)

Optional script that reconstructs a human-readable file tree on a client
machine (the user's WSL/Windows for Orca, optionally also the Nextcloud
folder as a passive mirror) from the portal master. Pull-only; never push.
Operates exclusively via the API.

### 6.1 Location and invocation

```
scripts/hydrate_local_tree.py
```

Invocation (from WSL):

```
python scripts/hydrate_local_tree.py \
    --portal-url   http://192.168.2.190:8090 \
    --token-file   ~/.config/3d-portal/agent.token \
    --target       /mnt/c/Users/ezope/Nextcloud/3d_modelowanie \
    --kinds        stl                    # default; comma-list, see 6.2
    [--prune-deleted]                     # remove local files not in master
    [--include-soft-deleted]              # include models with deleted_at != NULL
    [--dry-run]
```

The credentials in `~/.config/3d-portal/agent.token` are the same as those
used for mutations. On day 1 these are JWT-based — a small wrapper
re-authenticates daily; see Section 8.2 open question 2 for the longer-lived
PAT alternative.

### 6.2 Default kinds

`stl` only — the primary use case is "Orca opens an STL". Optional
`--kinds stl,image,print` for a richer mirror. Never `source` or
`archive_3mf` — large, rarely needed locally.

### 6.3 Local layout

```
<target>/
├── decorum/
│   └── articulated_figures/
│       └── articulated-dragon-and-dragon-egg-001/        # category slug / [subcategory slug] / model slug
│           ├── original-dragon.stl
│           └── original-egg.stl
├── drukarka_3d/
│   └── k1_max/
│       └── nozzle-cleaner-bracket-042/
│           └── bracket-v2.stl
└── .hydrate-state.json                                    # local state — sha256 per file
```

`<model-slug>` is frozen in the DB → renaming a model does not move the local
folder (good: links in tools do not break). `<category-slug>` is also stable.
`name_pl` is not used in folder names — the slug is derived from `name_en`
(EN-default decision).

The trailing `-001` is `legacy_id` if present, otherwise an 8-char UUID
suffix — guarantees folder uniqueness even when two models share `name_en`.

### 6.4 Algorithm

```
1.  GET /api/models?include_deleted=<flag>&limit=200&cursor=...   (paginate)
2.  For each model:
       expected_path = target / category_path / model_slug
       mkdir -p expected_path
       GET /api/models/{id}/files?kind={kinds}                    (filter)
       For each file:
            local_file = expected_path / file.original_name
            local_sha = local_state.get(local_file.relative)
            if local_sha == file.sha256:
                skip                                              (in sync)
            elif local_file exists and sha matches:
                update state, skip download
            else:
                GET /api/models/{id}/files/{file_id}/content       (streamed)
                write to .tmp.<file>, fsync, rename
                update state
3.  If --prune-deleted:
       For each path under <target>/<category>/<model>:
            if not in master:
                remove local file (with confirmation prompt unless --yes)
4.  Write .hydrate-state.json (sha256 per relative path + master last-seen).
5.  Print summary: N models, M files downloaded, K skipped, P pruned.
```

`.hydrate-state.json` avoids re-hashing local files on every run. Fields:
`last_run_at`, `state_version`, `paths: {relative_path: sha256}`.

### 6.5 Idempotency, retry

- A full run is idempotent — a subsequent run does nothing (other than for
  newly added or modified models).
- Interruption mid-run: `.tmp.*` orphans are cleaned on the next run.
- Network failure: per-file retry with exponential backoff (3 attempts), then
  skip + report.

### 6.6 Cron / scheduling

Sample crontab on WSL:

```
*/30 * * * *  ~/repos/3d-portal/scripts/hydrate_local_tree.py --target /mnt/c/Users/ezope/Nextcloud/3d_modelowanie >> ~/.local/log/hydrate.log 2>&1
```

Every 30 minutes, log to a dedicated file. The Nextcloud desktop client picks
the changes up and propagates them across the user's other devices (the
"passive mirror" tier of the architecture).

### 6.7 Master-side audit logging

`GET /api/models/{id}/files/{file_id}/content` does **not** generate an
`audit_log` row. Read-only operations have no value in the audit log; only
mutations are recorded.

### 6.8 What reverse-sync does **not** do

- Does not push files to master.
- Does not modify metadata (read-only).
- Does not rebuild `_index/index.json` (the legacy file is not needed —
  Orca consumes STL paths, not metadata).
- Does not pull `audit_log`, `model_external_link`, or `model_note`
  (portal-only entities).

### 6.9 Errors

- Token expired / 401 → log + abort (no automatic re-auth — user decides).
- Master unreachable → log + abort (the next cron run retries).
- Local free space < 1 GB → log + abort.
- Sha256 mismatch after a download → retry (3x), then log error and skip
  the file.

## 7. Backup, cutover, rollback

### 7.1 Backup (after cutover, two streams)

**Stream 1 — DB (extended `backup-sqlite.sh`):**
```
*/15 * * * *  /opt/3d-portal/infra/scripts/backup-sqlite.sh --full
```
Changes vs today:

- Frequency: nightly → every 15 minutes (DB now holds the entire SoT;
  RPO 15m is reasonable).
- Retention: 24h × 4 (every 15 min), 30 days × 1 (daily), 12 months × 1
  (monthly).
- Location: `/mnt/raid/backups/db/portal.db.<ts>`.
- Sanity check: after each dump, `sqlite3 file '.schema'` must return
  non-zero rows.

**Stream 2 — files (`portal-content` via restic):**
```
0 3 * * *  /opt/3d-portal/infra/scripts/backup-content.sh
```
A new script:

- `restic backup /mnt/raid/3d-portal-content/ --tag content` to a restic
  repo on an external drive (`/mnt/external/restic-repo`, mounted only
  during backup).
- Retention: `restic forget --keep-daily 7 --keep-weekly 4 --keep-monthly 12`.
- Health: `restic check` weekly.

**Stream 3 (free, already in place) — Nextcloud passive mirror.**
Reverse-sync writes into `/mnt/c/.../Nextcloud/3d_modelowanie/`. Nextcloud
adds multi-device + version history → copies on every machine and in the
cloud. **Not** a substitute for the DB stream (only files), but it
guarantees we never lose STLs even if the DB and the restic backup both
fail.

**Backup matrix:**

| Item | DB dump | restic | Nextcloud | external |
|---|---|---|---|---|
| metadata (model, tags, notes, audit_log) | ✓ | – | – | – |
| binary files | – | ✓ | partially (post-reverse-sync) | – |
| reverse-sync state (.hydrate-state.json) | – | – | ✓ (on the client) | – |
| restic repo metadata | – | – | – | (location itself) |

7.  Compose update: drop the `catalog-data` mount (the RO Nextcloud sync

9.  Smoke tests:

### 7.3 Rollback

Three risk windows, each with its own plan:

**1. Before step 5 (before `migrate_from_index_json.py`).**
- State: alembic up (new tables empty), api stopped.
- Rollback: `alembic downgrade -1` × N (down to the pre-cutover
  revision), restore `pre-cutover-<ts>.db`, restart api with the previous
  compose. The old SoT (`index.json` + rsync) is untouched. Re-enable the
  sync-data cron.
- Time: ~10 min.

**2. After migration but before api start (steps 5-7).**
- State: data is in the DB and on the FS, but no one has used it.
- Rollback: `alembic downgrade` + `rm -rf /mnt/raid/3d-portal-content/models/*`
  + restore pre-cutover.db + revert compose changes + re-enable sync.
- Time: ~15 min.

**3. After cutover, after api start (post-D, once anyone has edited
  anything).**
- State: portal is live on the new SoT; agent/admin may have edited data.
- Rollback: heavy. Restore pre-cutover.db (lose all post-cutover edits),
  `rm -rf portal-content`, revert compose, re-enable sync, announce
  "the last X hours of edits were lost".
- Time: ~30 min, **lossy**.
- Mitigation: in D to D+3 we deliberately hold off the agent sweep and
  any large admin-side work. The clean-rollback window grows.

**Rollback triggers:**

- `migrate_from_index_json.py` fails and cannot be finished after a fix.
- Model count in the DB ≠ 89 (or the expected post-migration count).
- Smoke test fails (UI cannot load the list / `GET /api/models` 5xx).
- Render worker fails to start.
- Sentry/GlitchTip burst > X in 5 minutes after start.

Other issues (e.g. minor agent bug in the API) — **not** rollback
triggers; hot-fix instead.

### 7.4 Post-cutover monitoring (intensive)

- `audit_log` — does the mutation pattern look sensible?
- `portal-content` free space — alert at <10%.
- Render worker — pre-renders thumbnails for models without own images;
  counts must match.
- 5xx rate on `/api/models` — should be 0.
- Reverse-sync — runs and the sha256s match.

## 8. Testing strategy, open questions, recap

### 8.1 Testing strategy

Principle: TDD for logic, integration over unit, visual regression mandatory
for UI (per the project's CLAUDE.md).

**1. Alembic migrations:**
- pytest: applies all migrations on a fresh DB, downgrades, re-upgrades.
  Each new revision has its own test (idempotent, no data loss).
- Smoke: applies migrations on a copy of the **pre-cutover DB** (where
  `render_selection` / `thumbnail_override` carry stringy `model_id="001"`)
  and asserts the FK rewires correctly without losing rows.

**2. Migration script (`migrate_from_index_json.py`):**
- Synthetic fixture: `tests/fixtures/migration/index.json` with 3-4 models
  covering the cases:
  - Full metadata (printables_id, notes, prints[], multiple PL+EN tags).
  - Minimal (name + path + 1 STL).
  - Pathological (no STL on FS) → assert script exit 1 with a descriptive
    error.
  - With a 3MF original under `_archive/`.
- Tests:
  - Happy path: `model` table has 3 rows with `legacy_id` matched.
  - Idempotency: re-run = no-op (zero duplicates, zero file copies).
  - Tag dedup: a PL+EN pair → one tag with `name_en` + `name_pl`.
  - Sha256 consistency: every `model_file.sha256` matches its file on disk.
  - audit_log: post-migration every new entity has an audit row with
    `actor_id=migration_user`.
  - Report: generates a markdown report with the expected sections.

**3. API endpoints (pytest + httpx test client):**
- `POST /api/admin/models` → 201, audit_log row, `slug` derived from
  `name_en`.
- `POST /api/admin/models/{id}/files` with an idempotency key → second
  call returns the same file id without duplicating the file on FS;
  sha256 unchanged.
- 500 MB upload — streamed to tmp, sha256 computed during the write,
  atomic rename, on failure tmp is cleaned up.
- `DELETE /api/admin/models/{id}` (soft) → `deleted_at` set; `GET
  /api/models/{id}` returns 404 (unless `?include_deleted=true`); the
  `model_file` rows still exist.
- `DELETE …/{id}?hard=true` as agent → 403, as admin → 200, file
  physically removed, audit_log retains `before_json`.
- `POST /api/admin/models/{id}/restore` → `deleted_at=NULL`, audit row
  written.
- `PUT /api/admin/models/{id}/tags` (replace) → idempotent; multi-call
  identical effect.
- `POST /api/admin/tags/merge` → `model_tag` rewired; source tag deleted.
- `POST /api/admin/categories` with `parent_id` → creates subcategories;
  `DELETE` of an in-use category → 409.
- `POST /api/admin/models/{id}/external-links` → UNIQUE per source enforced
  (second call for the same source → 409).
- Service-account login + JWT with `role=agent`: access to admin endpoints
  OK; access to `?hard=true` blocked.
- Audit log entries appearing for **every** mutation (parametrized test
  iterating endpoints).

**4. Storage layer (atomic ops, GC):**
- "Kill mid-upload": writes the tmp file, raises before rename → DB
  rollback, tmp orphaned.
- "GC orphans": creates a file on FS without a DB row; the scan finds and
  cleans it (with a mocked grace period).
- "GC the other way": a DB row with no file → logs a warning, writes an
  audit row, does **not** delete the row.

  - `--prune-deleted` removes files no longer in the API (with

  - Network failure mid-run → retry, then clean log error; partial state

  - Sha256 mismatch after a download → retry, then skip + log.

### 8.2 Open questions / risks (resolve during writing-plans)

1. **`mime_type` detection** — `python-magic` (libmagic, accurate) vs
   stdlib `mimetypes` (extension-based, less accurate). STL is sometimes
   detected as `application/octet-stream` by libmagic. Recommendation: per-
   `kind` override (`kind=stl → mime=model/stl` hardcoded), the rest from
   `python-magic` with `mimetypes` fallback.

2. **Service-account auth on day 1** — JWT 24h with password (like every
   user), with a future plan to add a long-lived PAT. Alternative: PAT
   from day 1 (separate `personal_access_token` table). Suggestion:
   start with JWT plus a daily re-auth cron on the WSL host.

3. **Slug rename — when does it actually happen?** Suggestion:
   `PATCH /api/admin/models/{id}` accepts a `slug` change only when the
   new slug does not collide (UNIQUE) and the request carries a
   `X-Slug-Change-Reason: <text>` header — a deliberate guardrail
   against accidental clicks. Audit log carries the reason.

4. **3MF originals — what if some are missing on Windows** (e.g. agent
   converted and discarded long ago)? Migration: no 3MF → no
   `model_file kind=archive_3mf` row, nothing else. Logged as info in the
   report. Not blocking.

5. **Render worker after cutover** — today reads `catalog-data` (RO
   Nextcloud rsync). After cutover it must read `portal-content`. Small
   compose change + path change in code. The render code already has
   path-generation logic — review for any hardcoded assumption about the
   old layout.

6. **`audit_log` retention** — `before_json/after_json` over 100
   mutations/day × 365 days = a not-tiny table. Decision: no retention in
   v1; the table is indexed on `(entity_id, at DESC)` so per-model
   history queries stay fast. A retention rule (e.g. 2 years) is a
   separate slice.

7. **OpenAPI schema versioning** — when queue/cost-calculator are added
   later, the schema grows. Suggestion: `/api/openapi.json` always
   reflects the current build; the first-version snapshot lives in
   `docs/agent/openapi-snapshots/v1-<date>.json` for audit.

### 8.3 Recap of decisions

- **SoT scope:** D — everything (binaries + metadata) on `.190` as
  master; an optional reverse-sync script hydrates the local STL tree.
- **Agent workflow:** B — agent stays but operates exclusively through
  the portal API (never touches the FS directly).
- **Storage layout:** B — UUID-based, `models/<uuid>/files/<file-uuid>.<ext>`;
  the DB is the only name dictionary.
- **Model identifier:** B — UUID PK + `slug` (frozen) + `legacy_id`
  (nullable, used during the migration window).
- **File scope:** A — every model-related binary goes into portal storage
  (STL + images + prints + CAD sources + 3MF originals), distinguished by
  `model_file.kind`.
- **Cutover:** A — hard cutover; Nextcloud becomes a passive mirror
  (reverse-sync target); Nextcloud version history kept as a safety net.
- **Schema:** C — full "do it properly" refactor + 20 best-practice
  decisions.
- **Soft delete:** YES on `model`, hard cascade on related rows.
- **Service account for the agent:** YES, separate user
  `agent@portal.local` with `role=agent`.
- **Hard delete (`?hard=true`):** YES, admin-only; the agent is forbidden.
- **`UserRole`:** `admin | agent | member` from day 1.
- **`model.rating`:** `float 1.0-5.0`, FE rounds to one decimal place.
- **`date_added`:** preserved from `index.json` during the migration.
- **Migration:** per-model transaction (one bad row does not block the
  others).
- **3MF originals:** rsync from Windows to the `.190` staging area before
  migration (option B).
- **Pre-migration:** Data Quality Report distinguishing informational vs
  actionable.
- **Post-migration:** agent sweep through the new API as the "first real
  workload" (after 72h of passive monitoring).
- **Reverse-sync target:** `/mnt/c/Users/ezope/Nextcloud/3d_modelowanie/`
  (overlays the old folder).
- **`--prune-deleted`:** default `false` (opt-in).
- **Language:** EN as default; PL secondary in UI/slugs.
